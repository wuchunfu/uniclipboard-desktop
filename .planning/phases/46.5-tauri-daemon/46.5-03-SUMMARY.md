## Decisions Made

- 46.5 完成后，frontend pairing 命令失败只走 typed command return，不再保留第二条事件通道。
- host confirm 的 daemon `/setup/state` 契约允许 `Completed` 状态配合 `host-confirm-peer` hint 表达“等待 host 确认”，测试应锁定这一契约，而不是强行假设 join-flow 状态枚举。
- daemon API regressions 不应再依赖真实 watcher channel 或隐式网络初始化；这些前置条件必须在 fixture 中显式表达。

## Deviations from Plan

### Auto-fixed Issues

**1. [Task 2] daemon API fixtures needed explicit runtime prerequisites**

- **Found during:** `pairing_unpair_route_unpairs_device_via_daemon_usecase` and `setup_confirm_peer_routes_host_confirmation_through_daemon_pairing_host`
- **Issue:** test fixtures assumed watcher control and pairing transport were already ready, which caused `Watcher channel closed` and `pairing service not initialized` failures unrelated to the route logic under test
- **Fix:** added a local noop watcher control for daemon setup fixtures, injected runtime into the unpair route fixture, and explicitly started the network before exercising `/pairing/unpair`
- **Files modified:** `src-tauri/crates/uc-daemon/tests/pairing_api.rs`, `src-tauri/crates/uc-daemon/tests/setup_api.rs`, `src-tauri/crates/uc-daemon/Cargo.toml`
- **Verification:** targeted `cargo test -p uc-daemon --test pairing_api ...` and `cargo test -p uc-daemon --test setup_api ...`

## Issues Encountered

- The first host-confirm regression expected `/setup/state` to surface `JoinSpaceConfirmPeer` directly, but daemon query projection legitimately surfaced `Completed` with `nextStepHint = host-confirm-peer` while the pairing snapshot was still in host verification.
- Running two `setup_api` test filters in parallel caused a transient SQLite `database is locked` failure. Sequential verification matched the planned workflow and eliminated that noise.
