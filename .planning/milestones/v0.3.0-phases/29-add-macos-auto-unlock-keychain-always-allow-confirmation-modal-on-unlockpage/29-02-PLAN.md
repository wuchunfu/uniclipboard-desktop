Add state:

Update `handleAutoUnlockChange`:

```typescript
const handleAutoUnlockChange = async (checked: boolean) => {
  if (checked && isMac) {
    setVerifyError(null)
    setShowKeychainModal(true)
    return
  }
  await updateSecuritySetting({ auto_unlock_enabled: checked })
}
```

Add verification handler:

```typescript
const handleKeychainVerify = async () => {
  setVerifying(true)
  setVerifyError(null)
  try {
    const granted = await verifyKeychainAccess()
    if (granted) {
      await updateSecuritySetting({ auto_unlock_enabled: true })
      setShowKeychainModal(false)
    } else {
      setVerifyError(t('unlock.keychainModal.error'))
    }
  } catch (error) {
    setVerifyError(t('unlock.keychainModal.error'))
  } finally {
    setVerifying(false)
  }
}

const handleKeychainCancel = () => {
  setShowKeychainModal(false)
  setVerifyError(null)
}
```

Add AlertDialog modal JSX (place before the closing `</div>` of the root):

Use `AlertDialog` with `open={showKeychainModal}` and `onOpenChange` that only closes via cancel handler (NOT the overlay click, since user needs to complete the flow). Use `AlertDialogContent` with `onEscapeKeyDown` and `onInteractOutside` that call `e.preventDefault()` to prevent accidental dismissal while verifying.

Modal layout:

- `AlertDialogHeader` with title and description
- Ordered list (`<ol>`) with 3 steps using `t()` for each step
- Small note text mentioning this is a macOS system dialog
- If `verifyError` is non-null: red error message with `text-destructive` styling, `border border-destructive/20 bg-destructive/5 rounded-lg p-3` container
- `AlertDialogFooter` with:
  - Cancel button (AlertDialogCancel or regular Button with `variant="outline"`) that calls `handleKeychainCancel`, disabled during verification
  - Confirm/verify button (regular Button, NOT AlertDialogAction which auto-closes) that calls `handleKeychainVerify`, shows `Loader2` spinner when `verifying` is true, disabled during verification
  - The confirm button should use `variant="secondary"` per user spec ("should feel like a confirmation action, not a primary action")

IMPORTANT: Do NOT use `AlertDialogAction` for the confirm button because it auto-closes the dialog. Use a regular `Button` inside the footer instead, so the modal stays open on failure.

Do NOT use `onOpenChange` to set `showKeychainModal` directly -- only close via the explicit handlers to prevent race conditions between modal close and setting update (per RESEARCH.md pitfall 2).
