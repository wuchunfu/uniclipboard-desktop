# Plan 14-02 — Restore lifecycle status UI and add contract tests

<goal>
Restore the lifecycle status UI using the aligned DTO contract and add tests to lock down frontend/backend contract (DTO shape and serialization).
</goal>

<must_haves>

- Visible lifecycle status indicator in the UI reflecting `LifecycleStatusDto` state.
- UI responds correctly to key lifecycle states (e.g., uninitialized, initialized, running, shutting down).
- Tests verify JSON contract for `get_lifecycle_status` (including camelCase and error shapes) and frontend decoding.
  </must_haves>

<tasks>
  <task id="14-02-01" wave="2">
    <summary>Audit and restore lifecycle status UI locations</summary>
    <details>
      - Identify screens/components that should display lifecycle status (dashboard, settings, onboarding).
      - Determine how status should be rendered (icon, text, color) per state.
      - Sketch or document the intended UX based on current architecture guidelines.
    </details>
    <verification>
      - Short design note or checklist of components that will show lifecycle status.
    </verification>
  </task>

  <task id="14-02-02" wave="2">
    <summary>Implement lifecycle status UI using DTO</summary>
    <details>
      - Implement or update components to consume the aligned DTO types from Plan 14-01.
      - Bind UI state to lifecycle state enum or equivalent representation.
      - Ensure error cases (e.g., failed status fetch) are surfaced in a user-friendly way.
    </details>
    <verification>
      - Manual run showing lifecycle status changes in UI when backend state changes.
    </verification>
  </task>

  <task id="14-02-03" wave="2">
    <summary>Add backend tests for lifecycle status command contract</summary>
    <details>
      - Add integration tests that call `get_lifecycle_status` and assert DTO schema (field names, enum values).
      - Verify `CommandError` serialization matches the discriminated union used on the frontend.
      - Cover at least one success and one error path.
    </details>
    <verification>
      - New tests in Rust integration suite pass and would fail if DTO schema changes unexpectedly.
    </verification>
  </task>

  <task id="14-02-04" wave="2">
    <summary>Add frontend contract/decoder tests (if feasible)</summary>
    <details>
      - Add lightweight TypeScript tests that decode sample `LifecycleStatusDto` and `CommandError` payloads.
      - Ensure decoding uses DTO types from Plan 14-01 without unsafe casts.
      - Optionally add snapshot or schema checks to prevent regressions.
    </details>
    <verification>
      - Frontend test suite (or targeted script) passes and fails on contract mismatches.
    </verification>
  </task>
</tasks>

<verification_criteria>

- Lifecycle status UI is visible and reflects backend state.
