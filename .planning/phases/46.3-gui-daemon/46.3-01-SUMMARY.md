# Phase 46.3 Plan 01: GUI Daemon Compatibility Contract Summary

**Shared release version contract plus typed daemon compatibility probing, so GUI startup can distinguish “no daemon” from “wrong daemon” before any replacement logic runs**

## Accomplishments

- Added a shared workspace package version source and switched `uc-daemon` / `uc-tauri` to `version.workspace = true`, removing the old GUI-vs-daemon `0.4.0-alpha.1` vs `0.1.0` split.
- Changed daemon `/health` and `/status` DTOs to expose `packageVersion` and `apiRevision`, and added regression tests that assert camelCase output plus the absence of the legacy top-level `version` field.
- Reworked GUI bootstrap probing from `bool` health checks into explicit `ProbeOutcome` classification, covering absent endpoints, healthy-and-compatible responses, and incompatible HTTP/payload/version cases.

## Decisions Made

- Kept compatibility identity minimal and explicit: one shared package version plus one API revision, instead of ad hoc route probing.
- Treated malformed or legacy health payloads as incompatible daemons, not as temporarily absent services.
- Preserved `DaemonBootstrapError::IncompatibleDaemon` as the bootstrap-facing error while sourcing it from typed probe outcomes rather than collapsing everything into `Probe`.
