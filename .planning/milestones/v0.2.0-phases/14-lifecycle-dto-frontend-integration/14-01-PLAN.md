# Plan 14-01 — Align frontend DTO with LifecycleStatusDto

<goal>
Ensure the frontend consumes `LifecycleStatusDto` and `CommandError` DTOs (not domain models) for lifecycle status, with TypeScript types and IPC wiring aligned to the backend contract.
</goal>

<must_haves>

- Frontend TypeScript types reflect `LifecycleStatusDto` and `CommandError` contracts (no direct domain model usage).
- IPC handlers return DTO-shaped JSON (camelCase fields) consistent with backend.
- Existing lifecycle flows compile and type-check against the new DTO types.
  </must_haves>

<tasks>
  <task id="14-01-01" wave="1">
    <summary>Inventory existing frontend lifecycle status usage</summary>
    <details>
      - Search frontend codebase for lifecycle status usage (components, hooks, API modules).
      - Identify all places that depend on lifecycle status or related commands.
      - Document any direct usage of domain models or legacy response shapes.
    </details>
    <verification>
      - Short note listing all files/modules where lifecycle status is consumed.
    </verification>
  </task>

  <task id="14-01-02" wave="1">
    <summary>Define/confirm TypeScript DTOs for LifecycleStatusDto and CommandError</summary>
    <details>
      - Based on backend `LifecycleStatusDto` and `CommandError` definitions, declare TypeScript interfaces/types.
      - Ensure discriminated union structure for `CommandError` (code/message) matches backend serde config.
      - Place DTO types in a shared location (e.g., `src/api/types` or equivalent) to be reused across components.
    </details>
    <verification>
      - TypeScript file with DTO definitions type-checks and is imported from at least one consumer.
    </verification>
  </task>

  <task id="14-01-03" wave="1">
    <summary>Align IPC/command wiring to emit DTOs</summary>
    <details>
      - Inspect Tauri command handling for lifecycle status (`get_lifecycle_status` or equivalent).
      - Ensure the command returns serialized `LifecycleStatusDto` and `CommandError` objects, not domain types.
      - Update any legacy response mapping to match the DTO shape (including camelCase field names).
    </details>
    <verification>
      - Manual check or small integration test that command JSON matches expected DTO schema.
    </verification>
  </task>

  <task id="14-01-04" wave="1">
    <summary>Refactor frontend callers to use DTOs</summary>
    <details>
      - Update all frontend usages from task 14-01-01 to depend on the new DTO types.
      - Replace any stringly-typed state checks with enum/discriminated-union-friendly checks if available.
      - Ensure error handling branches use `CommandError` shape (code/message) instead of raw strings.
    </details>
    <verification>
      - Frontend TypeScript build succeeds with the new DTO types and no implicit `any` related to lifecycle status.
    </verification>
  </task>
</tasks>

<verification_criteria>

- All lifecycle-related frontend modules import and use DTO types.
- Command handlers and frontend agree on JSON shape for lifecycle status and errors.
- TypeScript checks pass without casting away types for lifecycle flows.
  </verification_criteria>
