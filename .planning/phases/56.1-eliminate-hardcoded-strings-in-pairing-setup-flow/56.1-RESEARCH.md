# Phase 56.1: Eliminate Hardcoded Strings in Pairing/Setup Flow - Research

**Researched:** 2026-03-25
**Domain:** Rust constant extraction / wire-protocol string deduplication across uc-daemon, uc-daemon-client, uc-core
**Confidence:** HIGH

## Summary

Phase 56.1 targets a concrete, bounded refactoring problem: hardcoded string literals that exist independently on both sides of a wire protocol (daemon server side and daemon-client consumer side). When these strings drift, messages are silently dropped or misrouted — no compiler catches the mismatch. The goal is to extract these into shared `const` values in `uc-core` (the only crate that both `uc-daemon` and `uc-daemon-client` can import without a dependency cycle), then replace every usage site.

The prior analysis is accurate and complete. All five categories of strings were confirmed present in the live codebase. The codebase already has a well-established model for this pattern: `protocol_ids.rs` in `uc-core/src/network/` defines wire-protocol identifiers as a typed enum with a `const fn as_str()`. This is the exact pattern to extend.

**Primary recommendation:** Create `uc-core/src/network/daemon_api_strings.rs` (or equivalent module) to house all wire-protocol string constants. Migrate usage sites in three waves: (1) WS topic/event strings, (2) pairing session state labels, (3) HTTP path constants. Keep the frontend TypeScript strings in scope but treat them as low-risk because they are consumed via camelCase event type dispatch which the bridge already encapsulates.

No external dependencies are needed — this is a pure source-level refactoring within the existing workspace.

## Architecture Patterns

### Where to Place Constants

The only crate that both `uc-daemon` (server) and `uc-daemon-client` (consumer) can import without creating a cycle is `uc-core`. Both already depend on `uc-core`. This is confirmed by:

- `uc-daemon/src/pairing/host.rs` uses `uc_core::*`
- `uc-daemon-client/src/ws_bridge.rs` imports `uc_core::ports::realtime::*`

The existing precedent is `uc-core/src/network/protocol_ids.rs`:

```rust
// Source: src-tauri/crates/uc-core/src/network/protocol_ids.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ProtocolId {
    Pairing,
    PairingStream,
    Business,
    FileTransfer,
}

impl ProtocolId {
    pub const fn as_str(&self) -> &'static str {
        match self {
            ProtocolId::Pairing => "/uc-pairing/1.0.0",
            // ...
        }
    }
}
```

For wire-protocol strings in Phase 56.1, plain `pub const` is preferred over an enum because the strings are not used as match arms in a type-safe dispatch — they are compared at runtime with `.as_str()` or matched in `match event.event_type.as_str()`. An enum wrapper would add complexity without safety benefit at the usage sites.

### Anti-Patterns to Avoid

- **Putting constants in `uc-daemon`:** `uc-daemon-client` would then depend on `uc-daemon`, which creates a unidirectional coupling that already exists (client imports daemon's DTOs) but should not be extended to constants. The current uc-daemon-client dependency on uc-daemon is for shared DTOs only.
- **String enums with `Display`:** Overcomplicated for constants that are only ever used as `&str` literals in JSON payloads.
- **One giant constants file:** Use submodules (`ws_topic`, `ws_event`, `pairing_stage`, etc.) to aid discoverability and grep-ability.

## Don't Hand-Roll

| Problem                 | Don't Build            | Use Instead                              | Why                                              |
| ----------------------- | ---------------------- | ---------------------------------------- | ------------------------------------------------ |
| Constant registry       | Custom registry struct | `pub const &str`                         | Zero runtime cost, works across crate boundaries |
| Type-safe event routing | New enum hierarchy     | Existing `RealtimeEvent` enum in uc-core | Already exists and is correct                    |
| HTTP path builder       | Path template DSL      | Plain `pub const &str`                   | Paths are static; no interpolation needed        |

### Group G: Frontend camelCase Event Names — LOW RISK (out of scope)

**`src/api/p2p.ts` and `src/api/setup.ts`:** These match `event.type` values like `'pairing.updated'`, `'pairing.verificationRequired'`, `'peers.changed'`, `'setup.stateChanged'`, `'setup.spaceAccessCompleted'`.

Note the casing inconsistency: Rust daemon emits `"pairing.verification_required"` (snake_case) but the bridge in `host_event_emitter.rs` emits `"pairing.verificationRequired"` (camelCase) to the frontend. These are different paths:

- **Daemon WS → DaemonWsBridge → RealtimeEvent → host_event_emitter.rs → frontend:** camelCase event types
- **Daemon WS raw events:** snake_case event types from `ws.rs` and `host.rs`

The frontend TypeScript strings are coupled to `host_event_emitter.rs` output, not to `ws.rs` constants directly. Extracting them would require TypeScript constant files — out of scope for this phase which targets Rust crates.

## Common Pitfalls

### Pitfall 1: Dependency Direction Violation

**What goes wrong:** Constants placed in `uc-daemon` instead of `uc-core`, then `uc-daemon-client` imports them, creating a reverse-direction import (client → server).
**Why it happens:** `uc-daemon` already defines the constants so it's tempting to just make them `pub`.
**How to avoid:** Always place shared wire-protocol constants in `uc-core`. The crate dependency graph is: `uc-daemon → uc-core ← uc-daemon-client`.
**Warning signs:** `cargo check` will fail if `uc-daemon-client` tries to import from `uc-daemon` and `uc-daemon` depends on `uc-daemon-client`.

### Pitfall 2: Missing the `host.rs` Inline Call Sites

**What goes wrong:** Constants are extracted to `uc-core` and the `ws.rs` module-level constants are replaced, but the inline strings in `emit_ws_event()` calls inside `host.rs` are missed.
**Why it happens:** There are two locations in the same crate — the `ws.rs` const block and the `host.rs` inline emit calls.
**How to avoid:** After adding constants, do a workspace-wide `cargo build` then grep for remaining string literals matching the patterns.
**Warning signs:** Remaining occurrences of `"pairing/session"`, `"pairing/verification"`, `"space-access"` as string literals inside `host.rs`.

### Pitfall 3: Double-Definition of Busy Reasons

**What goes wrong:** `ensure_inbound_admitted()` (struct method) and `handle_pairing_message()` (standalone fn in same file) both contain the same busy reason strings. If only one site is updated, the mismatch is re-introduced internally.
**Why it happens:** The struct method delegates to `reject_inbound_request()` but the standalone fn calls `reject_inbound_request()` directly — both need the same string.
**How to avoid:** Replace both sites; verify with grep after the change.

### Pitfall 4: WS Event Type Casing in map_daemon_ws_event()

**What goes wrong:** The event type strings emitted by the daemon (`"pairing.verification_required"`, `"setup.state_changed"`) use **snake_case**, while the frontend event types dispatched by `host_event_emitter.rs` use **camelCase** (`"pairing.verificationRequired"`, `"setup.stateChanged"`). These are two different layers.
**Why it happens:** `ws_bridge.rs` consumes daemon events and converts to `RealtimeEvent` (typed). Then `host_event_emitter.rs` re-serializes to frontend JSON with camelCase. The constants in `uc-core` must match the **daemon wire format** (snake_case), not the frontend format.
**How to avoid:** Keep the constant module focused on the daemon-to-bridge layer. Do not attempt to unify the frontend camelCase event names at this phase.

### Pitfall 5: HTTP Paths with Dynamic Segments

**What goes wrong:** Routes like `/pairing/sessions/:session_id/verify` cannot be shared as simple `pub const` because the client uses `/pairing/sessions/{session_id}/verify` with Rust's `format!()`.
**How to avoid:** Only extract the static base paths as constants. For dynamic segments, extract the static prefix (`"/pairing/sessions/"`) and keep the `format!()` at the call site, or skip HTTP path extraction for those specific routes.

## Open Questions

1. **HTTP path constants: full extraction or only high-risk routes?**
   - What we know: HTTP paths are duplicated between `routes.rs` (server) and `http/pairing.rs` (client) for about 9 routes.
   - What's unclear: Routes with dynamic segments (`/pairing/sessions/:session_id/verify`) cannot be extracted as simple string constants; they need format strings.
   - Recommendation: Extract static paths only. For dynamic-segment paths, define a base path constant and keep `format!()` at the call site. Prioritize WS strings (Groups A, B, C) over HTTP paths (Group F).

2. **Should `acknowledged()` state values in routes.rs be replaced?**
   - What we know: `acknowledged(session_id, true, "request")` in `initiate_pairing` uses `"request"` as an inline string.
   - What's unclear: These are HTTP response fields, not WS wire fields — lower mismatch risk.
   - Recommendation: Yes, use `pairing_stage::REQUEST` for consistency, but treat as lower priority than WS event strings.
