# Plan 46-02: Daemon Pairing Control Surface And Realtime Contract

## Goal

Add the daemon-owned control surface and realtime contract that Tauri will consume in Phase 46, while preserving the metadata boundary for ordinary session reads and reserving verification details for authenticated realtime delivery only.

- `InitiatePairingRequest { peer_id: String }`
- `VerifyPairingRequest { pin_matches: bool }`

- `SetPairingParticipantRequest { client_kind: String, ready: bool, lease_ttl_ms: Option<u64> }`

Update `src-tauri/crates/uc-daemon/src/api/server.rs` so `DaemonApiState` includes one daemon pairing host command facade/handle shared by HTTP routes and websocket fanout. Avoid route-local host state.

Update `src-tauri/crates/uc-daemon/src/api/routes.rs` to add these authenticated routes:

- `PUT /pairing/discoverability/current`
- `PUT /pairing/participants/current`
- `POST /pairing/sessions`
- `POST /pairing/sessions/:session_id/accept`
- `POST /pairing/sessions/:session_id/reject`
- `POST /pairing/sessions/:session_id/cancel`
- `POST /pairing/sessions/:session_id/verify`

All mutation routes must dispatch into daemon host methods with the same explicit names:

- `set_discoverability`
- `set_participant_ready`
- `initiate_pairing`
- `accept_pairing`
- `reject_pairing`
- `cancel_pairing`
- `verify_pairing`

Use these concrete HTTP semantics so Tauri does not have to rediscover them:

- successful mutation acknowledgment returns `202 Accepted` and `AckedPairingCommandResponse`
- active-session conflict returns `409` with `{"error":"active_pairing_session_exists"}`
- no-local-participant-ready returns `412` with `{"error":"no_local_pairing_participant_ready"}`
- non-discoverable host returns `409` with `{"error":"host_not_discoverable"}`
- missing session for follow-up mutation returns `404`
- malformed request body returns `400`

`POST /pairing/sessions` must accept only `peerId`; every follow-up route must key off `:session_id` and must not accept a replacement `peerId` field.

The control surface must preserve the product rule for CLI-driven pairing:

- daemon-only / ordinary CLI startup is non-discoverable
- CLI pairing mode must explicitly call `PUT /pairing/discoverability/current` before the daemon appears in peer discovery
- `PUT /pairing/participants/current` alone is insufficient to make a headless daemon discoverable
- GUI bridge may set discoverability during startup because desktop shell presence is the user-facing participant model for Phase 46

Control recovery rules are mandatory:

- discoverability and readiness registrations are lease-based (`lease_ttl_ms`) and must auto-expire
- bridge/CLI teardown must explicitly revoke discoverability/readiness (`discoverable=false`, `ready=false`) when possible
- lost client lease must not keep daemon discoverable indefinitely

- `pairing.updated`
- `pairing.verification_required`
- `pairing.complete`

Extend `src-tauri/crates/uc-daemon/src/api/ws.rs` to emit incremental discovery types used by the Tauri bridge:

- `peers.changed`
- `peers.name_updated`
- `peers.connection_changed`
- `paired-devices.changed`

Use one daemon-side broadcast path so authenticated websocket subscribers receive these incremental events after the initial `pairing.snapshot`. `pairing.snapshot` must continue to come from `RuntimeState` summaries. Verification payloads must come only from authenticated realtime fanout, not from `DaemonQueryService::pairing_session(...)`.

Add `src-tauri/crates/uc-daemon/tests/pairing_ws.rs` with tests for:

- snapshot contains `sessionId` and omits verification secrets
- incremental verification event contains `code`, `localFingerprint`, and `peerFingerprint`
- peers and paired-devices incremental events carry the fields required for `p2p-peer-discovery-changed`, `p2p-peer-name-updated`, and `p2p-peer-connection-changed` bridge mapping
- websocket event uses `type`, not `event_type`
- `/pairing/sessions/:session_id` response omits verification secrets even while realtime verification event is available

Field-level compatibility is part of the contract for bridge-facing discovery DTOs:

- `PeerChangedPayload` must serialize `peerId`, `deviceName`, `addresses`, `discovered`, and `connected`
- `PeerChangedPayload` must use `discovered=false` as the canonical lost/remove signal so the bridge can remove a device from the frontend discovery list without guessing
- `PeerNameUpdatedPayload` must serialize `peerId` and `deviceName`
- `PeerConnectionChangedPayload` must serialize `peerId`, `deviceName`, and `connected`
- all websocket payload structs used by the bridge must emit camelCase keys and must not leak snake_case keys
