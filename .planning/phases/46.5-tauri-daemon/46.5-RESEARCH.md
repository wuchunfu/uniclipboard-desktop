## User Constraints (from CONTEXT.md)

### Locked Decisions

### Runtime Ownership

- 在 `uc-bootstrap` 的平台层引入一个小而明确的 ownership/config 类型，表达“GUI 不拥有 pairing runtime，daemon 拥有 pairing runtime”。
- `Libp2pNetworkAdapter::new(...)` 增加显式配置参数，声明是否启用本地 pairing runtime。
- 不拆分 bootstrap 为两套 wiring；继续共享 composition root，但在创建 `Libp2pNetworkAdapter` 时显式传入 ownership 配置。

### GUI Boundary

- GUI 模式下 `Libp2pNetworkAdapter` 不初始化 `PairingStreamService`。
- GUI 模式下不启动 pairing accept loop，不注册 `/uniclipboard/pairing-stream/1.0.0`。
- GUI 模式下 `NetworkPorts.pairing` 不再绑定到 GUI 本地 `libp2p_network` 的真实 pairing transport。
- setup 阶段只允许通过 daemon-backed facade 与 daemon 交互，不允许回落到本地 pairing transport。
- Tauri commands 中配对相关命令只做 daemon route 调用与错误映射，不得直接驱动本地配对业务对象。

### Daemon Ownership

- daemon 保留当前 pairing stream、pairing host、pairing orchestrator，继续作为唯一配对入口。
- daemon 继续负责 inbound request、verification、complete、failed 的 WS/realtime 投递。
- GUI 的唯一配对业务事件源仍然是 daemon realtime，而不是本地 pairing 网络事件。

### Tauri Cleanup

- `bootstrap/wiring.rs` 中当前对本地 `PairingMessageReceived` / `PairingFailed` 的忽略逻辑不应继续作为运行时兜底；GUI 路径不应再收到这类事件。
- 任何依赖 `deps.network_ports.pairing` 承载 GUI 本地 pairing 业务的路径，都改为 daemon client / daemon realtime。
- 任何仅为本地 pairing runtime 存在的 Tauri 层桥接、日志、注释、回退路径，全部删除。

### Interface And Behavior Lock

- 不改变前端 wire contract；前端仍消费 `daemon://realtime`，包括：
  - `pairing.updated`
  - `pairing.verificationRequired`
  - `pairing.complete`
  - `pairing.failed`
- 允许 `uc-tauri` 保留配对命令 API 外壳，但这些命令只能调用 daemon，不得拥有本地业务状态。

### Claude's Discretion

- ownership/config 类型的具体命名与落点，只要语义足够明确且不破坏分层
- GUI 模式下 `NetworkPorts.pairing` 的具体装配形式，只要能够彻底切断真实本地 pairing transport
- 清理遗留代码时的模块边界和删除顺序，只要最终不再保留 GUI 本地 pairing runtime

### Deferred Ideas (OUT OF SCOPE)

- 重做前端配对交互或改动前端事件契约
- 拆分两套独立 bootstrap / wiring 实现
- 保留任何 GUI 本地 pairing 兜底模式
- 把 daemon pairing ownership 再次回退给 Tauri

## Summary

当前代码已经完成了 daemon 作为配对主宿主的大部分迁移，但还没有形成真正的“硬边界”。`uc-daemon` 的 `DaemonPairingHost`、HTTP pairing/setup routes、以及 `DaemonWsBridge` 已经是主线；同时，GUI 侧仍然会在共享 bootstrap 中创建真实 `Libp2pNetworkAdapter` pairing runtime，`uc-tauri::bootstrap::wiring.rs` 仍显式接收并忽略本地 `PairingMessageReceived` / `PairingFailed`，而 `commands/pairing.rs`、`commands/setup.rs` 仍有多条本地 use case 路径。只删 Tauri wiring 里的事件忽略逻辑不够，真正的切口必须在 `uc-bootstrap` 和 `uc-platform`。

研究得到的关键结论是：Phase 46.5 应把“pairing runtime 是否存在”做成 composition-root 级别的显式能力，而不是在 Tauri 层做软忽略。GUI builder 必须传入“shell-only” ownership/config，让 `Libp2pNetworkAdapter` 在 GUI 模式下完全不创建 `PairingStreamService`、不启动 accept loop、也不暴露真实本地 `PairingTransportPort`。与此同时，Tauri 命令面必须补全 daemon-only 收口，不能只保留 initiate/accept/reject/verify 这些 mutation，读接口、setup-pairing 相关命令、以及受本地 pairing transport 影响的残留路径都要一并审查。

另一个高价值发现是 setup 侧仍有隐藏的本地 pairing 依赖。`uc_app::usecases::setup::SetupActionExecutor` 里 `start_join_space_access()` 仍直接调用 `pairing_transport.open_pairing_session(...)`。这说明如果 planner 只切断 `NetworkPorts.pairing` 而不把 setup command shell 也切到 daemon，GUI setup 会在 Phase 46.5 自己打断自己。由于 Phase 47 才是前端直接切到 daemon API，本阶段最稳妥的方案是保留前端现有 Tauri command contract，但让 `uc-tauri` 的 pairing/setup-pairing commands 成为 daemon HTTP client 包装层。

**Primary recommendation:** 在 `uc-bootstrap`/`uc-platform` 引入显式 pairing ownership config，GUI 模式彻底不创建本地 pairing runtime，并把 `uc-tauri` 的 pairing 与 setup-pairing command surface 收口为 daemon-only wrappers，继续以 `DaemonWsBridge` 作为唯一前端事件源。

## Architecture Patterns

### Recommended Project Structure

```text
src-tauri/crates/
├── uc-bootstrap/src/          # runtime ownership config + GUI/daemon builders
├── uc-platform/src/adapters/  # libp2p transport + pairing runtime toggle
├── uc-daemon/src/
│   ├── api/                   # pairing/setup HTTP + WS routes
│   └── pairing/               # authoritative pairing host/session lifecycle
└── uc-tauri/src/
    ├── commands/              # daemon-only wrappers and error mapping
    └── bootstrap/             # DaemonWsBridge, setup hub, no local pairing loops
```

### Pattern 1: Explicit Runtime Ownership at Composition Root

**What:** 在共享 `build_gui_app()` / `build_daemon_app()` builder 中显式传入 pairing ownership/config，而不是在 `uc-tauri` 层靠条件分支“忽略本地 pairing 事件”。

**When to use:** 当同一个 `Libp2pNetworkAdapter` 被 GUI 和 daemon 复用，但两种 runtime mode 的 pairing capability 必须完全不同。

**Example:**

```rust
// Source: repository pattern in src-tauri/crates/uc-bootstrap/src/builders.rs
enum PairingRuntimeOwnership {
    GuiShell,
    DaemonHost,
}

let ownership = PairingRuntimeOwnership::GuiShell;
let libp2p = Libp2pNetworkAdapter::new(
    identity_store,
    policy_resolver,
    encryption_session,
    transfer_decryptor,
    transfer_encryptor,
    file_cache_dir,
    ownership,
)?;
```

### Pattern 2: Daemon-Only Command Shell

**What:** `uc-tauri::commands::pairing` 和必要的 `uc-tauri::commands::setup` 只负责 `request DTO -> daemon client -> response DTO/error`，不直接调用本地 pairing/setup business object。

**When to use:** 所有配对 mutation、setup pairing mutation、以及在切断 GUI pairing transport 后会失效的 setup action。

**Example:**

```rust
// Source: repository pattern in src-tauri/crates/uc-tauri/src/commands/pairing.rs
#[tauri::command]
async fn accept_p2p_pairing(
    session_id: String,
    daemon_connection: tauri::State<'_, DaemonConnectionState>,
) -> Result<(), CommandError> {
    TauriDaemonPairingClient::new(daemon_connection.inner().clone())
        .accept_pairing(&session_id)
        .await
        .map_err(|error| CommandError::InternalError(error.to_string()))
}
```

### Pattern 3: Single Realtime Source for Frontend and Setup Hub

**What:** pairing/setup 相关 frontend 事件只允许来自 `DaemonWsBridge`，setup 内部订阅继续走 `SetupPairingEventHub`，不再存在 GUI 本地 p2p event 桥。

**When to use:** 所有 `pairing.updated` / `pairing.verificationRequired` / `pairing.complete` / `pairing.failed` / `setup.stateChanged` 路径。

**Example:**

```rust
// Source: repository pattern in src-tauri/crates/uc-tauri/src/bootstrap/realtime_runtime.rs
start_realtime_runtime(
    daemon_connection_state,
    event_emitter,
    setup_pairing_event_hub,
    task_registry,
).await;
```

### Anti-Patterns to Avoid

- **Ignore-only cleanup:** 继续保留 GUI 本地 `PairingStreamService`，只是像 `bootstrap/wiring.rs` 那样忽略 `PairingMessageReceived` / `PairingFailed`。这不是切边界，只是把残留业务藏起来。
- **Local reopen after cutover:** 切断 GUI pairing runtime 后仍让 `SetupActionExecutor` 调 `pairing_transport.open_pairing_session(...)`。这会把 setup join 流直接打成 `pairing service not initialized`。
- **Partial command migration:** 只把 initiate/accept/reject/verify 迁到 daemon，读接口、`unpair`、setup-pairing commands 仍走本地 use cases。最终得到的是两套 truth。
- **Second transport host:** 为 GUI 再补一条本地 websocket/bridge/fallback 维持兼容。46.1/46.2 已明确 realtime 和 pairing host 只能有一条主路径。

## Don't Hand-Roll

| Problem                      | Don't Build                                                                | Use Instead                                                         | Why                                                          |
| ---------------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------ |
| runtime ownership 判定       | 到处散落 `if is_gui { ... }` / 环境变量判断                                | `uc-bootstrap` 注入的 typed ownership/config                        | 一处表达、全链可测、不会让 GUI/daemon 因后续代码漂移重新混线 |
| pairing realtime fanout      | 新的本地 bridge、额外 websocket、直接 `app.emit()` pairing 事件            | `DaemonWsBridge` + realtime consumers + `SetupPairingEventHub`      | 已有订阅、重连、背压、topic fanout 语义，不能再造第二套      |
| setup transport wrapper      | 在 `uc-tauri` 新手写一份和 CLI 不同的 setup 协议                           | 复用/抽取 `uc-cli::DaemonHttpClient` 的 `/setup/*` 模式             | daemon setup routes 已存在，复用最稳，最不容易与 CLI 漂移    |
| GUI pairing transport “断路” | 宽泛的 placeholder network、`Option<Arc<_>>` null 检查、到处 `if let Some` | 细粒度 disabled pairing adapter，或直接让相关 command 全部走 daemon | 保持 app-layer trait 边界稳定，同时让失败模式明确、可测      |

**Key insight:** 真正难的不是“让前端看不到本地 pairing 事件”，而是“让 GUI runtime 根本没有本地 pairing runtime 可用”。前者是表象；后者才是结构性修复。

## Common Pitfalls

### Pitfall 1: Half-Disabled GUI Pairing Runtime

**What goes wrong:** GUI 不再向前端发本地 pairing 事件，但仍然注册 `/uniclipboard/pairing-stream/1.0.0` 并启动 accept loop，结果形成隐藏的第二宿主。

**Why it happens:** 代码清理停留在 `uc-tauri::bootstrap::wiring.rs` 的事件忽略层，没有切到 `Libp2pNetworkAdapter::spawn_swarm()` 的 runtime creation 层。

**How to avoid:** 让 GUI builder 在创建 `Libp2pNetworkAdapter` 时显式禁用 pairing runtime；不创建 `PairingStreamService`，不调用 `spawn_accept_loop()`。

**Warning signs:** GUI 日志仍出现 pairing session started / pairing protocol listen 痕迹；`NetworkEvent::PairingMessageReceived` 在 GUI runtime 仍能收到。

### Pitfall 2: Setup Flow Quietly Re-Enters Local Pairing Transport

**What goes wrong:** setup 入口表面上已经 daemon-backed，但 `SetupActionExecutor::start_join_space_access()` 仍直接 `open_pairing_session()`，切断 GUI pairing runtime 后 setup join 直接回归失败。

**Why it happens:** 46.4 的 daemon setup flow 已存在，但 GUI setup commands 还没有收口到 daemon setup routes，本地 orchestrator 仍然持有 pairing transport。

**How to avoid:** 本 phase 要么把 `commands/setup.rs` 的 pairing-sensitive actions 包装到 daemon `/setup/*`，要么把本地 `SetupActionExecutor` 的 pairing reopen 全部替换为 daemon-backed path。第一种更符合“thin shell”目标。

**Warning signs:** join flow 在 `ProcessingJoinSpace` 后报 `pairing service not initialized`，或 GUI setup 只有在本地 pairing runtime 开着时才工作。

### Pitfall 3: Pairing Command Surface Only Migrated Halfway

**What goes wrong:** initiate/accept/reject/verify 已走 daemon，但 `list_paired_devices`、`get_paired_peers_with_status`、`unpair_p2p_device` 仍然走本地 use case。

**Why it happens:** migration 聚焦 mutation surface，忽略 read/unpair 也属于 pairing command surface。

**How to avoid:** planning 时把 `commands/pairing.rs` 全量过一遍，区分“可继续本地”的纯设置类能力和“必须 daemon-owned”的 pairing business surface。当前代码下 `unpair` 明显不能留本地。

**Warning signs:** GUI pairing 成功，但 paired devices 快照与 daemon `/paired-devices` 不一致；切断 GUI pairing transport 后 `unpair` 立即坏掉。

### Pitfall 4: Frontend Wire Contract Regression

**What goes wrong:** 切 daemon-only shell 时顺手改了事件名、topic 或 payload casing，导致前端静默失效。

**Why it happens:** 后端 DTO 和前端 TypeScript contract 没有一起验证，尤其是 `camelCase` 键名和 `daemon://realtime` listener name。

**How to avoid:** 保持 `pairing.updated` / `pairing.verificationRequired` / `pairing.complete` / `pairing.failed` 不变，并继续使用 `#[serde(rename_all = "camelCase")]`。

**Warning signs:** 前端 listener 还在，但 `payload.sessionId` 变成 `undefined`；`src/api/__tests__/p2p-realtime-contract.test.ts` 回归失败。

### Pitfall 5: Discoverability / Readiness Ownership Drift

**What goes wrong:** GUI 一边继续维护本地 participant state，一边 daemon 也在维护 lease，最终 discoverability/readiness 出现双写。

**Why it happens:** 把 GUI lease keeper 误当成 pairing host ownership，本质上它只是 daemon 的一个 client 行为，不是本地 pairing runtime。

**How to avoid:** GUI 只保留 daemon lease renewal，不保留任何本地 pairing host state；有冲突时 daemon truth 必须获胜。

**Warning signs:** daemon `/pairing/gui/lease` 已禁用，但 GUI 仍能在本地接收入站 pairing；或 GUI UI 状态与 daemon pairing state 分叉。

struct MyState(String);

fn my_custom_command(state: tauri::State<MyState>) {

### Emit Structured Frontend Events from Rust

```rust
// Source: https://v2.tauri.app/develop/calling-frontend
use serde::Serialize;
use tauri::{AppHandle, Emitter};

#[derive(Clone, Serialize)]
#[serde(rename_all = "camelCase")]
struct DownloadStarted<'a> {
    url: &'a str,
    download_id: usize,
}

fn emit_event(app: AppHandle, url: String) {
    let payload = DownloadStarted {
        url: &url,
        download_id: 1,
    };
    let _ = app.emit("download-started", payload);
}
```

## Open Questions

1. **`unpair` 要不要在本 phase 一起 daemon 化？**
   - What we know: `uc_app::usecases::pairing::UnpairDevice` 直接依赖 `PairingTransportPort::unpair_device()`；一旦 GUI pairing transport 被切断，当前 `unpair_p2p_device` 会变成坏路径。
   - What's unclear: daemon 目前已有 `/pairing/*` mutation 和 `/paired-devices` query，但还没有现成的 unpair route。
   - Recommendation: 规划阶段默认把 unpair 纳入 46.5，并在 daemon 增加 unpair mutation surface；否则就必须显式把它标记为 deferred，并在 GUI pairing cut 后留下 TODO 和行为保护。

2. **`commands/setup.rs` 是全量转 daemon shell，还是只转 pairing-sensitive actions？**
   - What we know: daemon 已提供完整 `/setup/*`；GUI 当前仍调用本地 `SetupOrchestrator`，而其 action executor 仍会触发本地 pairing transport reopen。
   - What's unclear: Phase 46.5 是否只要求“setup pairing”收口，还是直接把整个 Tauri setup command shell 统一切成 daemon-backed。
   - Recommendation: 直接把 `commands/setup.rs` 全量转 daemon shell。这样 46.5 完成“backend ownership hard cut”，47 再做 frontend transport cutover，阶段边界最清晰。
