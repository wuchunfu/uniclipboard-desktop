## User Constraints (from CONTEXT.md)

### Locked Decisions

### 兼容判定契约

- GUI 与 daemon 视为同一发布单元，当前 GUI 打包版本与运行中 daemon 版本只要不一致，就判定为不兼容。
- 版本号一致也不代表可接受；如果运行中 daemon 缺少 GUI 依赖的关键能力、关键路由或关键契约，仍然判定为不兼容。
- `/health` 返回异常格式、读不到可信版本信息、或健康响应不可信时，不按“暂时故障后继续尝试兼容”处理，而是直接归入不兼容分支。
- Phase 46.3 的升级哲学是“同包一致优先”，不是“尽量兼容旧 daemon”。

### 不兼容 daemon 的替换策略

- 当 GUI 在 UniClipboard 预期的本地 daemon 端点上发现不兼容 daemon 时，允许 GUI 自动终止旧 daemon 并拉起当前版本 daemon。
- 自动替换的边界限定在 UniClipboard 自己的预期本地端点上；本阶段不扩展为通用进程管理器，也不处理其他地址/其他服务。
- 自动替换成功时保持静默，不额外打扰用户；只在替换失败时显式告知。
- 一次自动替换流程只允许有界重试；若终止旧 daemon 或启动新 daemon 仍失败，必须收敛到明确错误态，而不是无限后台重试。

### 启动失败与恢复体验

- 如果 GUI 最终无法获得健康且兼容的 daemon，不应静默进入主界面；应停在专门的启动错误页。
- 错误提示必须是持久、明确、可执行恢复动作的状态，不接受仅用一次性 toast 掩盖问题。
- 恢复路径同时提供“有界自动重试”和“用户手动重试入口”。
- 如果应用在错误页期间检测到 daemon 已恢复健康并兼容，GUI 应自动继续正常启动流程并切回主界面，不要求用户重启应用。

- 启动错误页的具体文案、视觉样式与重试节奏
- 追踪与日志字段的细节，以及与 `StartupBarrier` / `DaemonConnectionState` 的集成方式

## Summary

46.3 不是“给现有 health probe 多加一个 if”这么小。当前 GUI 启动链路把 daemon bootstrap 当作旁路任务处理，失败只记日志；窗口显示与 `StartupBarrier` 完成只受 backend 初始化影响，不受 daemon 健康/兼容结果约束。[`src-tauri/src/main.rs`](src-tauri/src/main.rs) 里 `bootstrap_daemon_connection()` 的失败不会阻止主窗口展示，而 `StartupBarrier` 也没有 daemon failure 状态。这与本阶段锁定决策直接冲突。

当前兼容判定也没有可落地的“同包一致”来源。`/health.version` 和 `/status.version` 来自 `uc-daemon` crate 自己的 `env!("CARGO_PKG_VERSION")`，而 `crates/uc-daemon/Cargo.toml` 仍是 `0.1.0`；GUI 打包版本在 `src-tauri/tauri.conf.json` 是 `0.4.0-alpha.1`。如果 planner 直接按字符串比较，会永久判定不兼容。因此，本阶段首先要建立共享发布版本契约，并把“能力/契约版本”显式暴露为健康探针的一部分。

自动替换也不能只靠当前 `std::process::Command::spawn()`。现有实现既不保留 child handle，也没有持久化 ownership metadata，无法对“上一次 GUI/旧包留下的 daemon”做受控终止。要实现“只接管 UniClipboard 预期端点上的旧 daemon”，必须补一个窄范围 ownership/replacement 机制，并把失败收敛到启动错误页。

**Primary recommendation:** 保持 46.3 在现有 Rust bootstrap 链路内完成，不切换到 Tauri shell sidecar；优先补“共享版本/能力契约 + 窄范围 daemon ownership/replacement state + startup error state machine”三件事。

## Architecture Patterns

### Recommended Project Structure

```text
src-tauri/
├── crates/uc-tauri/src/bootstrap/
│   ├── run.rs                  # bootstrap state machine + compatibility probe
│   ├── runtime.rs              # managed state: connection + daemon ownership
│   └── startup_state.rs        # new: startup error/readiness DTO if split needed
├── crates/uc-daemon/src/api/
│   ├── types.rs                # health/status DTO adds packageVersion/apiRevision
│   └── query.rs                # fill shared build metadata
└── src/main.rs                 # window-show gating + retry/recovery bootstrap loop

src/
├── pages/StartupErrorPage.tsx  # persistent startup failure surface
├── api/startup.ts              # startup status/retry bridge
└── hooks/useStartupStatus.ts   # auto-retry + recovery listener
```

### Pattern 1: Compatibility Before Readiness

**What:** 把 daemon bootstrap 从“健康即可继续”改成“无监听 -> spawn；有监听 -> compatibility probe；兼容才进入 ready”。

**When to use:** GUI setup 阶段决定是否显示主窗口时。

**Example:**

```rust
// Source: local repo + Phase 46.3 constraints
enum ProbeOutcome {
    Absent,
    Compatible(DaemonConnectionInfo),
    Incompatible { reason: String, observed_version: Option<String> },
}

// startup order
// 1. probe expected endpoint
// 2. if absent => spawn current daemon, wait healthy+compatible
// 3. if incompatible => bounded replace flow
// 4. only Compatible may mark startup ready
```

### Pattern 2: Separate Version Identity from Capability Revision

**What:** 兼容判定至少要区分两个维度：

- 发布包版本: 是否与当前 GUI 同包
- 契约修订: 是否具备 GUI 依赖的关键 HTTP/WS/startup 契约

**When to use:** `/health` / `/status` DTO 设计与 replacement decision。

**Example:**

```rust
// Source: local repo (`DAEMON_API_REVISION`) + planning recommendation
#[derive(Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct HealthResponse {
    pub status: String,
    pub package_version: String,
    pub api_revision: String,
}
```

### Pattern 3: Ownership State Is Narrow, Not Generic

**What:** 只记录 UniClipboard 预期端点 daemon 的 ownership/replacement 信息，例如：

- GUI 本进程是否 spawn 过 child handle
- 本地 daemon metadata 路径
- 最近一次 replacement attempt id / reason / deadline

**When to use:** bounded replace、app exit、错误页自动恢复。

**Example:**

```rust
// Source: planning recommendation
struct DaemonBootstrapState {
    connection: Option<DaemonConnectionInfo>,
    spawned_child: Option<tokio::process::Child>,
    startup_failure: Option<StartupFailure>,
}
```

### Anti-Patterns to Avoid

- **把 daemon 失败仅记录日志:** 当前 `main.rs` 就是这样，直接违背本阶段 UX 锁定决策。
- **只比对 crate `CARGO_PKG_VERSION`:** 当前 daemon/GUI 版本源不一致，会把所有运行中 daemon 判成旧版本。
- **为替换引入通用进程管理器:** 本阶段只处理 UniClipboard 预期本地端点，不做通用 supervisor。
- **把错误页建立在一次性 toast 上:** 错误必须持久，可执行，可自动恢复。

## Don't Hand-Roll

| Problem                   | Don't Build                         | Use Instead                                                                      | Why                                                        |
| ------------------------- | ----------------------------------- | -------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| GUI 退出时 child 清理语义 | 自己假设“drop child 就会自动杀进程” | 明确持有 child handle，并显式 `kill` + `wait` / timeout                          | Rust 官方文档明确说明 child 没有 `Drop`，不会自动等待/清理 |
| sidecar 打包能力          | 临时脚本复制二进制、临时权限洞      | 保持现有 Rust bootstrap；若未来切 sidecar，再完整引入 `externalBin` + capability | 46.3 现在没有 sidecar 配置，临时拼装会扩大范围             |
| 兼容性判定                | 手写一长串“请求 A 成功就算兼容”     | 显式 `packageVersion` + `apiRevision` + 少量关键健康校验                         | 路由探测本身易脆化，版本/契约字段更可验证                  |

**Key insight:** 本阶段真正复杂的不是“spawn 一个进程”，而是“如何在旧 daemon 已经占住固定端点时，安全且可证伪地判定、替换、失败收敛”。不要把它伪装成普通 health check。

## Common Pitfalls

### Pitfall 1: GUI 继续按 backend-ready 展示主窗口

**What goes wrong:** daemon bootstrap 失败后仍进入主界面，后续业务路径才在任意位置炸开。
**Why it happens:** 当前 `StartupBarrier` 只看 backend init；`bootstrap_daemon_connection()` 失败只记日志。
**How to avoid:** 把 daemon compatible-ready 纳入 window show gating；不满足时进入 startup error state。
**Warning signs:** 日志里出现 daemon bootstrap error，但主窗口仍显示 dashboard。

### Pitfall 2: “同包版本”没有共享来源

**What goes wrong:** health/status 永远回 `0.1.0`，GUI 包版本却是 `0.4.0-alpha.1`。
**Why it happens:** daemon 版本源来自 crate 自己的 `CARGO_PKG_VERSION`。
**How to avoid:** 引入共享发布版本来源，health/status 只暴露共享 package version。
**Warning signs:** 同一次构建产物内，GUI 与 daemon 的 version 字段不一致。

### Pitfall 3: 只会 spawn，不会 replace

**What goes wrong:** 端点上已有旧 daemon 时，GUI 没有 ownership 信息，也没有受控 terminate 路径。
**Why it happens:** 当前只返回 `Child` 后立即丢弃，未持久化任何 ownership 元数据。
**How to avoid:** 增加 bootstrap ownership state，并为非当前进程 child 准备窄范围 replacement 机制。
**Warning signs:** “port already in use” 或 GUI 反复 probe 同一旧 daemon。

### Pitfall 4: 启动错误态与业务 lifecycle 混在一起

**What goes wrong:** 复用 `WatcherFailed` / `NetworkFailed` banner，导致 daemon bootstrap failure 语义失真。
**Why it happens:** 现有 lifecycle 状态只覆盖 watcher/network，不覆盖 daemon bootstrap。
**How to avoid:** 单独建 startup bootstrap state，或明确扩展 lifecycle 枚举，不要偷借 dashboard banner。
**Warning signs:** 错误态只能在 dashboard 看见，而不是启动阶段就阻断。

### Pitfall 5: 当前测试基线并不干净

**What goes wrong:** planner 假设当前 `uc-tauri` bootstrap tests 是绿的，结果 implementation 阶段才发现本地分支已经有未完成改动。
**Why it happens:** `run.rs` tests 已引用 `IncompatibleDaemon`，但 enum 里还没有该 variant。
**How to avoid:** 把 bootstrap test baseline 修正列为 Wave 0 前置。
**Warning signs:** `cargo test -p uc-tauri startup_helper_treats_timeout_as_error` 编译失败。

### Child Processes Require Explicit Cleanup

```rust
// Source: https://doc.rust-lang.org/std/process/struct.Child.html
// Plan implication: a spawned daemon must be tracked explicitly if 46.3 wants
// deterministic replacement or shutdown behavior.
let mut child = std::process::Command::new("uniclipboard-daemon").spawn()?;
child.kill()?;
let _status = child.wait()?;
```

### Tokio Child Policy Must Be Explicit

```rust
// Source: https://docs.rs/tokio/latest/tokio/process/struct.Command.html
let mut child = tokio::process::Command::new("uniclipboard-daemon")
    .kill_on_drop(false)
    .spawn()?;
```

// Source: src-tauri/crates/uc-tauri/src/bootstrap/run.rs

## Open Questions

1. **替换旧 daemon 的 durable ownership 机制用什么最小闭环？**
   - What we know: 当前只有 child 临时句柄，且被立即丢弃；没有 pid/meta 文件。
   - What's unclear: 是否已有打包内可复用的进程标识/退出控制基础。
   - Recommendation: planner 先把“ownership metadata + bounded terminate strategy”列成首个设计任务，再决定是 pid file 还是局部 admin stop contract。

2. **打包后的 daemon sibling 是否真实存在？**
   - What we know: `run.rs` 依赖 sibling binary；`tauri.conf.json` 没有 `externalBin`；capability 也没 shell 权限。
   - What's unclear: release bundle 是否已有其他复制/嵌入流程。
   - Recommendation: planner 在 Wave 0 加一个打包核查任务；如果当前 bundle 没有 daemon binary，46.3 不能只修 runtime code。

3. **启动错误态是否扩展现有 lifecycle enum，还是单独新建 startup state？**
   - What we know: 当前 lifecycle 只覆盖 watcher/network。
   - What's unclear: 直接扩 enum 会不会污染 dashboard 语义。
   - Recommendation: 优先独立 startup state，除非实现阶段确认现有 lifecycle surface 可以无副作用扩展。
