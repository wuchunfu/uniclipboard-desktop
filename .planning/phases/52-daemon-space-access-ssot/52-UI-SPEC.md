> Visual and interaction contract for Phase 52: Daemon as Single Source of Truth for Space Access State.
> Generated by gsd-ui-researcher.

## Phase UI Scope

**This is a backend-dominant refactor phase.** The v0.4.0 milestone explicitly declares "GUI feature changes ŌĆö Out of Scope" (REQUIREMENTS.md). No new pages, flows, or visual components are introduced.

Frontend changes are limited to three narrow areas:

1. Replace `setupRealtimeStore.ts` space access consumption ŌĆö swap local `onSpaceAccessCompleted` listener for new `onSpaceAccessStateChanged` daemon WebSocket event subscription.
2. Add `onSpaceAccessStateChanged` listener function in `src/api/setup.ts` (or `src/api/spaceAccess.ts`) that maps daemon `space_access.state_changed` events to typed `SpaceAccessState` callbacks.
3. Remove any frontend references that depended on the GUI-local `SpaceAccessOrchestrator` (none visible in current frontend code ŌĆö confirmed from code scan).

No UI regressions are expected. All existing setup flow screens (Welcome, CreateSpaceInputPassphrase, JoinSpaceSelectDevice, JoinSpaceConfirmPeer, JoinSpaceInputPassphrase, ProcessingCreateSpace, ProcessingJoinSpace, Completed) are unchanged in their visual presentation.

## Interaction States

These apply to any minimal frontend code written in this phase (event subscription wiring, loading indicators during initial HTTP fetch):

| State                        | Visual treatment                                                                                                                                                                                                                    |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Loading (initial HTTP fetch) | No new spinner UI. Re-use existing `ProcessingJoinSpace` / `ProcessingCreateSpace` skeleton loading that was already present. If state is null/unhydrated, render nothing ŌĆö existing guard in `setupRealtimeStore.ts` handles this. |
| Connected (daemon WS active) | No user-visible indicator change. Existing connection status is managed by `DaemonWsBridge` outside setup flow.                                                                                                                     |
| Disconnected / retry         | Re-use existing `scheduleRetry()` pattern already in `setupRealtimeStore.ts`. No new UI treatment.                                                                                                                                  |
| Error (subscription failure) | Console error only (`console.error`). Same treatment as existing `onSpaceAccessCompleted` failure handling at line 143 of `setupRealtimeStore.ts`.                                                                                  |
