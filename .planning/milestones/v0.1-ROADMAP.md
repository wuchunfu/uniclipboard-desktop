# Milestone v0.1: Daily Driver

**Status:** ✅ SHIPPED 2026-03-03
**Phases:** 1-3
**Total Plans:** 6

## Overview

Evolve from basic clipboard sync into a daily-driver productivity tool with quick-paste, image support, and robust data transfer. This milestone focused on the infrastructure layer: unified chunked transfer protocol, multi-representation payloads, and streaming decode.

## Phases

### Phase 1: Add Download Progress Display

**Goal**: Real-time download progress bar in update dialogs via Tauri IPC Channel
**Depends on**: None
**Plans**: 1 plan

Plans:

- [x] 01-01: Bridge download callbacks to React progress bar

**Details:**
Bridged Tauri updater's empty download callbacks to the React frontend using `tauri::ipc::Channel`. Progress bar shows download percentage when content-length is known, or an indeterminate pulse animation otherwise. Buttons disabled during download/install.

### Phase 2: 实现统一数据传输层

**Goal:** Replace V1 text-only clipboard sync with a unified chunked transfer layer: all clipboard representations (text/image) bundled, chunk-level XChaCha20-Poly1305 encrypted (deterministic nonces via blake3), transferred over existing libp2p transport, receiver validates, reassembles, and writes highest-priority representation to clipboard. No UI changes — content appears silently.
**Requirements**: UTL-01, UTL-02, UTL-03, UTL-04, UTL-05, UTL-06, UTL-07
**Depends on:** Phase 1
**Plans:** 3 plans

Plans:

- [x] 02-01: V2 protocol type contracts in uc-core (ClipboardMessage v2 field, ClipboardMultiRepPayloadV2, chunk AAD helper)
- [x] 02-02: ChunkedEncoder/Decoder in uc-infra (XChaCha20-Poly1305 per-chunk AEAD, binary wire format)
- [x] 02-03: Wire V2 into sync_outbound + sync_inbound use cases; raise transport limits

**Details:**
Established backward-compatible ClipboardPayloadVersion enum (V1/V2), created streaming chunked encoder/decoder with per-chunk AEAD encryption, wired V2 outbound (all representations packed) and V2 inbound (priority-based selection: image > html > rtf > plain) through existing use cases. Transport limits raised to 300MB/120s.

### Phase 3: True Inbound Streaming

**Goal:** Eliminate the `read_to_end` bottleneck in `libp2p_network.rs` — separate the outer `ProtocolMessage` JSON envelope from the V2 binary payload so `ChunkedDecoder::decode_from` can operate at the stream level, reducing peak memory from ~2× payload size to ~1× chunk size.
**Requirements:** (no new UTL REQ-IDs — tech debt resolution)
**Depends on:** Phase 2
**Plans:** 2 plans

Plans:

- [x] 03-01: Two-segment wire framing: outbound sender produces [length-prefix JSON header][raw V2 payload]
- [x] 03-02: Inbound streaming: libp2p_network.rs reads length-prefix, then streams V2 remainder to ChunkedDecoder via SyncIoBridge + spawn_blocking

**Details:**
Added `ProtocolMessage::frame_to_bytes()` for two-segment wire format production. Restructured inbound stream handler with two-segment framing reader, ProcessedMessage enum dispatch, and transport-level V2 streaming decode. Changed channel type throughout codebase to carry pre-decoded plaintext. Peak memory reduced from ~2x payload to ~1x chunk (256KB).

## Milestone Summary

**Key Decisions:**

- Separate floating window (not popup in main window) for quick-paste
- Image sync before file sync — images are most common non-text clipboard content
- Chunked transfer as infrastructure layer — supports both image sync and future file sync
- WebDAV deferred to next milestone — focus on core experience first
- serde_with Base64 for JSON encoding (serde_bytes only optimizes binary formats)
- Option B for outbound streaming — encode to Vec<u8>, defer transport streaming
- V2 dedup by message.id only — OS clipboard doesn't hold all reps
- Two-segment wire framing with 4-byte LE length prefix
- EncryptionSessionPort as constructor parameter to Libp2pNetworkAdapter
- 64KB JSON header cap for security

**Issues Resolved:**

- serde_bytes doesn't produce base64 in JSON — switched to serde_with
- read_to_end bottleneck for inbound V2 payloads — replaced with streaming decode
- ~33% base64 overhead on V2 wire format — eliminated with raw binary trailing payload

**Issues Deferred:**

- Quick-paste floating window — deferred to next milestone
- Image clipboard capture and display UI — deferred to next milestone
- True outbound stream-pass-through (Option A) — requires ClipboardTransportPort interface changes

**Technical Debt Incurred:**

- Phase 01 missing formal VERIFICATION.md
- Pre-existing test failure: business_command_timeouts_cover_stream_operation_budgets
- broadcast_clipboard port stub unimplemented
- Human verification pending: real-device memory profiling and E2E with encryption
