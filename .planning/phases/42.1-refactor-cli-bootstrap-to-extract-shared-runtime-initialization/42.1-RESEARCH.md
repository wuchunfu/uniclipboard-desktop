# Phase 42.1: Refactor CLI bootstrap to extract shared runtime initialization - Research

**Researched:** 2026-03-19
**Domain:** Rust CLI bootstrap/runtime composition for direct-mode commands
**Confidence:** HIGH

## Summary

Phase 42.1 is a scoped refactor to remove repeated direct-mode CLI runtime assembly before more CLI phases land. The current `uc-cli` command handlers rebuild the same non-GUI runtime flow in six places: `build_cli_context_with_profile()` -> `get_storage_paths(&ctx.config)` -> `build_non_gui_runtime(ctx.deps, storage_paths)`. That duplication now exists in `devices`, `space-status`, and all three `clipboard` handlers.

The codebase already has the right low-level ownership boundaries. `uc-bootstrap` owns both the CLI dependency builder (`CliBootstrapContext` in `builders.rs`) and the non-GUI `CoreRuntime` constructor (`build_non_gui_runtime()` in `non_gui_runtime.rs`). The missing piece is a convenience helper that composes those existing pieces for direct-mode CLI callers. That helper should live in `uc-bootstrap`, not in each command module, so the composition-root boundary remains intact.

Keep the helper at the `CoreRuntime` level, not the `CoreUseCases` level. `devices` and `clipboard` create `CoreUseCases::new(&runtime)`, but `space-status` calls `runtime.is_encryption_ready()` and `runtime.encryption_state()` directly. Returning `CoreRuntime` preserves both patterns and avoids inventing a second accessor abstraction that would immediately need exceptions.

**Primary recommendation:** Add `uc_bootstrap::build_cli_runtime_with_profile(log_profile)` plus a default `build_cli_runtime()` wrapper, re-export them from `uc-bootstrap`, and migrate direct-mode CLI commands to that helper without touching daemon or GUI bootstrap paths.

## Architecture Patterns

### Recommended Project Structure

```text
src-tauri/crates/uc-bootstrap/src/
├── builders.rs          # Low-level CLI dependency builder
├── non_gui_runtime.rs   # Add build_cli_runtime[_with_profile]()
└── lib.rs               # Re-export new helper

src-tauri/crates/uc-cli/src/
├── commands/            # Command handlers stay business-focused
├── output.rs            # Existing json/human output helper
└── exit_codes.rs        # Existing stable exit codes
```

### Pattern 1: Compose Once in `uc-bootstrap`

**What:** Add a convenience helper that performs the full direct-mode CLI runtime assembly by composing existing bootstrap steps internally.

**When to use:** Every CLI command that talks to application state directly instead of going through daemon RPC.

**Example:**

```rust
// Source: src-tauri/crates/uc-bootstrap/src/non_gui_runtime.rs
use anyhow::Context;

pub fn build_cli_runtime_with_profile(
    log_profile: Option<uc_observability::LogProfile>,
) -> anyhow::Result<uc_app::runtime::CoreRuntime> {
    let ctx = crate::build_cli_context_with_profile(log_profile)
        .context("failed to initialize CLI context")?;
    let storage_paths = crate::get_storage_paths(&ctx.config)
        .context("failed to resolve storage paths")?;
    crate::build_non_gui_runtime(ctx.deps, storage_paths)
        .context("failed to build CLI runtime")
}

pub fn build_cli_runtime() -> anyhow::Result<uc_app::runtime::CoreRuntime> {
    build_cli_runtime_with_profile(Some(uc_observability::LogProfile::Cli))
}
```

### Pattern 2: Keep Command Handlers Business-Focused

**What:** Command modules should do only three things: resolve `verbose` to a log profile, initialize the shared runtime once, and execute command-specific business logic/output mapping.

**When to use:** `devices`, `space-status`, `clipboard`, and future direct-mode commands like settings and sync.

**Example:**

```rust
// Source: src-tauri/crates/uc-cli/src/commands/devices.rs
let profile = if verbose {
    Some(uc_observability::LogProfile::Dev)
} else {
    Some(uc_observability::LogProfile::Cli)
};

let runtime = match uc_bootstrap::build_cli_runtime_with_profile(profile) {
    Ok(runtime) => runtime,
    Err(e) => {
        eprintln!("Error: failed to initialize CLI runtime: {}", e);
        return exit_codes::EXIT_ERROR;
    }
};

let usecases = uc_app::usecases::CoreUseCases::new(&runtime);
```

### Anti-Patterns to Avoid

- **Helper returns `CoreUseCases` instead of `CoreRuntime`:** `space-status` already proves commands are not all use-case-only.
- **Refactor daemon or GUI bootstrap in the same phase:** this phase is CLI-only and should stay atomic.
- **Hide command behavior behind a macro:** runtime init is shared, but result mapping and error text remain command-specific.
- **Move `get_storage_paths()` logic into `uc-cli`:** that reverses the composition-root extraction done in Phase 40.

## Don't Hand-Roll

| Problem                        | Don't Build                                   | Use Instead                                        | Why                                                                                     |
| ------------------------------ | --------------------------------------------- | -------------------------------------------------- | --------------------------------------------------------------------------------------- |
| CLI direct-mode runtime init   | Per-command three-step bootstrap sequence     | `uc_bootstrap::build_cli_runtime[_with_profile]()` | Centralizes composition, preserves non-GUI defaults, reduces future command boilerplate |
| Command service facade         | Custom CLI-only wrapper around every use case | `CoreUseCases::new(&runtime)`                      | Existing accessor is already lightweight and shared                                     |
| Cross-mode runtime abstraction | One helper for GUI, daemon, and CLI           | Scene-specific helpers in `uc-bootstrap`           | GUI and daemon have different lifecycle and channel needs                               |

**Key insight:** The hard part is not constructing `CoreRuntime`; it is preserving the existing scene boundary decisions from Phases 40-41 while removing repeated CLI assembly code.

## Common Pitfalls

### Pitfall 1: Returning the Wrong Abstraction

**What goes wrong:** The new helper returns `CoreUseCases` or a command-specific wrapper, and `space-status` no longer fits the shared path.

**Why it happens:** Most current direct-mode commands use `CoreUseCases`, so it is easy to assume they all do.

**How to avoid:** Return `CoreRuntime` from the new helper and let each command construct `CoreUseCases::new(&runtime)` only if it needs them.

**Warning signs:** The refactor introduces a special case just for `space-status` or starts adding runtime passthrough methods to a helper struct.

### Pitfall 2: Losing Stage-Specific Error Context

**What goes wrong:** All bootstrap failures collapse into a generic “runtime init failed” message.

**Why it happens:** The current command code handles three separate failure points explicitly; a naive helper can erase that detail.

**How to avoid:** Use `anyhow::Context` inside the helper so callers still see whether config resolution, storage path resolution, or runtime construction failed.

**Warning signs:** CLI stderr no longer distinguishes “failed to initialize CLI context” from “failed to resolve storage paths”.

### Pitfall 3: Expanding Scope into Daemon or GUI Paths

**What goes wrong:** The refactor starts touching `build_daemon_app()`, `src-tauri/src/main.rs`, or `uc-tauri`.

**Why it happens:** The words “bootstrap” and “runtime” appear in all three entry points.

**How to avoid:** Limit edits to `uc-bootstrap` non-GUI helpers, re-exports, and direct-mode CLI command handlers.

**Warning signs:** New diffs show up in GUI startup or daemon worker wiring files.

### Pitfall 4: Stopping at Compile Green

**What goes wrong:** The refactor compiles, but a command silently changes behavior or loses JSON/help output coverage.

**Why it happens:** Runtime-init extraction is mechanical, so it looks “safe” enough to skip validation.

**How to avoid:** Keep `uc-cli` smoke tests green and add at least one focused bootstrap test at the `uc-bootstrap` layer.

**Warning signs:** Only `cargo check` is run after the refactor.

## Open Questions

1. **Should the phase add both `build_cli_runtime()` and `build_cli_runtime_with_profile()`?**
   - What we know: `build_cli_context()` already offers both default and explicit-profile variants.
   - What's unclear: only the explicit-profile variant is needed immediately.
   - Recommendation: add both for API parity; commands use `_with_profile()` now, future callers can use the default safely.

2. **Should bootstrap coverage be unit tests or a new integration test file?**
   - What we know: `uc-bootstrap` currently has unit tests in source files and no `tests/` directory.
   - What's unclear: whether helper behavior needs filesystem-heavy setup to test meaningfully.
   - Recommendation: start with source-level tests in `non_gui_runtime.rs`; add a `tests/` integration target only if the helper grows beyond pure composition.
