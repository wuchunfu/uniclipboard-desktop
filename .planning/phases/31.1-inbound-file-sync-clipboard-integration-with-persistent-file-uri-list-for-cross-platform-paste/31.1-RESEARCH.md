# Phase 31.1: Inbound File Sync Clipboard Integration - Research

**Researched:** 2026-03-14
**Domain:** Clipboard file write integration, persistent file URI storage, cross-platform paste
**Confidence:** HIGH

## Summary

This phase integrates received files with the system clipboard so users can paste (Cmd+V / Ctrl+V) into Finder/Explorer/file managers after file transfer completes. The work spans three layers: (1) backend clipboard write triggered by file transfer completion events in `wiring.rs`, (2) a new `copy_file_to_clipboard` Tauri command for manual Dashboard copies with file existence validation, and (3) frontend Dashboard enhancements for file entry display with stale-file handling, icons by extension, and multi-file batch rendering.

The existing codebase has strong foundations: `set_files()` in `uc-platform/src/clipboard/common.rs:463-469` already handles cross-platform file clipboard write via `text/uri-list`; `ClipboardChangeOrigin` enum supports origin tracking for clipboard race detection; the `fileTransferSlice` Redux store already has `clipboardWriteCancelled` flag; and `FileContextMenu` already handles copy/delete/open-location actions. The primary work is wiring these pieces together and adding persistence + validation layers.

**Primary recommendation:** Wire file transfer completion event in `wiring.rs` to clipboard write via `SystemClipboardPort::write_snapshot()`, create a `CopyFileToClipboardUseCase` for manual Dashboard copies, and extend the frontend `ClipboardItemRow` to render file entries with extension-based icons and stale-file visual treatment.

### Pattern 1: Auto-Write Clipboard on File Transfer Complete

**What:** When `NetworkEvent::FileTransferCompleted` fires in `wiring.rs`, construct a `SystemClipboardSnapshot` with `text/uri-list` representation containing `file://` URIs, then call `SystemClipboardPort::write_snapshot()`.
**When to use:** Every inbound file transfer completion.
**Key integration point:** `wiring.rs:2128-2209` already handles `FileTransferCompleted` -- add clipboard write logic after `handle_transfer_complete()` succeeds.

```rust
// In wiring.rs FileTransferCompleted handler, after successful handle_transfer_complete:
// 1. Check clipboardWriteCancelled (needs state check)
// 2. Set ClipboardChangeOrigin::RemotePush to prevent re-capture loop
// 3. Build snapshot with text/uri-list representation
// 4. Call local_clipboard.write_snapshot(snapshot)

let uri_list = format!("file://{}", result.file_path.to_string_lossy());
let snapshot = SystemClipboardSnapshot {
    ts_ms: chrono::Utc::now().timestamp_millis(),
    representations: vec![ObservedClipboardRepresentation::new(
        RepresentationId::generate(),
        FormatId::from("files"),
        Some(MimeType::uri_list()),
        uri_list.into_bytes(),
    )],
};
// Set origin to avoid re-capture loop
clipboard_change_origin
    .set_next_origin(ClipboardChangeOrigin::RemotePush, Duration::from_secs(2))
    .await;
local_clipboard.write_snapshot(snapshot)?;
```

### Pattern 3: Clipboard Entry Persistence for Files

**What:** Reuse existing `ClipboardEntry` + `Representation` model. File entries are stored as a ClipboardEntry with a single `text/uri-list` representation containing `file://` URIs.
**When to use:** After file transfer completes, persist the entry in clipboard history.

The entry must be created via the same persistence path as normal clipboard captures. The `content_type` field on the entry projection should be `"file"` for Dashboard to render with file-specific UI.

### Pattern 4: Stale File Detection (Lazy)

**What:** Only check file existence when user attempts Copy from Dashboard. No startup scan.
**When to use:** On `copy_file_to_clipboard` command invocation.

```rust

### Anti-Patterns to Avoid

- **Startup file existence scan:** Context explicitly says lazy-only validation. Never scan all file entries on startup.
- **Adding text/plain representation for files:** Decision says only `set_files()` -- no additional representations.
- **Custom clipboard race detection:** Use existing `clipboardWriteCancelled` flag in `fileTransferSlice` and `ClipboardChangeOrigin` system.
- **Separate file history storage:** Must reuse existing `clipboard_entry` + `representation` model, not a separate table.

## Common Pitfalls

### Pitfall 1: Clipboard Re-capture Loop

**What goes wrong:** Writing files to clipboard triggers `ClipboardWatcher`, which captures the change as a new clipboard event, creating duplicate entries or infinite loops.
**Why it happens:** The clipboard watcher monitors all clipboard changes including programmatic writes.
**How to avoid:** Set `ClipboardChangeOrigin::RemotePush` via `clipboard_change_origin.set_next_origin()` BEFORE calling `write_snapshot()`. The capture pipeline checks origin and skips re-capture for `RemotePush` and `LocalRestore` origins. See `RestoreClipboardSelectionUseCase::restore_snapshot()` for the exact pattern.
**Warning signs:** Duplicate file entries appearing in clipboard history after transfer.

### Pitfall 2: File URI Format Differences Across Platforms

**What goes wrong:** `file:///path/to/file` works on macOS/Linux but Windows needs `file:///C:/path/to/file`.
**Why it happens:** Different OS path conventions.
**How to avoid:** Use `url::Url::from_file_path(&path)` to construct URIs (handles platform differences), and `url::Url::to_file_path()` to parse them back.
**Warning signs:** Paste works on macOS but fails on Windows.

### Pitfall 3: Multi-File Batch Timing

**What goes wrong:** Each file in a batch triggers a separate clipboard write, overwriting the previous one.
**Why it happens:** Files in a batch complete at different times; each `FileTransferCompleted` event fires independently.
**How to avoid:** Need batch detection logic -- group files by transfer batch (Phase 29's batch concept), wait for all files to complete before writing. The CONTEXT says "clipboard write waits for all files in batch to complete."
**Warning signs:** Only the last file in a batch ends up on clipboard.

### Pitfall 4: Stale File Cascade on Entry Delete

**What goes wrong:** Deleting a clipboard entry should also delete the cache file, but the existing `delete_clipboard_entry` use case doesn't know about file cache.
**Why it happens:** Current `DeleteClipboardEntry` use case only handles DB cleanup, not file system.
**How to avoid:** Extend the delete flow to check if entry has `text/uri-list` representation, parse file paths, and delete cache files before removing the DB entry. Or add a post-delete hook.
**Warning signs:** Orphaned cache files consuming disk space.

### Pitfall 5: wiring.rs Clipboard Port Access

**What goes wrong:** The `handle_network_event` function in `wiring.rs` doesn't currently have access to `SystemClipboardPort` or `ClipboardChangeOriginPort`.
**Why it happens:** These ports weren't needed for file transfer handling before.
**How to avoid:** Thread the clipboard port and origin port through to the event handler, similar to how `inbound_file_settings` is already passed. Clone the Arc references before the event loop.
**Warning signs:** Compilation errors about missing clipboard ports in `handle_network_event`.

pub async fn restore_snapshot(&self, snapshot: SystemClipboardSnapshot) -> Result<()> {
    if !self.mode.allow_os_write() {
        return Err(anyhow::anyhow!("System clipboard writes disabled"));
    }
    self.clipboard_change_origin
        .set_next_origin(ClipboardChangeOrigin::LocalRestore, Duration::from_secs(2))
        .await;
    if let Err(err) = self.local_clipboard.write_snapshot(snapshot) {
        self.clipboard_change_origin
            .consume_origin_or_default(ClipboardChangeOrigin::LocalCapture)
            .await;
        return Err(err);
    }
    Ok(())
}
```

## Open Questions

1. **Multi-file batch detection**
   - What we know: Phase 29 has batch concept; files from same sender in same batch should be grouped
   - What's unclear: Exact batch ID/grouping mechanism in NetworkEvent::FileTransferCompleted -- need to check if transfer_id encodes batch info
   - Recommendation: Check Phase 29's transfer model for batch grouping; may need a batch accumulator in wiring.rs that collects completed files and writes to clipboard when batch is done

2. **set_files() input format**
   - What we know: `common.rs` passes `Vec<String>` of lines from URI list to `ctx.set_files()`
   - What's unclear: Whether `set_files()` expects `file:///path` URIs or plain `/path` strings
   - Recommendation: Test with the actual clipboard-rs API; the common.rs code passes raw lines which are `file:///...` URIs, suggesting clipboard-rs handles URI format

3. **Entry creation path for received files**
   - What we know: Normal clipboard captures go through CaptureClipboardUseCase; file transfers complete in wiring.rs
   - What's unclear: Whether to create the ClipboardEntry via CaptureClipboardUseCase (by feeding a synthetic snapshot) or via a new dedicated persistence path
   - Recommendation: Creating a synthetic snapshot and feeding it through the normal capture pipeline is cleaner -- it reuses existing persistence, selection policy, and content type classification. The origin should be set to RemotePush to avoid re-syncing.
