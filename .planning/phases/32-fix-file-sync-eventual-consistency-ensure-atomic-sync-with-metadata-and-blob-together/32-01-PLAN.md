  <name>Task 1: Add core/app transfer tracking contract and metadata-seeding use case</name>
  <files>

    src-tauri/crates/uc-app/src/usecases/clipboard/sync_inbound.rs
  </files>
  <action>
    1. Add a new hexagonal port in `uc-core` for receiver-side transfer tracking. The contract must support:
       - seeding pending records from clipboard metadata with `entry_id`, `origin_device_id`, `transfer_id`, `filename`, and intended local cache path
       - backfilling announce metadata when later available
       - marking `transferring`, `completed`, and `failed`
       - refreshing in-flight liveness on later receiving progress events without changing semantic status
       - listing or sweeping expired in-flight rows based on `updated_at_ms` and the locked timeout budgets
       - bulk-marking stale in-flight rows failed during startup reconciliation
       - computing aggregate transfer status for an entry

    2. Define explicit domain types in the same module, not as raw strings:
       - `TrackedFileTransferStatus`
       - `TrackedFileTransfer`
       - `PendingInboundTransfer`
       - `EntryTransferSummary`

    3. Add `TrackInboundTransfersUseCase` in `uc-app` that owns state transitions. It should stay free of Diesel, Tauri, and filesystem implementation details. Required methods:
       - `record_pending_from_clipboard(...)`
       - `mark_transferring(...)`
       - `refresh_activity(...)`
       - `mark_completed(...)`
       - `mark_failed(...)`
       - `list_expired_inflight(...)`
       - `reconcile_inflight_after_startup(...)`

    4. Extend `InboundApplyOutcome::Applied` so file-backed clipboard messages return both the persisted `entry_id` and a `pending_transfers` payload. The payload should be sufficient for the Tauri layer to emit pending status without guessing transfer-to-entry linkage later.

    5. In `SyncInboundClipboardUseCase`, after `CaptureClipboardUseCase::execute_with_origin(...)` succeeds for a message with `file_transfers`, call the new app use case to seed pending transfer records. Use:
       - `entry_id`
       - `message.origin_device_id`
       - every `transfer_id`
       - the local cache target path already derived from `{cache_dir}/{transfer_id}/{filename}`

    6. In `deps.rs`, add `file_transfer_repo: Arc<dyn FileTransferRepositoryPort>` to the `StoragePorts` bundle. The file transfer repository is a storage-layer port (it persists transfer state to SQLite), so it belongs alongside other storage ports rather than in a separate bundle.

    7. Model timeout and cleanup handoff explicitly in the app contract:
       - use `updated_at_ms` as the durable liveness timestamp
       - have the reconciliation/expiration path return cleanup targets (for example cached file path or transfer directory) that platform code can delete later

    8. Keep this plan strictly at core/app level. Do not add Diesel schema, migrations, Tauri payloads, or frontend changes here.
