<objective>
Migrate inbound clipboard sync (remote peer content -> OS clipboard write) from GUI to daemon ownership.

Purpose: Per D-09, the daemon must be the single process interacting with the OS clipboard for both reads (watcher, from 57-01) AND writes (inbound sync). The `SyncInboundClipboardUseCase` already supports both Full mode (writes to OS clipboard) and Passive mode (persists only). The daemon runs it in Full mode; the GUI's instance runs in Passive mode (set by 57-02's Passive mode switch). The write-back loop prevention is handled by `ClipboardChangeOriginPort::remember_remote_snapshot_hash()` + `set_next_origin()` which the use case already calls in Full mode.

Output: Daemon service that receives inbound clipboard messages from network peers and writes to OS clipboard. GUI inbound path becomes Passive (persist-only).
</objective>

    ```rust
    use std::sync::Arc;
    use async_trait::async_trait;
    use tokio::sync::mpsc;
    use tokio_util::sync::CancellationToken;
    use tracing::{info, warn, info_span, Instrument};
    use uc_app::usecases::clipboard::sync_inbound::SyncInboundClipboardUseCase;
    use uc_core::network::ClipboardMessage;
    use crate::service::{DaemonService, ServiceHealth};

    pub struct InboundClipboardSyncWorker {
        usecase: Arc<SyncInboundClipboardUseCase>,
        clipboard_rx: mpsc::Receiver<(ClipboardMessage, Option<Vec<u8>>)>,
    }

    impl InboundClipboardSyncWorker {
        pub fn new(
            usecase: Arc<SyncInboundClipboardUseCase>,
            clipboard_rx: mpsc::Receiver<(ClipboardMessage, Option<Vec<u8>>)>,
        ) -> Self {
            Self { usecase, clipboard_rx }
        }
    }

    #[async_trait]
    impl DaemonService for InboundClipboardSyncWorker {
        fn name(&self) -> &str { "inbound-clipboard-sync" }

        async fn start(&self, cancel: CancellationToken) -> anyhow::Result<()> {
            // NOTE: DaemonService::start takes &self, but we need &mut self for clipboard_rx.
            // This may require using Mutex<Option<Receiver>> pattern or restructuring.
            // Check existing daemon service patterns for how they handle owned receivers.
            loop {
                tokio::select! {
                    _ = cancel.cancelled() => {
                        info!("inbound clipboard sync worker stopping");
                        break;
                    }
                    msg = self.clipboard_rx.recv() => {
                        match msg {
                            Some((message, pre_decoded)) => {
                                let msg_id = message.id.clone();
                                if let Err(e) = self.usecase.execute(message, pre_decoded).await {
                                    warn!(error = %e, message_id = %msg_id, "inbound clipboard sync failed");
                                }
                            }
                            None => {
                                info!("inbound clipboard channel closed");
                                break;
                            }
                        }
                    }
                }
            }
            Ok(())
        }

        async fn stop(&self) -> anyhow::Result<()> { Ok(()) }
        fn health_check(&self) -> ServiceHealth { ServiceHealth::Healthy }
    }
    ```

    **IMPORTANT**: The `DaemonService::start(&self, ...)` signature takes `&self`, not `&mut self`. The `mpsc::Receiver` requires `&mut self` to call `.recv()`. You have two options:
    - Use `Mutex<Option<mpsc::Receiver<...>>>` and `.lock().take()` inside `start()` (one-shot usage)
    - Restructure to pass the receiver into `start()` differently

    Check how other daemon services that consume channels handle this (look at existing workers). The clipboard_watcher.rs Task 2 creates a channel *inside* `start()`, which avoids this issue. For inbound sync, the receiver comes from outside (the network layer), so it must be passed in.
