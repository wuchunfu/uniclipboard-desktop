<objective>
Rewrite the outbound clipboard sync path to use V3 binary encoding, Arc<[u8]> zero-copy fanout, and tokio::join! parallelization. Update ClipboardTransportPort signatures from Vec<u8> to Arc<[u8]>. Delete all V1/V2 payload types and outbound encoding paths.

Purpose: This wires the V3 codec (from Plan 01) into the actual sync pipeline. The Arc<[u8]> change eliminates per-peer memory copies for large images. The tokio::join! parallelization overlaps encryption with network path setup for the first peer.

Output: Updated port trait, rewritten outbound use case, cleaned up protocol types, updated adapter implementations.
</objective>

Part B — Rewrite sync_outbound.rs outbound path:

1. Replace V2 payload construction: Instead of building `ClipboardMultiRepPayloadV2` and serializing with `serde_json::to_vec`, build `ClipboardBinaryPayload` with `BinaryRepresentation` entries from the snapshot, then call `encode_to_vec()`.
2. Replace outbound send logic with parallel first-peer pattern:

   ```rust
   let plaintext_bytes = v3_payload.encode_to_vec()?;
   let first_peer = sendable_peers[0].clone();
   let remaining_peers = &sendable_peers[1..];

   // Parallel: encrypt + first peer's ensure_business_path
   let (encrypted_result, ensure_result) = tokio::join!(
       async {
           let master_key = encryption_session.get_master_key()?;
           let encrypted = transfer_encryptor.encrypt(&master_key, &plaintext_bytes)?;
           let framed = ProtocolMessage::Clipboard(header).frame_to_bytes(Some(&encrypted))?;
           Ok::<Arc<[u8]>, anyhow::Error>(Arc::from(framed.into_boxed_slice()))
       },
       self.clipboard_network.ensure_business_path(&first_peer.peer_id)
   );
   let outbound_bytes = encrypted_result?;
   ensure_result?;

   // Send to first peer
   self.clipboard_network.send_clipboard(&first_peer.peer_id, outbound_bytes.clone()).await?;

   // Serial for remaining peers
   for peer in remaining_peers {
       self.clipboard_network.ensure_business_path(&peer.peer_id).await?;
       self.clipboard_network.send_clipboard(&peer.peer_id, outbound_bytes.clone()).await?;
   }
   ```

   Note: The exact structure depends on how `&self` borrows work with tokio::join!. If `&self` causes "future is not Send" issues, extract encryption into a standalone async block that captures only the needed owned/cloned values (encryptor, master_key, plaintext_bytes) — not `&self`. See RESEARCH.md Pitfall 7.

3. Update `ProtocolMessage` header construction to use `ClipboardPayloadVersion::V3`
