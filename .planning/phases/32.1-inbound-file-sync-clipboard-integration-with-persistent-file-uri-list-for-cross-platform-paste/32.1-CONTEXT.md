# Phase 32.1: Inbound file sync clipboard integration with persistent file URI list for cross-platform paste - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Integrate received files with the system clipboard — after file transfer completes, automatically write file references to clipboard so users can paste (Cmd+V / Ctrl+V) in Finder/Explorer/file managers. Includes persistent file URI storage in clipboard history, file existence validation, cache lifecycle management, and Dashboard file entry display with stale-file handling.

</domain>

<decisions>
## Implementation Decisions

### Clipboard Write Trigger

- Small files: auto-write to clipboard after transfer completes, with system notification ("收到 N 个文件，已写入剪贴板")
- Large files: same behavior as small files — download completes → notify + auto-write to clipboard
- Multi-file batch: merged notification ("收到 N 个文件 from [device]，已写入剪贴板"), clipboard write waits for all files in batch to complete (Phase 30 decision)
- Clipboard race: if user copies other content during transfer, cancel auto-write; send notification "文件已收到，可在 Dashboard 复制"

### File URI Persistence

- Reuse existing `clipboard_entry` + `representation` model — file entries stored as ClipboardEntry with `text/uri-list` representation
- URI content: absolute `file://` URIs (e.g., `file:///path/to/cached/file.pdf`), multiple files newline-separated
- Stale file handling: entry preserved in history but marked invalid when cache file deleted; right-click Copy disabled, Delete still available
- Restore validation: when user clicks Copy in Dashboard, check file existence first; show "文件已删除" if file missing

### Cross-Platform Paste Experience

- Paste = copy file to target location (OS default behavior — clipboard holds file reference, OS handles copy on paste)
- Text editor paste: shows file path (OS default, no extra text/plain representation needed)
- Original filenames preserved in cache directory (Phase 28 filename validation module handles safety)
- Only write file references via `set_files()` — no additional `text/plain` representation

### File Cache Lifecycle

- Delete Dashboard entry = delete cache file (cascade)
- Quota full: LRU cleanup — oldest received files deleted first, history entries marked invalid
- Startup consistency: lazy check only — validate file existence when user attempts Copy, not on app startup

### Dashboard File Entry Display

- Stale files: grey text + strikethrough style, Copy disabled in context menu, Delete still available
- File icons: different icons by file extension (e.g., PDF, image, archive, document, generic)
- Multi-file batch: single entry showing file list, expandable to see individual files
- Summary (collapsed): filename + size + source device (e.g., "photo.jpg (2.3 MB) from MacBook"); multi-file: "3 个文件 (5.1 MB) from MacBook"

### Clipboard Write Technical Path

- Auto-write (inbound sync): via event bus — SyncInboundFileUseCase emits event on completion, wiring.rs event handler calls clipboard write
- Manual copy (Dashboard): new Tauri command `copy_file_to_clipboard` — frontend calls command, backend UseCase validates file existence then writes to clipboard
- Only `set_files()` called — no additional text/plain write

### Multi-Device File Reception

- Last-completed transfer overwrites clipboard (consistent with text/image sync behavior)
- All received files visible in Dashboard regardless of which device's files are in clipboard
- Notifications per-device: "收到 2 个文件 from MacBook", "收到 1 个文件 from iPad"

### Claude's Discretion

- File extension to icon mapping implementation details
- Exact event types and naming for file transfer completion events
- Cache directory structure and cleanup implementation
- Multi-file batch detection and grouping logic
- Stale file detection implementation approach

</decisions>
