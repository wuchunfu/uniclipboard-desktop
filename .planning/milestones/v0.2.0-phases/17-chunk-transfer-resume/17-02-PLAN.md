<objective>
Implement transfer resume protocol using V3 transfer_id for interrupted clipboard transfers.

Purpose: When a large clipboard transfer is interrupted (network glitch, stream timeout), the receiver can request the sender to resume from the last successfully received chunk instead of retransmitting the entire payload. This saves bandwidth and time for large transfers (images, rich content >1MB).

Output: ProtocolMessage::TransferResume variant, in-memory TTL resume cache on sender side, resume request/response handling in the libp2p network adapter.
</objective>

1. **Resume state cache** -- Add an in-memory TTL cache for sender-side encrypted payloads:

Near the top of the file, define:

```rust
const RESUME_CACHE_TTL: Duration = Duration::from_secs(300); // 5 minutes
const RESUME_MIN_PAYLOAD_SIZE: usize = 1024 * 1024; // 1 MB -- only cache payloads worth resuming
```

Define a cache entry struct:

```rust
struct CachedTransferPayload {
    /// The full framed wire bytes (JSON header + V3 encrypted payload)
    data: Arc<[u8]>,
    /// When this entry was cached
    cached_at: std::time::Instant,
    /// Offset into `data` where the V3 encrypted payload begins (after JSON header frame)
    v3_payload_offset: usize,
}
```

Add a `resume_cache: Arc<RwLock<HashMap<String, CachedTransferPayload>>>` field to the state that the business command loop has access to (either in `PeerCaches` or as a separate Arc passed to the command handler). Use the `transfer_id` (hex-encoded) as key.

**Cache eviction**: On each `SendClipboard` completion, evict entries older than `RESUME_CACHE_TTL`. This is a simple sweep -- no background timer needed since clipboard sends are frequent enough.

2. **Populate cache on successful send** -- After a successful outbound `SendClipboard`, if `data.len() >= RESUME_MIN_PAYLOAD_SIZE`:
   - Parse the V3 header from the payload to extract `transfer_id` (bytes [json_header_end+9..json_header_end+25])
   - Hex-encode the transfer_id
   - Insert into `resume_cache` with current Instant
   - Also store `v3_payload_offset` (= 4 + json_len, the offset where V3 bytes begin)

3. **Handle inbound TransferResume requests** -- In the inbound stream handler's `match message` block, add a new arm:

   ```rust
   ProtocolMessage::TransferResume(resume_msg) => {
       // Look up the cached payload by transfer_id
       // If found and not expired:
       //   - Open a new outbound business stream to the requesting peer
       //   - Write only the V3 header + chunks starting from resume_msg.start_chunk
       //   - This requires parsing the cached V3 payload to skip chunks
       // If not found or expired:
       //   - Log warning, return ProcessedMessage indicating resume-not-available
       //   - The receiver will fall back to requesting a full re-send
   }
   ```

   **Resume re-send logic**: Given the cached `Arc<[u8]>` and `v3_payload_offset`:
   - The V3 payload starts at `data[v3_payload_offset..]`
   - Read V3 header (37 bytes) to get `total_chunks`
   - Skip chunks 0..start_chunk by reading their 4-byte length prefix and skipping ciphertext bytes
   - Write a new JSON header frame with `ProtocolMessage::Clipboard(...)` (reuse the original ClipboardMessage)
   - Write V3 header with modified metadata (same transfer_id, but indicating this is a partial -- OR just re-send the full V3 header and let the receiver handle offset)

   **Simpler approach (recommended)**: Re-send the full V3 header unchanged, then send only chunks from `start_chunk` onward. The receiver knows `start_chunk` from its own request, so it can reconstruct correctly. This avoids modifying the V3 header format.

4. **Receiver-side resume request** -- When the inbound V3 chunked read (from Plan 01) encounters an I/O error mid-transfer:
   - Record `transfer_id` (from the V3 header already read) and `last_successful_chunk` index
   - After the stream closes/errors, if `last_successful_chunk < total_chunks - 1`:
     - Send a `ProtocolMessage::TransferResume` to the peer via a new business stream
     - Wait for the resumed stream to arrive (the sender opens a new stream)
   - Store partial state: `transfer_id -> (header_bytes, chunks_received_so_far: Vec<u8>)` in a receiver-side map

   **Important edge cases**:
   - If the receiver has already received all chunks (error was on close, not read), just proceed with decrypt
   - If resume fails (sender cache expired), log a warning. The next clipboard change will trigger a full re-send naturally.
   - Do NOT retry indefinitely -- one resume attempt per transfer, then give up

5. **Wire the resume cache** into the existing `run_event_loop` / business command handler:
   - Create the `resume_cache` HashMap before the event loop starts
   - Pass it (as `Arc<RwLock<...>>`) to both the outbound command handler and the inbound stream handler
   - The outbound handler writes to the cache after send
   - The inbound handler reads from the cache when it receives a TransferResume request

6. Use `tracing::info_span!` for resume operations:
   - `info_span!("transfer.resume.request", transfer_id = %id, start_chunk)` for incoming resume requests
   - `info_span!("transfer.resume.send", transfer_id = %id, chunks_remaining)` for outbound resume sends
