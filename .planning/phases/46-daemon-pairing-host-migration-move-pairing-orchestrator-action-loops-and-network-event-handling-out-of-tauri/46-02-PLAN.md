# Plan 46-02: Daemon Pairing Control Surface And Realtime Contract

## Goal

Add the daemon-owned control surface and realtime contract that Tauri will consume in Phase 46, while preserving the metadata boundary for ordinary session reads and reserving verification details for authenticated realtime delivery only.

<action>
Create `src-tauri/crates/uc-daemon/src/api/pairing.rs` and export it from `src-tauri/crates/uc-daemon/src/api/mod.rs`.

Define these exact request/response DTOs in `api/pairing.rs` with `#[serde(rename_all = "camelCase")]`:

- `InitiatePairingRequest { peer_id: String }`
- `VerifyPairingRequest { pin_matches: bool }`
- `SetPairingParticipantRequest { client_kind: String, ready: bool }`
- `AckedPairingCommandResponse { session_id: String, accepted: bool, state: String, error: Option<String> }`

Update `src-tauri/crates/uc-daemon/src/api/routes.rs` to add these authenticated routes:

- `PUT /pairing/participants/current`
- `POST /pairing/sessions`
- `POST /pairing/sessions/:session_id/accept`
- `POST /pairing/sessions/:session_id/reject`
- `POST /pairing/sessions/:session_id/cancel`
- `POST /pairing/sessions/:session_id/verify`

All mutation routes must dispatch into daemon host methods with the same explicit names:

- `set_participant_ready`
- `initiate_pairing`
- `accept_pairing`
- `reject_pairing`
- `cancel_pairing`
- `verify_pairing`

Use these concrete HTTP semantics so Tauri does not have to rediscover them:

- successful mutation acknowledgment returns `202 Accepted` and `AckedPairingCommandResponse`
- active-session conflict returns `409` with `{"error":"active_pairing_session_exists"}`
- no-local-participant-ready returns `412` with `{"error":"no_local_pairing_participant_ready"}`
- missing session for follow-up mutation returns `404`
- malformed request body returns `400`

`POST /pairing/sessions` must accept only `peerId`; every follow-up route must key off `:session_id` and must not accept a replacement `peerId` field.
</action>

<action>
Update `src-tauri/crates/uc-daemon/src/api/types.rs` and `src-tauri/crates/uc-daemon/src/api/ws.rs` so the existing `pairing` topic supports both metadata snapshots and authenticated incremental updates.

Add these exact payload structs in `api/types.rs` with `#[serde(rename_all = "camelCase")]`:

- `PairingSessionChangedPayload { session_id: String, state: String, peer_id: Option<String>, device_name: Option<String>, updated_at_ms: i64 }`
- `PairingVerificationPayload { session_id: String, peer_id: String, device_name: Option<String>, code: String, local_fingerprint: String, peer_fingerprint: String }`
- `PairingFailurePayload { session_id: String, peer_id: Option<String>, error: String }`

Keep `PairingSessionSummaryDto` and `pairing.snapshot` metadata-only. They must not include:

- `code`
- `localFingerprint`
- `peerFingerprint`
- `keyslotFile`
- raw `challenge`

Extend `src-tauri/crates/uc-daemon/src/api/ws.rs` to emit these exact realtime type strings for the `pairing` topic:

- `pairing.updated`
- `pairing.verification_required`
- `pairing.complete`
- `pairing.failed`

Use one daemon-side broadcast path so authenticated websocket subscribers receive these incremental events after the initial `pairing.snapshot`. `pairing.snapshot` must continue to come from `RuntimeState` summaries. Verification payloads must come only from authenticated realtime fanout, not from `DaemonQueryService::pairing_session(...)`.

Add `src-tauri/crates/uc-daemon/tests/pairing_ws.rs` with tests for:

- snapshot contains `sessionId` and omits verification secrets
- incremental verification event contains `code`, `localFingerprint`, and `peerFingerprint`
- websocket event uses `type`, not `event_type`
- `/pairing/sessions/:session_id` response omits verification secrets even while realtime verification event is available
  </action>
