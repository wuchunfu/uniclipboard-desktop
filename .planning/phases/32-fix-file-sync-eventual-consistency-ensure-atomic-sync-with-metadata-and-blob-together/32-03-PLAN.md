  <action>
    0. **Rename existing event channels to `file-transfer://` prefix**: Update `transfer_progress.rs` to emit `file-transfer://progress` (was `transfer://progress`) and `file-transfer://error` (was `transfer://error`). This unifies all file transfer events under the `file-transfer://` namespace.

    1. **Extract file transfer wiring**: Create `file_transfer_wiring.rs` containing all 6 new behaviors (pending emit, progress promotion, timeout sweep, completion, failure, startup reconciliation). Add `pub mod file_transfer_wiring;` to `bootstrap/mod.rs`. `wiring.rs` retains only a call site into `file_transfer_wiring` for integration.

    2. In the inbound clipboard receive loop, when `InboundApplyOutcome::Applied` contains `pending_transfers`, emit `file-transfer://status-changed` for each transfer after the pending rows are already persisted.

    3. Use the first receiving-side `TransferProgress` event to mark `transferring`. Later progress updates must still refresh durable liveness by updating the repository timestamp even if they do not rewrite the semantic status.

    4. Add a periodic timeout sweep using `tokio::spawn` + `tokio::time::interval(Duration::from_secs(15))`:
       - fails `pending` rows older than 60 seconds
       - fails `transferring` rows older than 5 minutes without new chunk activity
       - emits `file-transfer://status-changed` for the resulting failures
       - 15-second interval ensures 60-second pending timeout is detected within 75 seconds (worst-case 15s delay)
       - sweep task holds a weak reference or clone of `AppRuntime`, avoids blocking shutdown
       - sweep failures only `warn!` log, never panic

    5. In timeout and failure paths, delete partial cache artifacts before or immediately after marking the row failed. For startup reconciliation, use the cleanup targets returned by the app use case and delete the associated `cached_path` or transfer directory best-effort.

    6. In the `FileTransferCompleted` path:
       - keep the existing hash verification and clipboard restore behavior
       - mark the durable transfer row completed before emitting the frontend status update
       - continue emitting the existing `file-transfer://completed` event (existing UI code depends on it). The new `file-transfer://status-changed` is the durable status authority. Both events are emitted in parallel; the old event may be removed in a future phase.

    7. In the `FileTransferFailed` path:
       - mark the durable row failed with the error reason
       - clean any partial download path owned by that transfer
       - emit `file-transfer://status-changed` with `status = "failed"`

    8. Add a startup background step that runs the app-layer reconciliation use case and marks lingering `pending`/`transferring` rows failed. Delete any returned stale cache artifacts, log failures as warnings, and keep startup non-fatal.

    9. Implement the event payload as a dedicated struct with `#[serde(rename_all = "camelCase")]` and add tests that assert `entryId` and `transferId` serialize in camelCase.

  </action>
