## Decisions Made

- Reset is idempotent: missing key material or missing paired-device rows are treated as already-clean rather than fatal.
- Pairing host leases are cleared centrally in `DaemonPairingHost::reset_setup_state()` so setup reset does not need to guess client kinds.
- CLI retry wording is fixed to `Passphrase rejected; retrying current join session`, and the join flow stays on the passphrase prompt path instead of aborting.

## Operator Walkthrough

1. Terminal A: `UC_PROFILE=peerA UC_CLIPBOARD_MODE=full uniclipboard-cli setup host`
2. Terminal B: `UC_PROFILE=peerB UC_CLIPBOARD_MODE=passive uniclipboard-cli setup join`
3. On B, choose peerA from the discovered peer list.
4. On A, accept the inbound join request.
5. On B, enter a wrong passphrase once.
6. Confirm B prints `Passphrase rejected; retrying current join session` and remains in the current join session.
7. On B, enter the correct passphrase.
8. After completion, run `uniclipboard-cli devices` on both A and B.
9. Run `uniclipboard-cli setup reset` on both A and B.
10. Rerun the full host/join sequence:
    - Terminal A: `UC_PROFILE=peerA UC_CLIPBOARD_MODE=full uniclipboard-cli setup host`
    - Terminal B: `UC_PROFILE=peerB UC_CLIPBOARD_MODE=passive uniclipboard-cli setup join`

## Success Criteria for the Walkthrough

- A and B `devices` outputs both include the opposite peer.
- B `setup status` or terminal end-state shows join/encryption-ready completion, not only pairing success.
- Both peers can run `uniclipboard-cli setup reset` and start a second host/join round without manual DB, vault, or profile cleanup.
