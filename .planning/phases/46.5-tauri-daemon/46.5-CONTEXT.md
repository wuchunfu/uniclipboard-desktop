# Phase 46.5: 将配对业务逻辑从 Tauri 层彻底移除，统一收口到 daemon - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

让 `uc-tauri` 不再承载任何实际配对业务逻辑，只保留前端命令转发、daemon 连接管理、以及 daemon realtime 事件消费；配对协议入口、配对状态机、配对会话、accept/reject/verify、以及 setup 阶段的配对桥接，全部由 daemon 独占。

本阶段聚焦于：

- 在共享 bootstrap 中显式表达 pairing runtime ownership，并区分 GUI / daemon
- GUI 模式下彻底禁用本地 pairing runtime、本地 pairing accept loop、以及本地 pairing 协议注册
- daemon 模式下保留当前 pairing stream / pairing host / pairing orchestrator，作为唯一配对入口
- 清理 `uc-tauri` 内所有本地 pairing 业务路径、兜底逻辑、桥接与注释，只保留 daemon adapter 责任
- 保持前端现有 pairing wire contract 不变，前端继续只消费 daemon realtime

本阶段不包括：

- 重做前端配对交互或改动前端事件契约
- 拆分两套独立 bootstrap / wiring 实现
- 保留任何 GUI 本地 pairing 兜底模式
- 把 daemon pairing ownership 再次回退给 Tauri

</domain>

<decisions>
## Implementation Decisions

### Runtime Ownership

- 在 `uc-bootstrap` 的平台层引入一个小而明确的 ownership/config 类型，表达“GUI 不拥有 pairing runtime，daemon 拥有 pairing runtime”。
- `Libp2pNetworkAdapter::new(...)` 增加显式配置参数，声明是否启用本地 pairing runtime。
- 不拆分 bootstrap 为两套 wiring；继续共享 composition root，但在创建 `Libp2pNetworkAdapter` 时显式传入 ownership 配置。

### GUI Boundary

- GUI 模式下 `Libp2pNetworkAdapter` 不初始化 `PairingStreamService`。
- GUI 模式下不启动 pairing accept loop，不注册 `/uniclipboard/pairing-stream/1.0.0`。
- GUI 模式下 `NetworkPorts.pairing` 不再绑定到 GUI 本地 `libp2p_network` 的真实 pairing transport。
- setup 阶段只允许通过 daemon-backed facade 与 daemon 交互，不允许回落到本地 pairing transport。
- Tauri commands 中配对相关命令只做 daemon route 调用与错误映射，不得直接驱动本地配对业务对象。

### Daemon Ownership

- daemon 保留当前 pairing stream、pairing host、pairing orchestrator，继续作为唯一配对入口。
- daemon 继续负责 inbound request、verification、complete、failed 的 WS/realtime 投递。
- GUI 的唯一配对业务事件源仍然是 daemon realtime，而不是本地 pairing 网络事件。

### Tauri Cleanup

- `bootstrap/wiring.rs` 中当前对本地 `PairingMessageReceived` / `PairingFailed` 的忽略逻辑不应继续作为运行时兜底；GUI 路径不应再收到这类事件。
- 任何依赖 `deps.network_ports.pairing` 承载 GUI 本地 pairing 业务的路径，都改为 daemon client / daemon realtime。
- 任何仅为本地 pairing runtime 存在的 Tauri 层桥接、日志、注释、回退路径，全部删除。

### Interface And Behavior Lock

- 不改变前端 wire contract；前端仍消费 `daemon://realtime`，包括：
  - `pairing.updated`
  - `pairing.verificationRequired`
  - `pairing.complete`
  - `pairing.failed`
- 允许 `uc-tauri` 保留配对命令 API 外壳，但这些命令只能调用 daemon，不得拥有本地业务状态。

### Claude's Discretion

- ownership/config 类型的具体命名与落点，只要语义足够明确且不破坏分层
- GUI 模式下 `NetworkPorts.pairing` 的具体装配形式，只要能够彻底切断真实本地 pairing transport
- 清理遗留代码时的模块边界和删除顺序，只要最终不再保留 GUI 本地 pairing runtime

</decisions>

<specifics>
## Specific Ideas

- 默认决策：不保留任何“GUI 本地 pairing 兜底模式”。
- 默认决策：不拆分 bootstrap 为两套独立实现，而是在共享 composition root 上显式表达 ownership。
- 默认决策：前端配对体验的唯一事件源是 daemon realtime，不接受本地网络事件桥接回前端。
- 默认决策：Tauri 层允许保留配对命令与 facade，但它们仅是 daemon adapter，不属于业务逻辑。

</specifics>

<validation>
## Validation Expectations

### Targeted Tests

- `uc-platform`
  - 新增单元测试：当本地 pairing runtime 被禁用时，`spawn_swarm()` 不初始化 pairing service，不启动 pairing accept loop。
  - 保留 daemon / 启用模式下现有 pairing transport 测试通过。
- `uc-bootstrap` 或 `uc-tauri` 集成测试
  - GUI bootstrap 构建结果中，本地网络不提供实际 pairing protocol ownership。
  - daemon bootstrap 构建结果中，pairing runtime 仍正常可用。
- `uc-tauri` / 前端契约测试
  - dashboard 状态下仅通过 daemon realtime 驱动配对 toast。
  - GUI 不再因为本地 pairing 网络事件触发任何业务路径。

### Regression Checks

- 复现当前场景：peerB 发起 join 后，peerA 的提示必须来自 daemon realtime。
- 运行时日志中，GUI 路径不应再出现本地 pairing session started 这类 pairing protocol 启动痕迹。
- daemon 路径保留该能力并正常完成 accept / reject / verify。

</validation>

<execution_order>

## Suggested Execution Order

1. 先在 `uc-platform` 给 `Libp2pNetworkAdapter` 加本地 pairing runtime 开关，并补失败测试。
2. 在 `uc-bootstrap` 把 GUI / daemon 的 ownership 显式传入共享 wiring。
3. 调整 `NetworkPorts.pairing` 的 GUI 装配，切断 GUI 对真实本地 pairing transport 的依赖。
4. 清理 `uc-tauri` 中本地 pairing 事件相关兜底路径与注释，确保只剩 daemon 驱动。
5. 跑针对性 Rust 测试与前端 / 契约测试，确认 GUI 不再承接配对业务，daemon 仍完整工作。

</execution_order>
