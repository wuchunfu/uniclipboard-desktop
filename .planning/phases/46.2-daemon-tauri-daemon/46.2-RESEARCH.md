# Phase 46.2: Daemon Pairing Flow Hard Cutover - Research

**Researched:** 2026-03-21
**Domain:** Daemon-hosted pairing flow with Tauri bridge hard cutover
**Confidence:** HIGH

<user_constraints>

## User Constraints (from CONTEXT.md)

### Locked Decisions

- Phase 46.2 completes the migration so that all pairing business logic and session lifecycle run on the daemon, not inside `uc-tauri`, while preserving the existing user-facing pairing flow.
- `uc-tauri` must become a thin daemon client + realtime bridge for pairing: it must not own any pairing state machine, session lifecycle, or retry/timeout policy.
- All pairing business decisions (discoverability, admission, concurrency, timeouts, recovery) live only in `uc-app` + `uc-daemon` after this phase.
- Tauri pairing entrypoints are reduced to translating GUI user intent into daemon API calls, forwarding daemon pairing/realtime events into the existing frontend event contract, and keeping at most thin UI-facing flags for local view state.
- After 46.2 there must be no alternative Tauri-owned pairing host path; any legacy pairing loops, orchestrators, or direct libp2p flows under `src-tauri/crates/uc-tauri/` that bypass the daemon must be removed or converted into daemon-client shims.
- Existing desktop frontend pairing semantics must remain intact in this phase: request → verification → verifying → complete/failed, with the same high-level steps and mental model.
- Business decisions about discoverability, active-session checks, and admission rules remain on the daemon. Frontend/Tauri may track local UI gating state but must not form a second divergent pairing state machine.
- There is no runtime rollback switch: once 46.2 lands, there is no feature flag to route pairing back through legacy Tauri hosts; recovery is via git (fix-forward or revert).

### Claude's Discretion

- Exact internal naming and decomposition of daemon pairing client helpers in `uc-tauri` / `uc-cli`, provided they remain thin bridges.
- Detailed structure of daemon pairing DTOs and client interfaces, as long as they respect Phase 45/46 sensitivity and metadata boundaries (no verification secrets in generic read models).
- Specific error-code taxonomy for busy/active-session/no-local-participant scenarios.
- Wiring details for UI gating flags (subscriptions vs on-demand queries) so long as there is no second business state machine inside Tauri/frontend.

### Deferred Ideas (OUT OF SCOPE)

- New or richer pairing UX flows (additional steps, modes, recovery UI, multi-session queueing) — reserved for later phases (47/48).
- Rich CLI pairing UX and status reporting beyond reusing the daemon pairing surface.
- Operator/administrative pairing controls (force-close, inspection beyond end-user needs).

## Summary

Phase 46.2 是对 46 / 46.1 的“硬切换”收尾：目标是让所有配对业务逻辑（会话生命周期、超时、并发控制、发现/参与就绪判断）只在 daemon（`uc-daemon` + `uc-app`）里运行，`uc-tauri` 只负责当壳和桥——把 GUI 意图翻译成 daemon API 调用，把 daemon 的实时事件转回现有前端事件契约。

当前代码已经基本实现 daemon 侧的配对 host 能力：`uc-daemon::pairing::host` 管配对状态机和会话、`uc-app::usecases::pairing::orchestrator` 提供 Tauri-free use case、`uc-daemon::api::{server,ws,state}` 提供 HTTP/WS 和 runtime state。前端也已经通过统一的 `DaemonWsBridge` 接收 daemon 事件。然而，Tauri 仍然保留了几条“旧路”：`commands/pairing.rs` 里还有直接使用 `State<Arc<PairingOrchestrator>>` 的命令，`bootstrap/wiring.rs` 之前的配对事件订阅和 host 逻辑需要确保彻底退出历史舞台。46.2 要做的，就是把这些残余 host 路全清干净，并确认所有 GUI 可见的配对入口（PairingDialog、PairingNotificationProvider、Setup 流）都已经只通过 daemon surface 工作。

对 planner 来说，核心是：不要“改 UI 体验”或“重新设计 API”，而是精确识别和删掉所有非 daemon host 的路径，同时用测试和日志确认新路径在行为上与旧路径等价。实现顺序也很重要：先锁定 daemon host 和 API 已经完整可用，再逐步把 Tauri 命令和事件桥指到 daemon，然后才删除旧 wiring，最后做一次 UX + 集成回归。任何在 Tauri 里新建“简易状态机”或“本地超时逻辑”的改动，都是和本阶段目标冲突的。

**Primary recommendation:** 以“单一 daemon pairing host + 单一 DaemonWsBridge”为唯一真实来源，系统性删除所有 Tauri 本地 host/loop，实现“桥比最薄还薄”，并用对照测试保证 GUI 配对流与旧实现行为等价。

### Pattern 1: 单一 daemon pairing host + 统一 WS bridge

**What:**

- 只允许 daemon（`uc-daemon::pairing::host` + `uc-app` usecases）对 pairing 状态机有“写权限”和“生命周期控制”。
- 所有 GUI/CLI 通过：
  - HTTP/JSON-RPC mutation（initiate / accept / reject / verify / cancel），以及
  - WebSocket topic（pairing verification events, space access completed events, discovery 更新）。
- `uc-tauri` 通过 `DaemonWsBridge` 订阅这些 topic，然后用现有事件名强类型转发给前端（`p2p-pairing-verification`, `p2p-space-access-completed`, `p2p-peer-discovery-changed` 等）。

**When to use:**

- 任何需要改配对行为、增加新事件、或扩展 pairing host 行为时，都必须在 daemon host / app usecase 中改，而不是在 Tauri 或前端创建“本地状态机”。

**Example:**（简化示意，非完整代码）

```rust
// src-tauri/crates/uc-daemon/src/pairing/host.rs

pub struct DaemonPairingHost {
    orchestrator: Arc<PairingOrchestrator>,
    sessions: Arc<PairingSessionManager>,
    // ... runtime state, event tx, etc.
}

impl DaemonPairingHost {
    pub async fn initiate(&self, peer_id: String) -> Result<SessionId> {
        // 所有业务规则（是否允许新 session、busy、headless 等）在这里判断
        self.ensure_can_start_new_session(&peer_id).await?;
        self.orchestrator.initiate_pairing(peer_id).await
    }

    pub async fn handle_user_accept(&self, session_id: &str) -> Result<()> {
        self.orchestrator.user_accept_pairing(session_id).await
    }

    // ... 处理网络 PairingMessage / 超时等
}
```

### Pattern 2: Tauri 只做“桥”的 commands

**What:**

- `src-tauri/crates/uc-tauri/src/commands/pairing.rs` 中所有命令要统一形态：
  - 从 `State<Arc<AppRuntime>>` 或 `State<DaemonPairingClient>` 获取 daemon 客户端。
  - 调用 daemon HTTP/JSON-RPC API，处理错误为 `CommandError`，并在必要时发 `p2p-command-error` 事件。
  - 不持有 `PairingOrchestrator`，也不直接 manipulate `PairingSessionManager`。

**When to use:**

- 任何新增/修改 pairing 相关 Tauri 命令时。

**Example:**

```rust
// 目标模式：不再依赖 State<Arc<PairingOrchestrator>>

#[tauri::command]
pub async fn initiate_p2p_pairing(
    runtime: State<'_, Arc<AppRuntime>>,
    request: P2PPairingRequest,
    _trace: Option<TraceMetadata>,
) -> Result<P2PPairingResponse, CommandError> {
    let span = info_span!("command.pairing.initiate", peer_id = %request.peer_id);
    record_trace_fields(&span, &_trace);
    async {
        let client = runtime.daemon_clients().pairing();
        client
            .initiate(request.peer_id)
            .await
            .map_err(|e| map_pairing_client_error("initiate_p2p_pairing", e))
    }
    .instrument(span)
    .await
}
```

> 规划时要明确：现有实现仍然是 `State<Arc<PairingOrchestrator>>` 直接调用 orchestrator，本阶段要规划出替换到 daemon client 的步骤，再删掉 `manage(pairing_orchestrator)`。

### Anti-Patterns to Avoid

- **Anti-pattern 1:** 在 Tauri 或前端新建本地 pairing state machine。
  - 问题：造成“双源事实”，与 daemon host 行为不一致；debug 极难。
  - 正确做法：把所有业务状态留在 daemon，仅在 front/Tauri 维护显示用 UI state（dialog open、当前选中 peer）。

- **Anti-pattern 2:** 继续在 `bootstrap/wiring.rs` 里维护配对事件订阅/重试 loop。
  - 问题：与 46.1 的 DaemonWsBridge 统一 realtime 模型冲突，会出现重复事件或错乱。
  - 正确做法：所有 pairing 事件订阅必须通过 DaemonWsBridge；wiring 中只保留“非 pairing” 的后台任务。

- **Anti-pattern 3:** 在 Tauri 里实现“如果 daemon 不可用就 fallback 旧配对流程”。
  - 问题：违反 46.2 “硬切换，无 feature flag”的决策，会留下第三种 host 模式。
  - 正确做法：如果 daemon 不可用，命令直接失败，前端给出明显错误；不回退到 legacy host。

## 不要手搓（Don’t Hand-Roll）

| Problem                               | Don't Build                                      | Use Instead                                                                             | Why                                                                                       |
| ------------------------------------- | ------------------------------------------------ | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Pairing 状态机、事件处理、超时策略    | 自己在 Tauri/前端/CLI 里做简单 `enum State` 切换 | `uc-core::network::pairing_state_machine` + `PairingOrchestrator` + `DaemonPairingHost` | 现有实现已处理 protocol version、超时、重试、会话 ID 等细节，自己重写很难不引入边缘 bug。 |
| Pairing realtime 通道                 | 额外的 tauri event channel 或自定义 WS 通道      | `DaemonWsBridge` + daemon `ws.rs` topics                                                | 统一 WS 通道便于调试和权限控制，多通道很难 trace。                                        |
| Pairing 结果缓存 / 本地“当前会话”管理 | 在 React/Tauri 中维护“当前 sessionId”全局变量    | 使用 daemon session snapshot + events                                                   | daemon 已持久化/管理生命周期，本地缓存容易和真实状态不同步。                              |
| 设备发现/连接状态维护                 | 前端自己轮询 + 本地数组                          | daemon discovery read models + `onP2PPeerDiscoveryChanged` 等事件                       | 已有 snapshot-first + 增量更新模式，手写轮询浪费资源且易错。                              |

**Key insight:** 本阶段价值不在于重造 pairing 轮子，而是在于把轮子搬到 daemon，并保证只有一个轮子在转。任何在 Tauri/前端重现业务逻辑的尝试，都等价于新造一个轮子。

## Current vs Target Implementation Map

### 1. 用户可见 pairing 流程概览

用户视角有两大类 pairing 入口：

1. **主动发起 pairing（initiator）** — 通过 `PairingDialog`：
   - 打开对话框 → 搜索设备（Discovery）→ 选择设备 → 等待 PIN → 对比 PIN → 确认/拒绝 → 成功/失败提示。
2. **被动接受 pairing（responder）** — `PairingNotificationProvider`：
   - 收到 “Pairing Request” toast → 用户点 Accept / Reject → 若接受则显示 PIN 对话框 → 对比 PIN → 成功/失败 toast + setup 流结束。

此外还有 Setup 流里的 pairing（加入空间、信任确认等），通过 `SetupOrchestrator` 和 `SetupActionExecutor` 消费 pairing 事件驱动。

### 2. 现有实现下的分层映射（Legacy + 部分 daemon）

#### 主动发起 pairing（PairingDialog）

1. 前端：`PairingDialog` 调用 `getP2PPeers()` → `p2p.ts` → `invoke('get_p2p_peers')` → `commands/pairing.rs::get_p2p_peers`。
   - `get_p2p_peers` 使用 `runtime.usecases().list_discovered_peers()` + `list_connected_peers()`（已经是 daemon/网络抽象后的 usecase）。
2. 用户选择 peer 后，`handleConnect` 调用 `initiateP2PPairing` → `invoke('initiate_p2p_pairing', { request })` → `commands/pairing.rs::initiate_p2p_pairing`。
   - 目前 `initiate_p2p_pairing` 直接依赖 `State<Arc<PairingOrchestrator>>`：
     ```rust
     let session_id = orchestrator
         .initiate_pairing(request.peer_id)
         .await?
     ```
   - Orchestrator 内部使用 `PairingSessionManager` + libp2p 网络端口执行 pairing actions（通过 Tauri wiring 中的 NetworkPorts）。
3. Daemon 侧现在也有 pairing host（`uc-daemon::pairing::host`），但发起 path 仍然走 Tauri 本地 orchestrator，这就是“第二 host”。
4. PIN 显示：`PairingDialog` 通过 `onP2PPairingVerification` 监听 `p2p-pairing-verification` Tauri event，事件来自 Tauri wiring 对 pairing domain events 的转发。

#### 被动接受 pairing（PairingNotificationProvider）

1. 通过 `onP2PPairingVerification` 监听 pairing 事件：
   - kind=`request` → toast → Accept/Reject 按钮。
   - Accept → 调用 `acceptP2PPairing(sessionId)` → `invoke('accept_p2p_pairing')` → `commands/pairing.rs::accept_p2p_pairing` → orchestrator.
   - Reject → `rejectP2PPairing` 同理。
2. kind=`verification` → 显示 PIN 对话框（被动端）。
3. kind=`verifying`/`complete`/`failed` → 更新 UI + toast。
4. `onSpaceAccessCompleted` 用于 setup 相关事件：监听 `p2p-space-access-completed`。

#### Setup 流

- `uc-app::setup::{orchestrator,action_executor}` 使用 `PairingEventPort` 消费 pairing domain events，并驱动 SetupState（EnsurePairing → SpaceAccess 等）。
- Tauri 目前在 wiring 中订阅 pairing 网络事件，再翻译为 pairing domain events 和 frontend events。

#### Daemon 侧

- `uc-daemon::pairing::host` 已经管理 pairing 生命周期并将 session metadata 投影到 `RuntimeState`。
- `uc-daemon::api::server` 暴露 pairing 相关 HTTP/JSON-RPC endpoint（从 46/45 上下文推断）。
- `uc-daemon::api::ws` 发布 pairing 相关 websocket 事件（session summaries、verification updates 等）。
- Guessed/来自 context：Tauri 通过 `DaemonWsBridge` 统一订阅这些 topic，再派发给前端（但旧 wiring 可能仍有残留）。

### 3. 目标实现下的完整链路（46.2 之后）

#### 主动发起 pairing

1. 前端（PairingDialog）：
   - Discovery 流保持不变：`getP2PPeers()` → `get_p2p_peers` 命令 → usecases → daemon/网络层（已经 daemon-friendly）。
   - 发起：`initiateP2PPairing({ peerId })` → `invoke('initiate_p2p_pairing')`。
2. Tauri pairing command：
   - 不再注入 `State<Arc<PairingOrchestrator>>`。
   - 改为使用 `AppRuntime` 暴露的 `daemon_clients().pairing()` 或直接 HTTP client：
     ```rust
     let session = runtime
         .daemon_clients()
         .pairing()
         .initiate(peer_id)
         .await?;
     ```
3. Daemon pairing host：
   - 接到 `initiate` API 后，在 `DaemonPairingHost` 内部调用 `PairingOrchestrator::initiate_pairing`；host 决定是否允许新会话、并更新 runtime state。
4. Verification 流事件：
   - Daemon 通过 ws topic（例如 `pairing.verification`）发事件。
   - `DaemonWsBridge` 接收后，统一映射为 `p2p-pairing-verification` Tauri event；事件 payload 保持现有 `P2PPairingVerificationEvent` shape。
   - 前端 `PairingDialog` 和 `PairingNotificationProvider` 继续订阅 `onP2PPairingVerification`，无需更改类型或事件名。

#### 被动接受 pairing & Setup 流

1. 被动端 pairing request：
   - Daemon host 接到远端 pairing request → 调用 app-layer 事件（PairingEvent）→ runtime 将其投影为 ws topic event。
   - Tauri 通过 DaemonWsBridge 订阅该 topic，并 emit `p2p-pairing-verification` kind=`request`。
   - `PairingNotificationProvider` 行为不变，只是事件真正来自 daemon。
2. 用户 Accept/Reject：
   - `acceptP2PPairing`/`rejectP2PPairing` 命令改为调用 daemon pairing API，而非本地 orchestrator。
3. Setup：
   - Setup 应该已经在 46/46.1 中调整为消费 app-layer pairing events，经由统一事件 hub；46.2 只需确保这些事件现在来自 daemon host，而不是 Tauri wiring 中某个遗留 loop。

### 4. Planner 需要知道的差异点

- 对 GUI 和 TS API 而言，变化应“不可见”：所有函数名、事件名、结构体保持不变。
- 对 runtime 的变化：
  - Tauri 不再 `manage(pairing_orchestrator)`；AppRuntime 也不再需要知道 pairing host 的内部细节。
  - `start_background_tasks` 中不应该再有任何 pairing event 订阅或 host 逻辑；daemon host 取而代之。
  - DaemonWsBridge 必须订阅 pairing topics，并覆盖之前 Tauri 本地 host 发事件的职责。

## Common Pitfalls / Risks

### Pitfall 1: 留下第二个 pairing host（Tauri + daemon 双 host）

**What goes wrong:**

- 部分路径（例如 PairingDialog 发起）走 Tauri 本地 orchestrator，其他路径（CLI 或 future flows）走 daemon host。
- 会话 ID、验证状态在不同 host 中不一致；日志很难追踪。

**Why it happens:**

- 迁移时只改了一部分命令/事件，或为了“兼容旧代码”临时保留了 orchestrator 注入。

**How to avoid:**

- 规划时列出所有 `PairingOrchestrator` 的使用点（目前 Tauri commands 和 wiring），逐一替换为 daemon 客户端，最后删掉 Tauri 对 orchestrator 的 `manage`。
