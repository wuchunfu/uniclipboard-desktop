## Summary

当前代码已经把 46.3 做到了“启动期兼容探测 + 一次有界 replacement”，但并没有建立“运行期 GUI 对 daemon 的父子所有权”。根因很直接：`src-tauri/crates/uc-tauri/src/bootstrap/run.rs` 在真实 bootstrap 路径里临时创建 `DaemonBootstrapOwnershipState::default()`，随后只把 `spawn_daemon_process()` 的 `child.id()` 记进快照，`Child` 句柄本身立刻丢弃；bootstrap 结束后，这个 ownership state 也不再被任何 managed state 持有。结果是 GUI 既不知道自己是否真的拥有这个 daemon，也没有可回收的 live child handle。

同时，当前仓库对“关闭窗口”和“退出应用”是严格分开的：`src-tauri/src/main.rs` 在主窗口 `CloseRequested` 时 `prevent_close + hide`，macOS 还会隐藏 Dock 图标；真正退出只发生在 tray `Quit` 的 `app.exit(0)` 或其他 app-level quit 路径，并最终落到 `RunEvent::ExitRequested/Exit`。所以 46.6 不能把关闭主窗口当成退出信号，否则会直接破坏当前的 tray 常驻语义。

46.6 最稳妥的规划方向是：保留 46.3 的 bootstrap snapshot 作为“替换事实记录”，另外新增一个单独的、可 `.manage()` 的 runtime lifecycle owner contract，专门持有 GUI-owned daemon 的 live child handle、pid、spawn reason 与 exit-in-progress guard；实际退出时在 `RunEvent::ExitRequested` 做 `prevent_exit`，完成有界清理后再放行/重新触发退出。不要把 live child handle 塞回 46.3 那个 snapshot-only state，也不要退化成只靠 pid file 的 kill-by-pid。

**Primary recommendation:** 新增单独的 GUI daemon lifecycle owner contract，并在 `RunEvent::ExitRequested` 上做显式 cleanup；46.3 的 `DaemonBootstrapOwnershipState` 保持为 bootstrap facts，不承担运行期 child ownership。

## Architecture Patterns

### Recommended Project Structure

```text

├── crates/uc-tauri/src/bootstrap/
```

### Pattern 1: Split Bootstrap Facts from Live Ownership

**What:** 把“replacement attempt / last incompatible reason / spawned pid snapshot”与“当前 GUI 是否拥有一个可清理的 daemon child handle”拆成两层状态。

**When to use:** 46.3 的 replace-or-fail 逻辑继续保留事实快照；46.6 的 app exit cleanup 只读取 live owner contract。

**Example:**

```rust
// Source: local repo evidence + recommended contract
#[derive(Debug, Clone, PartialEq, Eq, Default)]
pub struct DaemonBootstrapOwnershipSnapshot {
    pub replacement_attempt: u8,
    pub spawned_child_pid: Option<u32>,
    pub last_incompatible_reason: Option<String>,
}

pub struct GuiOwnedDaemonState {
    child: std::sync::Mutex<Option<OwnedDaemonChild>>,
    exit_in_progress: std::sync::atomic::AtomicBool,
}

pub struct OwnedDaemonChild {
    pub pid: u32,
    pub spawn_reason: SpawnReason,
    pub child: std::process::Child,
}
```

### Pattern 2: Register Ownership Only on Spawn Paths

**What:** 只有 `ProbeOutcome::Absent` 直接 spawn，或 `ProbeOutcome::Incompatible` replacement 后 spawn 成功，才登记 `GuiOwnedDaemonState`；`ProbeOutcome::Compatible` 不登记。

**When to use:** `bootstrap_daemon_connection()` 内部完成 compatible-ready 后。

**Example:**

```rust
// Source: local repo semantics + recommended contract
match probe().await? {
    ProbeOutcome::Compatible(_) => owner.clear(),
    ProbeOutcome::Absent => {
        let child = spawn_daemon_process()?;
        owner.record_spawned(child, SpawnReason::Absent);
        wait_for_daemon_health(...).await?;
    }
    ProbeOutcome::Incompatible { .. } => {
        replace_incompatible_daemon(...).await?;
        let child = spawn_daemon_process()?;
        owner.record_spawned(child, SpawnReason::Replacement);
        wait_for_daemon_health(...).await?;
    }
}
```

### Pattern 3: ExitRequested Is the Cleanup Gate

**What:** 在 `RunEvent::ExitRequested` 中，如果存在 GUI-owned daemon，就 `prevent_exit()`、启动一次有界 cleanup、成功后再调用 `app.exit(0)`；没有 owned child 时直接放行。

**When to use:** tray `Quit`、Cmd+Q、app-level quit 等真实退出路径。

**Example:**

```rust
// Source: Tauri docs + repo recommendation
match event {
    tauri::RunEvent::ExitRequested { api, .. } if owner.begin_exit_cleanup() => {
        api.prevent_exit();
        let app = app_handle.clone();
        let owner = owner.clone();
        tauri::async_runtime::spawn(async move {
            let _ = owner.shutdown_owned_daemon().await;
            owner.finish_exit_cleanup();
            app.exit(0);
        });
    }
    tauri::RunEvent::Exit => {
        tracing::info!("application exiting");
    }
    _ => {}
}
```

### Pattern 4: CloseRequested Keeps Tray Semantics Untouched

**What:** 主窗口 `CloseRequested` 继续 `prevent_close + hide`，绝不能复用这条路径做 daemon shutdown。

**When to use:** `main` window 关闭事件。

**Example:**

```rust
// Source: current repo behavior in src-tauri/src/main.rs
if let tauri::WindowEvent::CloseRequested { api, .. } = event {
    if window.label() == "main" {
        api.prevent_close();
        let _ = window.hide();
    }
}
```

### Anti-Patterns to Avoid

- **把 hide-to-tray 当成 exit:** 这会在用户只是关闭窗口时误杀 daemon，并破坏现有托盘常驻行为。
- **只靠 pid file 做 exit cleanup:** pid file 解决的是 expected endpoint replacement，不是 parent-child ownership。
- **spawn 后立刻丢 `Child` 句柄:** 这是当前孤儿进程的直接根因。
- **在 `ExitRequested` 里不做 guard 就再次 `app.exit(0)`:** 很容易递归触发退出事件或重复清理。
- **把 46.6 扩大成 daemon 管理 API phase:** 这会和未来 CLI-F04 的 daemon stop surface 混淆。

## Don't Hand-Roll

| Problem                   | Don't Build                                               | Use Instead                                                                  | Why                                                                           |
| ------------------------- | --------------------------------------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | ----------------------------------------------------------- |
| GUI-owned daemon 退出回收 | `ps                                                       | grep` / 全机进程扫描                                                         | live `Child` handle + bounded exit cleanup                                    | 只有 live child handle 能证明“这是当前 GUI spawn 的 daemon” |
| 实际退出前 cleanup 时序   | 直接在 `RunEvent::ExitRequested` 里裸 `kill` 然后立刻退出 | `prevent_exit` + one-shot cleanup guard + completion-triggered `app.exit(0)` | 否则 cleanup 可能来不及执行，或重复执行                                       |
| graceful stop 通道        | 新增 daemon `/shutdown` 路由                              | 复用本地 parent-child ownership 与现有 signal helper                         | 46.6 的问题不是远程控制，而是本地 GUI spawn/exit 生命周期                     |
| replacement 后退出识别    | 再加一份“是谁 spawn 的” pid 文件                          | 在 owner state 中记录 `SpawnReason::{Absent, Replacement}`                   | pid file 面向 expected endpoint，不适合表达当前 GUI runtime 的 live ownership |

**Key insight:** 46.6 的核心不是“找到一个 pid 然后 kill 掉”，而是“让 GUI 对它亲自 spawn 的 daemon 保持一段可验证、可清理、不会误伤外部 daemon 的运行期所有权”。

## Common Pitfalls

### Pitfall 1: 误把主窗口关闭当成应用退出

**What goes wrong:** 用户关闭主窗口只是隐藏到 tray，daemon 却被杀掉，应用表面还在运行。
**Why it happens:** 当前 `src-tauri/src/main.rs` 的 `CloseRequested` 已经显式 `prevent_close + hide`，和退出不是一回事。
**How to avoid:** 只在 `RunEvent::ExitRequested` / `Exit` 链路做 daemon cleanup。
**Warning signs:** 点击窗口关闭按钮后，tray 仍在，但 daemon 已经不可达。

### Pitfall 2: 继续复用 46.3 的 pid metadata 做 exit ownership

**What goes wrong:** GUI 退出时把独立启动的兼容 daemon 也干掉。
**Why it happens:** pid file 表示的是“预期本地端点当前 daemon pid”，不是“由当前 GUI spawn 的 live child”。
**How to avoid:** pid file 只保留给 replacement；exit ownership 必须来自 live child handle。
**Warning signs:** GUI 连接到已有 compatible daemon 后，退出 GUI 会把那个 daemon 也杀掉。

### Pitfall 3: live child handle 没有跨 bootstrap 持久化

**What goes wrong:** 代码里明明“记录了 spawned pid”，但 exit 时无从 cleanup。
**Why it happens:** 当前真实路径里 `DaemonBootstrapOwnershipState` 是 `bootstrap_daemon_connection()` 内部局部值，bootstrap 后即被丢弃。
**How to avoid:** 把 live owner contract `.manage()` 到 Tauri app state，或由 `main.rs` 长生命周期持有。
**Warning signs:** 退出钩子拿不到任何 owned daemon state，只能再次读 pid file。

### Pitfall 4: 退出清理缺少幂等 guard

**What goes wrong:** tray quit、Cmd+Q、重复 `app.exit(0)` 或 cleanup 完成后再次触发 `ExitRequested`，导致双 kill、重复 wait、或退出死循环。
**Why it happens:** `RunEvent::ExitRequested` 是事件入口，不是天然 one-shot。
**How to avoid:** 给 owner contract 加 `exit_in_progress`，第二次进入时直接放行或忽略。
**Warning signs:** 日志里同一 pid 被终止两次，或应用退出卡住。

### Pitfall 5: Cargo 验证命令写成子串过滤

**What goes wrong:** `cargo test -p uc-tauri daemon_bootstrap_contract` 看起来绿了，实际上 3 个集成测试一个都没跑。
**Why it happens:** cargo 会把子串同时用于测试二进制和测试名过滤。
**How to avoid:** 集成测试固定用 `--test <binary>` 精确命令。
**Warning signs:** 输出里全是 `running 0 tests`，但退出码仍是 0。

## Open Questions

1. **是否要对 GUI-owned daemon 先发 TERM 再 fallback force-kill？**
   - What we know: `uc-daemon` 当前监听 `SIGTERM`/`Ctrl-C` 做优雅 shutdown；46.3 replacement 也使用 `kill -TERM`。
   - What's unclear: 46.6 是否只需“一定回收”还是“尽量优雅回收”。
   - Recommendation: 规划时按“两段式”设计，先复用现有 TERM helper，再在超时后用 live child handle 强制 kill 作为兜底。

2. **46.6 是否依赖 46.3-03/04 的 startup state/UI？**
   - What we know: 当前代码里 46.3-03/04 尚未落地，主窗口仍可能提前显示。
   - What's unclear: 46.6 的执行顺序是否会晚于 46.3-03/04。
   - Recommendation: 46.6 规划保持 backend-only；不要依赖 startup error page 才能实现 owned-daemon cleanup。

3. **Cmd+Q / app menu Quit 在当前 macOS 打包形态下是否全部稳定落到同一 `ExitRequested` 路径？**
   - What we know: Tauri 官方文档和当前 repo 的 tray quit 都表明 app-level exit 会经过 `RunEvent::ExitRequested/Exit`。
   - What's unclear: 本仓库在 dev/build 两种形态下是否存在平台差异。
   - Recommendation: 保留一个手工 UAT，用 `bun run tauri:dev` 分别验证 close-to-tray、tray quit、Cmd+Q 三条路径。
