# Phase 46.4 - Research

**Date:** 2026-03-21
**Phase:** 46.4
**Status:** Complete

## Research Question

What must be true to plan a CLI + daemon setup flow that reuses the existing setup business flow, stays daemon-owned, and is testable in a strict two-terminal `peerA`/`peerB` same-machine workflow?

## Executive Summary

Phase 46.4 should not implement a second setup stack inside `uc-cli`. The daemon already boots with a real `SetupOrchestrator`, while the normal CLI bootstrap path still builds a placeholder setup bundle that cannot initiate pairing. That means the structurally correct direction is:

1. add a daemon-owned setup control/query surface
2. add CLI `setup` commands as an interactive adapter over that daemon surface
3. keep pairing/session/state truth in daemon/app layers
4. validate with both targeted cargo tests and an operator-facing two-terminal walkthrough

The phase is not blocked by missing setup domain logic. The main gap is missing transport and operator-facing command flow around already-existing setup actions.

## Current State

### 1. Daemon already owns the real setup runtime

- `src-tauri/crates/uc-daemon/src/main.rs` builds the daemon with `build_non_gui_runtime_with_setup(...)`.
- `src-tauri/crates/uc-bootstrap/src/non_gui_runtime.rs` shows this path wires a real `SetupOrchestrator` through explicit `SetupAssemblyPorts`.
- `src-tauri/crates/uc-bootstrap/src/assembly.rs` shows `SetupAssemblyPorts::from_network(...)` uses the real pairing orchestrator, discovery port, and space access orchestrator.

Implication: the daemon process already has the right business objects to own setup state and setup actions.

### 2. CLI direct runtime is structurally the wrong owner for this phase

- `src-tauri/crates/uc-cli/src/main.rs` currently exposes only `status`, `devices`, `space-status`, and `clipboard`.
- `src-tauri/crates/uc-bootstrap/src/non_gui_runtime.rs` shows `build_cli_runtime(...)` uses `build_non_gui_runtime(...)`.
- `build_non_gui_runtime(...)` injects `SetupAssemblyPorts::placeholder(...)`.
- `src-tauri/crates/uc-bootstrap/src/assembly.rs` shows that placeholder setup facade cannot initiate pairing and exists only for tests/fallback call sites.

Implication: if CLI tries to drive setup by calling the normal direct bootstrap path, it will violate the intended daemon-owned topology and hit placeholder behavior. The CLI must talk to daemon-owned setup state instead of owning setup itself.

### 3. Setup business flow already exists and is granular enough for CLI

- `src-tauri/crates/uc-app/src/usecases/setup/orchestrator.rs` already exposes:
  - `new_space()`
  - `join_space()`
  - `select_device(peer_id)`
  - `submit_passphrase(...)`
  - `verify_passphrase(...)`
  - `confirm_peer_trust()`
  - `cancel_setup()`
  - `get_state()`
- `src-tauri/crates/uc-tauri/src/commands/setup.rs` already maps GUI/Tauri commands one-to-one onto those orchestrator calls.

Implication: Phase 46.4 is mostly a transport + CLI interaction problem, not a missing domain-state-machine problem.

### 4. Daemon transport already exposes most pairing primitives, but not setup orchestration

- `src-tauri/crates/uc-daemon/src/api/routes.rs` already exposes:
  - pairing discoverability and participant lease control
  - session initiation / accept / reject / cancel / verify
  - session query
- `src-tauri/crates/uc-daemon/src/api/pairing.rs` already has request/response DTOs for those pairing mutations.
- No `/setup/*` HTTP routes currently exist in daemon API.
- `src-tauri/crates/uc-cli/src/daemon_client.rs` currently supports only authenticated `GET` requests for read models.

Implication: the shortest structurally-correct path is to extend daemon HTTP with setup-specific query/mutation routes and extend `DaemonHttpClient` with authenticated mutation helpers, instead of inventing a second transport or embedding setup ownership in CLI.

### 5. Two known setup/pairing race incidents are directly relevant

- `.planning/debug/setup-state-transition-stuck.md` diagnosed the prior race where setup subscribed to pairing events after initiating pairing; the current code now subscribes first.
- `.planning/debug/pairing-verification-prompt-missing.md` diagnosed a frontend-only accept/verification race; CLI-first flow avoids that exact React race but still must preserve prompt ordering and session correlation.
- `.planning/STATE.md` records both decisions:
  - subscribe before initiate
  - synchronously establish current session before accepting verification-critical actions

Implication: CLI planning must preserve daemon/session ordering guarantees and should not reintroduce a late-subscribe or ambiguous-session UX.

## File-Level Findings

### CLI surface and output conventions

- `src-tauri/crates/uc-cli/src/commands/devices.rs` and `src-tauri/crates/uc-cli/src/commands/status.rs` show the current CLI conventions:
  - human-readable output by default
  - `--json` for machine output
  - stable exit codes
  - daemon unreachable maps to exit code `5`
- `src-tauri/crates/uc-cli/src/output.rs` provides a reusable `print_result` helper, but some older commands still print directly with `println!`/`eprintln!`.

Planning consequence: the new `setup` command family should preserve the same JSON/human split and stable exit semantics, while tolerating long-lived interactive sessions.

### Setup action semantics already encode the failure mapping we need

- `src-tauri/crates/uc-app/src/usecases/setup/action_executor.rs` maps pairing failure reasons into setup-domain errors such as:
  - `PairingRejected`
  - `NetworkTimeout`
  - `PeerUnavailable`
  - `PassphraseInvalidOrMismatch`
- The same file already waits for joiner offer and transitions from pairing into space-access join.

Planning consequence: CLI should present setup-domain states/errors, not raw daemon/pairing transport errors, whenever setup orchestration is active.

### Existing daemon tests cover many prerequisites

- `src-tauri/crates/uc-daemon/tests/pairing_api.rs` already validates daemon pairing HTTP semantics.
- `src-tauri/crates/uc-daemon/tests/pairing_host.rs` already validates daemon-side readiness, busy rules, and host behavior.
- `src-tauri/crates/uc-app/src/usecases/setup/orchestrator.rs` already contains setup orchestration tests for verification-required, rejection, passphrase retry, and completion.

Planning consequence: Phase 46.4 should add a thin new test layer around daemon setup transport and CLI orchestration, not rebuild domain coverage from scratch.

## Architectural Conclusions

### Preferred architecture

The safest architecture for 46.4 is:

1. daemon owns setup session state and setup mutations
2. daemon exposes a setup API aligned with `SetupOrchestrator`
3. CLI polls/queries that state and sends setup actions
4. CLI optionally manages local daemon probe/start convenience, but never owns setup truth

### Avoid these approaches

- Do not call `build_cli_runtime()` and use the CLI process as the setup owner.
- Do not build a second setup state machine in CLI.
- Do not bypass setup orchestration by stitching raw pairing calls into a fake setup UX.
- Do not collapse the phase into frontend/UI work; the requirement is a CLI debug entrypoint over shared flow.

## Recommended Planning Shape

### Plan slice 1: daemon setup API surface

Add daemon-owned setup query/mutation routes and DTOs that mirror the existing Tauri setup commands:

- read current setup state
- start host/new-space flow
- start join flow
- select discovered peer
- confirm inbound request
- submit joiner passphrase
- cancel/reset current setup flow

This slice should also decide whether setup state is singleton-per-profile or explicitly session-scoped. Given the single-operator/two-terminal target and existing orchestrator shape, singleton-per-profile is likely sufficient for this phase.

### Plan slice 2: CLI setup command family

Add `uniclipboard-cli setup ...` commands:

- `setup host`
- `setup join`
- `setup status`
- `setup reset`

The CLI should:

- render an identity banner with `profile`, `mode`, `device name`, and `peer ID`
- use numbered selection for discovered peers in interactive join mode
- keep host/join sessions resident while awaiting the next daemon-visible step
- hide passphrase input by default
- preserve `--json` for read/status subcommands and possibly for future automation flags

### Plan slice 3: repeatability and smoke validation

Add repeatable reset/testing support for same-machine `peerA`/`peerB` runs:

- explicit reset command or equivalent daemon-backed cleanup
- targeted automated tests for daemon setup routes and CLI happy path scaffolding
- documented two-terminal walkthrough that proves:
  - new space creation
  - discovery
  - join request
  - host confirmation
  - wrong passphrase retry
  - successful join
  - paired-devices visibility on both peers
  - rerun after reset

## Design Constraints For Planning

### Constraint 1: keep daemon alive after flow completion

The context is explicit: setup closes the host/lease, not the daemon process. Any CLI convenience for auto-starting daemon must not auto-stop it at the end.

### Constraint 2: keep CLI and GUI on the same business flow

Tauri commands already reflect the canonical setup mutations. New daemon API and CLI commands should mirror those semantics so GUI can later reuse the same daemon setup transport.

### Constraint 3: profile and mode identity must be explicit

`docs/development/config.md` and socket resolution already support profile-scoped daemon isolation. The target operator flow should standardize:

- `UC_PROFILE=peerA`, `UC_CLIPBOARD_MODE=full`
- `UC_PROFILE=peerB`, `UC_CLIPBOARD_MODE=passive`

The CLI needs to print enough runtime identity to prevent terminal/operator confusion.

### Constraint 4: no secret leakage in logs or DTOs

The repository logging rules forbid logging raw passphrases and sensitive payloads. Setup/status DTOs and CLI rendering should expose state, peer identity, and hints, but not plaintext secrets.

## Open Decisions That Planning Must Lock

1. The exact daemon `/setup/*` route set and DTO names.
2. Whether `setup host` internally creates the space immediately or offers an explicit confirmation step before doing so.
3. The exact reset semantics:
   - clear setup status only
   - clear pairing host lease only
   - clear paired device records / space membership for repeatable local debug
4. Whether CLI long-polling is sufficient for interactive status refresh or whether daemon needs a setup-specific stream/realtime route for this phase.

Recommended default: use simple request/response polling first unless lack of promptness makes host/join interaction unusable.

## Risks

### Highest risk

Trying to reuse CLI direct runtime for setup will produce a structurally wrong design because the setup facade there is placeholder-backed.

### Medium risk

If daemon setup routes expose raw low-level pairing/session primitives instead of setup-domain states, the CLI will become a second orchestrator and diverge from GUI semantics.

### Medium risk

Reset semantics may be under-specified. Without a first-class reset path, same-machine dual-profile validation will degrade into manual state surgery, which violates the phase context.

## Validation Architecture

### Test infrastructure

- Framework: Rust `cargo test`
- Workspace config: `src-tauri/Cargo.toml`
- Quick command: `cd src-tauri && cargo test -p uc-cli`
- Full command: `cd src-tauri && cargo test -p uc-daemon pairing_api -- --test-threads=1 && cargo test -p uc-daemon pairing_host -- --test-threads=1 && cargo test -p uc-app setup_flow_integration_test -- --nocapture && cargo test -p uc-cli`
- Estimated feedback budget:
  - quick: under 30s on normal local iteration
  - full: 60-180s depending on incremental cache

### What must be tested

1. daemon setup route tests
   - setup state query
   - start host / join mutations
   - select/confirm/submit/cancel transitions
   - error mapping for missing/invalid state transitions
2. CLI command tests
   - command parsing and output rendering
   - daemon unreachable / auth / invalid response exit semantics
   - interactive helper logic where testable without a TTY
3. app/setup regression coverage
   - wrong passphrase returns to passphrase input
   - join completion marks setup complete
   - verification-required ordering remains intact
4. manual smoke walkthrough
   - `peerA` host + `peerB` join on one machine
   - wrong passphrase retry
   - successful pairing + space join
   - paired-device visibility on both sides
   - rerun after reset

### Suggested validation split

- Wave 0 or first execution slice should add any missing daemon setup transport tests before large CLI interactive work.
- Each plan touching daemon API should have a cargo-test acceptance command.
- Final plan wave should include the exact operator walkthrough and the reset/retry proof.

## Planning Recommendation

Proceed with research-backed planning. The phase is well-bounded and should likely decompose into 3 plans:

1. daemon setup API + DTO + tests
2. CLI setup command family + interactive host/join/status UX
3. reset/repeatability + automated smoke coverage + operator walkthrough

## RESEARCH COMPLETE

- Daemon already owns the real setup runtime; CLI direct runtime does not.
- The primary missing piece is daemon setup transport plus CLI adapter, not domain logic.
- Existing setup/pairing race fixes must be preserved in any new CLI/daemon interaction model.
- Validation should lean on existing daemon/app tests and add a thin transport/CLI layer plus a strict two-terminal manual walkthrough.
