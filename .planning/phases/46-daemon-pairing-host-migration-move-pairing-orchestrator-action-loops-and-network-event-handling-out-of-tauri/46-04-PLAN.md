<action>
Create `src-tauri/crates/uc-app/src/usecases/setup/pairing_facade.rs` and move the setup-facing contract there so `uc-app` and `uc-bootstrap` stop depending on a trait defined in `uc-tauri`. Define these exact members in the new app-layer contract:

- `pub trait SetupPairingFacadePort: Send + Sync`
- `async fn subscribe(&self) -> anyhow::Result<tokio::sync::mpsc::Receiver<PairingDomainEvent>>`
- `async fn initiate_pairing(&self, peer_id: String) -> anyhow::Result<String>`
- `async fn accept_pairing(&self, session_id: &str) -> anyhow::Result<()>`
- `async fn reject_pairing(&self, session_id: &str) -> anyhow::Result<()>`
- `async fn cancel_pairing(&self, session_id: &str) -> anyhow::Result<()>`
- `async fn verify_pairing(&self, session_id: &str, pin_matches: bool) -> anyhow::Result<()>`

Export that contract from `src-tauri/crates/uc-app/src/usecases/setup/mod.rs` and `src-tauri/crates/uc-app/src/usecases/mod.rs`.

Update `src-tauri/crates/uc-bootstrap/src/assembly.rs` so `SetupAssemblyPorts` no longer carries `pairing_orchestrator: Arc<PairingOrchestrator>`. Replace it with `setup_pairing_facade: Arc<dyn SetupPairingFacadePort>`, update `from_network(...)` and `placeholder(...)`, and pass the facade into `build_setup_orchestrator(...)`. `placeholder(...)` should create a no-op implementation of `SetupPairingFacadePort`; do not import or instantiate `DaemonBackedSetupPairingFacade` from `uc-tauri` here.

Update `src-tauri/crates/uc-app/src/usecases/setup/action_executor.rs` and `src-tauri/crates/uc-app/src/usecases/setup/orchestrator.rs` so setup code depends only on `Arc<dyn SetupPairingFacadePort>`. Remove the concrete `Arc<PairingOrchestrator>` field/constructor argument from setup runtime state. Change all setup-triggered pairing calls and subscriptions to use the facade trait, and keep the existing setup state-machine semantics unchanged.

Do not touch `src-tauri/crates/uc-tauri/src/bootstrap/setup_pairing_bridge.rs` in this task. Adapter implementation and bridge/runtime activation remain in the next gap-closure plan so this task stays inside the app/composition boundary.
</action>
