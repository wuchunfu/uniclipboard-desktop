**WAIT** — There is a design issue. `DaemonPairingHost::run(self: Arc<Self>, cancel)` takes `Arc<Self>`. But `DaemonService::start(&self, cancel)` takes `&self`. We cannot recover the original `Arc` from `&self`.

**Resolution**: Change `DaemonPairingHost::run` signature from `pub async fn run(self: Arc<Self>, cancel: CancellationToken)` to `pub async fn run(&self, cancel: CancellationToken)`. Inside the `run` method body, the code uses `self.runtime.clone()`, `self.pairing_orchestrator.clone()`, etc. — all of these work with `&self` because they clone `Arc` fields. The internal `JoinSet::spawn` calls pass cloned Arcs of the fields, NOT `Arc::clone(&self)`. So changing from `self: Arc<Self>` to `&self` is safe because `run()` never passes `self` to spawned tasks — it passes individual field clones.
