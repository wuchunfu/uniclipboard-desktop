# Phase 56.1: Eliminate Hardcoded Strings in Pairing/Setup Flow - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract scattered hardcoded string literals (WS topic names, WS event types, pairing session state labels, pairing busy reason codes, HTTP error codes) into shared `pub const` values in `uc-core/src/network/daemon_api_strings.rs`, eliminating cross-crate duplication between `uc-daemon` (server) and `uc-daemon-client` (consumer) and reducing wire-protocol mismatch risk.

**In scope:**

- WS topic subscription name constants (Group A)
- WS event type string constants (Group B)
- Pairing session stage label constants (Group C)
- Pairing busy reason code constants (Group D)
- HTTP API error code constants (Group E)
- Value assertion unit tests for all constants
- Replacing all usage sites in `uc-daemon` and `uc-daemon-client`

**Out of scope:**

- HTTP route path constants (decision: skip — low change frequency, risk acceptable)
- Frontend TypeScript constants (different layer — camelCase vs snake_case, defer to future phase)
- New functionality or protocol changes
- Axum router path definitions

</domain>

<decisions>
## Implementation Decisions

### Module naming and structure

- **D-01:** Constants module named `daemon_api_strings.rs`, placed at `uc-core/src/network/daemon_api_strings.rs`. Re-exported via `uc-core/src/network/mod.rs`.
- **D-02:** Use flat `pub const &str` organized in `pub mod` submodules: `ws_topic`, `ws_event`, `pairing_stage`, `pairing_busy_reason`, `pairing_error_code`. No enum wrappers — strings are compared at runtime, enum adds complexity without safety benefit.
- **D-03:** Follows existing precedent of `protocol_ids.rs` in the same `network/` directory.

### Extraction scope

- **D-04:** HTTP route path constants are explicitly excluded. HTTP paths are duplicated between server and client but change infrequently — risk is acceptable without constants.
- **D-05:** Frontend TypeScript event name strings are explicitly excluded. Frontend uses camelCase (via `host_event_emitter.rs` bridge), Rust daemon uses snake_case — these are different layers that cannot share constants.
- **D-06:** All five Rust-side string groups (A through E from research) are in scope, minus HTTP paths (Group F).

### Validation strategy

- **D-07:** Write value assertion unit tests in `uc-core` for all constants (e.g., `assert_eq!(pairing_stage::REQUEST, "request")`). Prevents silent typos that cargo check cannot catch.
- **D-08:** Per-task verification: `cd src-tauri && cargo check`. Per-wave verification: `cd src-tauri && cargo test -p uc-core -p uc-daemon -p uc-daemon-client`. Phase gate: full `cd src-tauri && cargo test`.

### Claude's Discretion

- Exact ordering of constants within each submodule
- Whether to add doc comments on individual constants (recommended for non-obvious ones)
- Whether snapshot event constants (e.g., `STATUS_SNAPSHOT`) and incremental event constants go in the same submodule or separate sections

</decisions>

## Existing Code Insights

### Reusable Assets

- `protocol_ids.rs`: Existing pattern for wire-protocol identifiers in `uc-core/src/network/` — exact location and style to follow
- `ws.rs` const block: Already has organized constants (TOPIC_STATUS, PEERS_SNAPSHOT_EVENT, etc.) — migration is a direct 1:1 replacement

- `uc-core` is the shared dependency for both `uc-daemon` and `uc-daemon-client` — the only cycle-free location for shared constants
- `pub mod` submodules within a single file for related constant groups (matches Rust idiomatic style)
- `emit_ws_event()` helper in `host.rs` accepts `&str` for topic and event_type — constants slot in with zero signature changes
