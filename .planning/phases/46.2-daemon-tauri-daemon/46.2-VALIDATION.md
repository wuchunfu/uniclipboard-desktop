## Validation Architecture

### Objective

Validate that, after Phase 46.2, **all user-visible pairing flows are fully backed by the daemon-based pairing host** and that **no legacy Tauri pairing host paths remain**, while preserving behavior and UX of the existing pairing flows.

### Scope

- **In scope**:
  - P2P pairing flows in the main app UI (主动配对 + 被动接收配对请求)
  - Setup onboarding flows that use pairing (e.g. 通过 pairing 建立第一台/第二台设备的可信关系)
  - Daemon pairing host API surface (commands + WebSocket topics + error codes)
  - Tauri pairing commands and wiring into AppRuntime / DaemonWsBridge
  - Frontend integration (API 调用、events 订阅、组件行为)

- **Out of scope** (for this phase):
  - 新的 pairing 模式（例如 QR code 扫码、远程码等）
  - 非 pairing 的 daemon API（clipboard, spaces, file sync）
  - 对 pairing 协议本身的重大变更（例如 key 交换算法、传输层协议）

### Invariants

1. **Single pairing host invariant**
   - 任何 pairing host 行为（生成 code、管理会话、验证）只能在 daemon 内部执行。
   - uc-tauri 内不再创建或持有 `PairingOrchestrator` 实例，也不能直接执行业务逻辑。

2. **Stable contract invariant**
   - Tauri 命令名、事件名和主要 payload shape 对前端保持兼容（允许 payload 内部字段新增，但不能破坏现有字段的语义）。

3. **Security boundary invariant**
   - 不将 pairing verification code、fingerprint、密钥材料等敏感信息保存到通用 read models 或 RuntimeState。
   - 所有敏感信息只能存在于 daemon 内部和加密通路（例如 ws 消息中的加密 payload）。

4. **Observability invariant**
   - pairing 关键路径具有足够 tracing span / event，可用于调试和用户支持。

## Validation Dimensions

### D1. Daemon Pairing Host API Completeness

**Goal:** daemon 提供的 pairing API 足以覆盖原 Tauri pairing host 功能，没有行为缺口。

**Checks:**

- 对照 46 CONTEXT 中的原 Tauri pairing host 能力列表（发起、接受、拒绝、取消、超时、verify 等），确保 daemon pairing host 均有对应 API。
- 每个 API 都有明确的错误码/状态（包括 busy、session 冲突、无本地参与者等）。

**Suggested Tests:**

- uc-daemon crate 单元/集成测试，针对 `pairing::host` 模块：
  - 成功发起/接受/拒绝/取消/超时/verify。
  - 并发请求 / 已有 session 时的错误行为。

### D2. Tauri Pairing Bridge Correctness

**Goal:** Tauri pairing 命令全部变成“纯桥接”：

- 接收来自前端的命令
- 调用 daemon pairing API
- 将结果转换为已有的 Tauri 响应和事件

**Checks:**

- `uc-tauri/src/commands/pairing.rs` 中不再出现 `PairingOrchestrator` 的直接使用。
- 所有 pairing 命令通过新的 `DaemonPairingClient`（命名可调整，但语义一致）调用 daemon。
- 错误与状态在命令返回值和事件中正确反映（例如 busy → 前端能收到“当前正在配对”的错误状态）。

**Suggested Tests:**

- uc-tauri crate 集成测试或模拟测试：
  - 使用 mock daemon pairing client，验证 Tauri pairing commands 对应的行为（输入输出）。
  - 增加 code-search 守卫，确认 `src-tauri/crates/uc-tauri/src/bootstrap/wiring.rs`、`src-tauri/src/main.rs` 中不再残留 `PairingOrchestrator`、`run_pairing_event_loop`、`pairing_events_subscribe_backoff_ms`。

### D3. Realtime Event Flow

**Goal:** pairing 相关 realtime 事件只从 daemon → DaemonWsBridge → Tauri → 前端，且事件内容满足前端需求。

**Checks:**

- `DaemonWsBridge` 对 pairing 主题的订阅和事件转发完整、无重复。
- 删除任何 legacy pairing event loop 或直接 libp2p 事件桥接。
- 前端组件只依赖 daemon ws 事件，不再依赖 legacy Tauri 封装事件。

**Suggested Tests:**

- uc-tauri `daemon_ws_bridge` 相关 tests 补充 pairing 事件用例：
  - bridge 收到 daemon pairing 事件 → 向前端发出正确 Tauri 事件。

### D4. Frontend Behavior Parity

**Goal:** 从用户视角看，pairing 体验不发生负面变化（成功/失败路径表现一致）。

**Checks:**

- 主动发起 pairing：
  - 点击“配对”→ 输入 code → 等待中 UI → 成功/失败状态。
- 被动接受 pairing：
  - 收到 pairing 请求通知 → 接受/拒绝 → 状态更新。
- Setup 旅程：
  - 使用 pairing 完成新设备首次连接。

**Suggested Tests:**

- 前端集成测试（例如 Playwright / Vitest + mock Tauri），对上述 flows 做回归。
  - 优先复用现有 Vitest 文件：`src/components/__tests__/PairingDialog.test.tsx`、`src/components/__tests__/PairingNotificationProvider.realtime.test.tsx`、`src/pages/__tests__/setup-peer-discovery-polling.test.tsx`。

### D5. Security Boundaries

**Goal:** pairing 敏感数据不向不该出现的层泄漏。

**Checks:**

- `RuntimeState`、session read models 中不含 pairing code / fingerprint / keyslots 原始值。
- 新增 DTO 仅携带必要的非敏感信息，如 device id、display name、pairing 状态等。
- 对敏感字段的序列化/反序列化路径有针对性测试。

### D6. Error Handling & UX

**Goal:** 常见错误有清晰 UX，且与 daemon 错误码对应。

**Checks:**

- daemon 对以下错误有明确分类：
  - 无法连接本地 daemon
  - pairing host busy
  - session not found / expired
  - invalid pairing code / fingerprint mismatch
- Tauri bridging 对应这些错误，将其转换为前端可理解的状态码/消息。
- 前端对这些状态有 UI 文案（英文/中文）和反馈（toast/message/对话框）。

### D7. Observability

**Goal:** pairing 关键路径有可追踪的日志和 span。

**Checks:**

- daemon pairing host 对关键操作（start, accept, reject, verify, timeout）均有 tracing span + 事件。
- uc-tauri pairing 命令和 DaemonWsBridge 对关键步骤有 span。
- 前端有合理的 logging（仅在调试模式）。

## Concrete Validation Tasks

> 这些任务不会在这里展开为完整 PLAN 任务，但会作为 Planner 的参考，用于编写 46.2-PLAN 中的 `<acceptance_criteria>`。

1. **Code 搜索守卫**
   - `uc-tauri` 内搜索 `PairingOrchestrator`，确认 46.2 完成后无引用。
   - 搜索 legacy pairing event loops / wiring，确保被清理。
   - 至少包含：`rg "run_pairing_event_loop|pairing_events_subscribe_backoff_ms|PairingOrchestrator" src-tauri/crates/uc-tauri src-tauri/src/main.rs -n` 无结果。

2. **API 对齐验证**
   - 列出所有 daemon pairing API 路径和方法，映射到原 Tauri host 能力列表。
   - 为每个映射编写最少一个 happy path + 一个失败 path 测试。

3. **事件流验证**
   - 在集成环境中记录 daemon pairing 事件序列和前端收到的事件序列，对比是否一一对应。

4. **黑盒行为回归**
   - 从 46 CONTEXT 中抽取旧行为描述，编写“行为对照表”（旧实现 / 新实现），做一次黑盒测试回归。
   - 至少覆盖主动发起配对、被动接受配对、以及 Setup 中由 pairing 驱动的 peer discovery / state progression。

5. **安全边界测试**
   - 针对可能泄漏敏感字段的 DTO/struct 编写 serde 单元测试，确保序列化输出中不含敏感字段。

6. **错误 UX 检查**
   - 模拟“daemon 未运行”、“busy”、“session not found / expired”、“code 错误”等场景，检查 UI 提示是否合理、是否有本地化文本。
