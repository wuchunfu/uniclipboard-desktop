    In `write_file_to_clipboard_after_transfer` (line ~2025), add a race detection check between the entry persistence block (lines 2045-2072) and the clipboard write block (lines 2074-2094).

    The logic: Before setting RemotePush origin and writing to clipboard, call `consume_origin_or_default(LocalCapture)`. If the returned origin is NOT `LocalCapture`, it means an origin was previously set (e.g., by a user copy or another programmatic write) and hasn't been consumed yet — indicating clipboard activity during the transfer. In that case, skip the clipboard write but keep the persisted entry.

    Specific changes to `write_file_to_clipboard_after_transfer`:

    1. After the entry persistence block (after line 2072), add race detection:
    ```rust
    // FCLIP-03: Check for clipboard race before writing
    // If an origin is already set (not the default LocalCapture),
    // the user or another operation touched the clipboard during transfer.
    let current_origin = clipboard_change_origin
        .consume_origin_or_default(uc_core::ClipboardChangeOrigin::LocalCapture)
        .await;
    if current_origin != uc_core::ClipboardChangeOrigin::LocalCapture {
        info!(
            origin = ?current_origin,
            file_count = file_paths.len(),
            "Clipboard race detected: skipping auto-write (user copied during transfer). Files available in Dashboard."
        );
        return;
    }
    ```

    2. Keep the existing `set_next_origin(RemotePush)` and `write_snapshot` blocks after the race check (lines 2074-2094 unchanged).

    3. Fix the doc comment on line 2023-2024 to accurately describe the behavior:
    ```rust
    /// Write file paths to system clipboard after transfer completes.
    /// Persists clipboard entry first, then checks for clipboard race (FCLIP-03):

    ```
