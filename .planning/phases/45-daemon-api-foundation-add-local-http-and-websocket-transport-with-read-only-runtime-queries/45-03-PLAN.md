# Plan 45-03: CLI Migration And Tauri Daemon Bootstrap

## Goal

Adopt the new daemon API boundary from both client surfaces covered by Phase 45: migrate CLI commands that now have daemon endpoints, and teach the Tauri shell to obtain daemon connection info and inject it at runtime without cutting over existing frontend business flows.

<action>
Update `src-tauri/crates/uc-cli/Cargo.toml` to add the client transport dependencies needed for the daemon HTTP API. Use one shared async HTTP client module rather than embedding request logic in each command. Prefer `reqwest` for the client side so command code stays thin and tests can validate status codes cleanly.

Create `src-tauri/crates/uc-cli/src/daemon_client.rs` with concrete helpers for:

- loading daemon connection info or reading the daemon auth token from the same local source as the daemon contract expects
- building authenticated requests with `Authorization: Bearer <token>`
- `get_status()`
- `get_paired_devices()`

Give the module one explicit entrypoint type and method surface, for example:

- `pub struct DaemonHttpClient`
- `pub fn new() -> anyhow::Result<Self>`
- `pub async fn get_status(&self) -> anyhow::Result<uc_daemon::api::types::StatusResponse>`
- `pub async fn get_paired_devices(&self) -> anyhow::Result<Vec<uc_daemon::api::types::PairedDeviceDto>>`

The client should resolve:

- daemon base URL from the same loopback address helper used by the daemon server
- auth token from the daemon token path helper introduced in Plan 45-01

Do not duplicate token filename or base URL construction logic inside `uc-cli`.

Refactor `src-tauri/crates/uc-cli/src/commands/status.rs` to call `/status` through the shared daemon HTTP client instead of speaking Unix socket JSON-RPC directly.

The human-readable output should stay functionally equivalent to current behavior:

- `Status: running`
- `Uptime: <formatted>`
- `Version: <version>`
- `Workers: X/Y healthy`
- `Connected peers: <n>`

Refactor `src-tauri/crates/uc-cli/src/commands/devices.rs` to call `/paired-devices` through the shared daemon HTTP client instead of direct bootstrap. Preserve the current output style as closely as possible.

For `devices`, preserve the current visible lines:

- `Paired devices: <count>`
- one line per device as `  <name> (id: <peer_id>)`

Move any now-unused direct-mode comment strings in `main.rs` and the command docs so the command descriptions no longer claim `devices` is direct mode for this phase. Do not change `space-status` or `clipboard` commands in this plan.

Keep error semantics stable:

- exit `0` on success
- exit `1` on general failure
- exit `5` when daemon is unreachable

Map HTTP-layer failures concretely:

- connection refused / timeout -> `EXIT_DAEMON_UNREACHABLE`
- `401` -> `EXIT_ERROR` with an auth-specific error message
- non-2xx unexpected status -> `EXIT_ERROR`
- JSON decode failure -> `EXIT_ERROR`

Do not remove direct-mode bootstrap helpers from `uc-bootstrap`; just stop using them for the commands covered by Phase 45.

Add targeted CLI tests for:

- status command returns exit code `5` when daemon HTTP endpoint is unreachable
- status command prints human output from a daemon HTTP response fixture
- devices command prints paired devices from a daemon HTTP response fixture
- `--json` output still serializes the fetched daemon DTOs

<action>
Update the Tauri host so it can act as daemon bootstrap/client shell without becoming a second business host.

The implementation should add a narrow runtime bootstrap path that:

- probes whether the daemon is already running
- starts the daemon if necessary or reports a clear startup failure
- obtains `DaemonConnectionInfo`
- injects connection info into the webview at runtime only

The injected data must include enough for the webview to connect later:

- loopback base URL
- loopback WebSocket URL
- bearer token

Use one explicit runtime state holder for injected daemon connection data, for example in `src-tauri/crates/uc-tauri/src/bootstrap/runtime.rs`:

- `pub struct DaemonConnectionState(Arc<RwLock<Option<DaemonConnectionInfo>>>)`

Register it with `.manage(...)` in `src-tauri/src/main.rs` before startup work begins. This gives Tauri setup/startup code one in-memory place to store the connection info without pushing it into frontend storage.

For daemon lifecycle integration, implement these concrete steps:

1. Probe the daemon HTTP `/health` endpoint on the loopback address.
2. If healthy, load token + connection info into `DaemonConnectionState`.
3. If not healthy, start `uniclipboard-daemon` as a child process from Tauri startup code.
4. Poll `/health` with a bounded timeout budget until the daemon is reachable or startup fails.
5. After success, emit or inject the connection info to the main webview.

Use a dedicated helper path rather than scattering child-process startup logic across `main.rs`. `src-tauri/crates/uc-tauri/src/bootstrap/run.rs` is the preferred home if it becomes the shell-runtime bootstrap helper for this phase.

Security rules for the injection path:

- no token in query strings
- no token written to `localStorage`
- no token persisted to disk by frontend code
- no raw token logging in Tauri or daemon startup paths

Use one explicit runtime-only delivery mechanism:

- emit a Tauri event `daemon://connection-info` to the `main` webview window after the page is ready

Do not implement a JS string injection fallback in this phase. The connection info must be delivered by this single event-based path and must not be written to disk or browser storage.

Keep existing Tauri commands and event flows intact. Phase 45 is not allowed to remove current frontend-to-Tauri business interactions yet. The Tauri work here is only daemon lifecycle integration plus connection info injection.

Explicit non-goals for this task:

- do not remove `invoke_handler(...)`
- do not remove existing pairing / clipboard / settings Tauri commands
- do not switch existing frontend API calls to daemon HTTP in this phase

Add smoke coverage or targeted tests where feasible, and document the remaining manual checks around storage persistence and devtools inspection. The tests should cover:

- daemon probe helper returns success on healthy `/health`
- startup helper treats timeout or spawn failure as startup error
- connection state stores `DaemonConnectionInfo` in memory
- emitted event payload shape contains `baseUrl`, `wsUrl`, and `token`
