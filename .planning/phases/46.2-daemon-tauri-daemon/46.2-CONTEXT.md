# Phase 46.2: Daemon Pairing Flow Hard Cutover - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Complete the migration so that **all pairing business logic and session lifecycle run on the daemon**, not inside `uc-tauri`, while **preserving the existing user-facing pairing flow**.

This phase is an urgent follow-up to Phases 46 and 46.1. It:

- Removes remaining Tauri-owned pairing flows and ad-hoc host logic so `uc-tauri` becomes a **thin daemon client + realtime bridge** for pairing
- Ensures every runtime path that performs pairing (GUI now; future CLI later) goes through the daemon-hosted pairing surface
- Keeps the visible pairing flow semantics (request → verification → verifying → complete/failed) unchanged for users

It does **not** introduce new pairing capabilities or major UI changes — those belong to later phases (47/48).

</domain>

<decisions>
## Implementation Decisions

### Tauri pairing entrypoint cleanup and bridge shape

- Phase 46.2 aims for **"minimal bridge only"**:
  - `uc-tauri` must not own any pairing state machine, session lifecycle, or retry/timeout policy
  - All pairing business decisions (who can pair, concurrency, timeouts, recovery) live only in `uc-app` + `uc-daemon`
- Tauri pairing entrypoints are reduced to:
  - Translating GUI user intent into daemon API calls (HTTP/JSON-RPC or dedicated pairing client)
  - Forwarding daemon-driven pairing/realtime events into the existing frontend event contract
  - Keeping at most thin UI-facing flags for local view state (e.g., "pairing dialog currently open")
- After 46.2 there must be **no alternative Tauri-owned pairing host path**:
  - Any legacy pairing loops, orchestrators, or direct libp2p flows under `src-tauri/crates/uc-tauri/` that bypass the daemon must be removed or converted into daemon-client shims
  - `commands/pairing.rs`, `bootstrap/setup_pairing_bridge.rs`, pairing-related code in `bootstrap/wiring.rs`, and `src-tauri/src/main.rs` background startup must either delegate to daemon APIs or be deleted
- CLI and future non-GUI flows that perform pairing are expected to **reuse the same daemon-hosted pairing contracts**, not introduce new non-daemon hosts.

### User pairing experience invariants

- **Visible flow stages remain the same:** GUI pairing keeps the same high-level stages as in Phase 46/46.1:
  - request → verification → verifying → complete / failed
  - The number and ordering of user-visible steps should not change
- **Mental model stays intact:**
  - Users still "start pairing", see a request/verification step, confirm the code/fingerprint, and observe completion/failure
  - The way users think about "how to pair two devices" must not materially change in this phase
- **Allowed differences:**
  - Micro-level changes in timing (e.g., when a spinner appears), toast order, or exact error phrasing are acceptable
  - Minor improvements in loading feedback or transient edge-case error handling are allowed as long as the overall flow shape and states are unchanged
- **Not allowed in this phase:**
  - Adding new user-visible pairing modes or branches
  - Introducing new steps that change the perceived workflow (e.g., extra confirmation screens beyond what exists today)

### Daemon vs frontend responsibilities

- **Business decisions on daemon/app side only:**
  - Whether a host is discoverable
  - Whether a new pairing session is allowed (concurrency rules, active session checks)
  - Session lifecycle (created → in progress → terminal, timeouts, retries)
  - Admission rules for headless/CLI vs GUI-hosted pairing
- **Frontend/Tauri keeps only UI gating responsibilities:**
  - Local UI state such as "pairing dialog open" or "current view is in pairing mode" may be tracked in frontend/Tauri for UX purposes
  - These flags must **not** form a second, divergent pairing state machine; they must be driven by or reconciled against daemon-truth when it matters
- **Entering/exiting pairing mode:**
  - GUI actions that enter or exit pairing mode must be reflected to the daemon via the existing app-layer facades (`SetupPairingFacadePort` / related use cases), not handled purely in Tauri
  - A host should not become pairing-available purely because the GUI flipped a local flag; daemon must own readiness/discoverability semantics
- **Single source of truth:**
  - When there is a disagreement between UI-local flags and daemon state, daemon state wins; planners should design APIs so the frontend can reconcile rather than guess

### Risk posture and rollback strategy

- **Hard cutover, no runtime rollback switch:**
  - Phase 46/46.1 already performed major architectural switches; 46.2 continues this pattern
  - After 46.2 lands, there is no feature flag or configuration knob to route pairing back through old Tauri-owned hosts
  - If a severe regression is found, recovery uses normal git workflows (fix-forward or revert), not an in-app toggle
- **Quality expectations for this hard cut:**
  - At minimum, the existing Phase 46 and 46.1 regression suites must pass, including:
    - Daemon pairing host and API tests
    - Daemon websocket bridge and realtime regression tests
    - Setup/pairing integration tests that exercise the end-to-end flow
  - Frontend pairing UX must be validated manually against the current stable flow to confirm that observable stages and key error/retry paths behave as before
- **Scope discipline:**
  - New pairing UX improvements, new error flows, or recovery UIs that go beyond the current behavior are explicitly out of scope and should be deferred to later phases

### Claude's Discretion

- Exact internal naming and decomposition of daemon pairing client helpers in `uc-tauri` / `uc-cli`, as long as they remain thin bridges
- Detailed structure of daemon pairing DTOs and client interfaces, provided they respect the sensitivity and metadata boundaries from Phase 45/46
- Specific error-code taxonomy used by daemon pairing endpoints for busy/active-session/no-local-participant scenarios
- How UI gating flags are wired to daemon state (e.g., subscriptions vs on-demand queries) as long as no second business state machine is introduced in Tauri/frontend

</decisions>

<specifics>
## Specific Ideas

- Phase description (original intent): **"彻底打通基于 daemon 的配对流程, 完全移除原 tauri 中相关的配对流程. 期望: 在不改变用户配对流程的情况下, 内部替换成基于 daemon 的配对流程实现"**
- The end result should feel like a **fully daemon-native pairing flow** with Tauri acting only as shell and compatibility bridge — users should not be able to tell that the internal host changed again in this phase
- This phase should remove the last places where Tauri can "own" pairing behavior; any future pairing work should naturally extend the daemon surface rather than adding new Tauri paths

<code_context>

## Existing Code Insights

### Reusable Assets

- **Daemon pairing host and metadata snapshots**: `DaemonPairingHost` and `DaemonPairingSessionSnapshot` in `uc-daemon` already centralize pairing session ownership and read models
- **App-layer pairing orchestrator**: `PairingOrchestrator` in `uc-app` exposes a Tauri-free mutation + event surface suitable for both daemon and non-GUI callers
- **Unified realtime bridge**: `DaemonWsBridge` and the realtime runtime wiring from Phase 46.1 provide a single authenticated websocket connection and topic dispatch mechanism
- **Setup pairing facade**: `SetupPairingFacadePort` and `SetupPairingEventHub` (Phase 46/46.1) already route setup and pairing events through shared app-layer facades

### Established Patterns

- Long-lived business ownership (pairing, workers, runtime state) belongs in **non-Tauri** crates (`uc-daemon`, `uc-app`, `uc-bootstrap`), with Tauri restricted to shell/bridge responsibilities
- Read models that may be exposed to frontend clients stay **metadata-only**; verification secrets and sensitive artifacts travel only over authenticated realtime channels
- Snapshot-first + incremental-update transport patterns are preferred over polling or full-table refresh on every change
- Runtime modes (GUI, CLI, daemon) should share app-layer use cases and ports rather than duplicating orchestration logic per entrypoint

### Integration Points

- Tauri pairing commands must shrink to daemon-client adapter calls that:
  - invoke the daemon pairing mutation surface, and
  - forward daemon pairing/realtime events through the unified realtime envelope into the existing frontend contract
- Any remaining pairing-related wiring in `bootstrap/wiring.rs`, `setup_pairing_bridge.rs`, or `main.rs` must be revisited so that **daemon + app** form the only host for pairing behavior
- Future CLI pairing commands should plug into the same daemon API and read models rather than reconstructing their own pairing flows

</code_context>
