# Plan 44-01: CLI Settings Commands — list, get, set

## Goal

Add `settings list`, `settings get <key>`, and `settings set <key> <value>` subcommands to the CLI. Uses dot-notation key paths (e.g., `general.auto_start`) and reuses existing `GetSettings`/`UpdateSettings` use cases. Follows the established `devices.rs` direct-bootstrap pattern.

<title>Create settings.rs command module with list/get/set subcommands</title>
<read_first>

</read_first>
<action>
Create `src-tauri/crates/uc-cli/src/commands/settings.rs` with:

1. A `SettingsSubcommand` enum (clap Subcommand derive) with three variants:
   - `List` — no args
   - `Get { key: String }` — positional arg for dot-notation key path
   - `Set { key: String, value: String }` — positional args for key and value

2. `pub async fn run(subcmd: SettingsSubcommand, json: bool, verbose: bool) -> i32` function that:
   - Builds CLI context via `uc_bootstrap::build_cli_context_with_profile(profile)`
   - Gets storage paths via `uc_bootstrap::get_storage_paths(&ctx.config)`
   - Builds runtime via `uc_bootstrap::build_non_gui_runtime(ctx.deps, storage_paths)`
   - Creates `uc_app::usecases::CoreUseCases::new(&runtime)`
   - Dispatches to `run_list`, `run_get`, or `run_set` based on subcmd

3. `run_list` function:
   - Calls `usecases.get_settings().execute().await`
   - Serializes Settings to `serde_json::Value` via `serde_json::to_value(&settings)`
   - If `json` flag: prints full JSON via `output::print_result`
   - If human mode: flattens the JSON Value into dot-notation key=value pairs using a recursive helper `flatten_json(prefix: &str, value: &serde_json::Value, out: &mut Vec<(String, String)>)` that:
     - For Object: recurses with `prefix.key` (or just `key` if prefix empty)
     - For Array: recurses with `prefix[i]`
     - For other values: pushes `(prefix, value.to_string())` stripping quotes from strings
   - Prints each pair as `{key} = {value}` one per line

4. `run_get` function:
   - Calls `usecases.get_settings().execute().await`
   - Converts to `serde_json::Value`
   - Converts dot-notation key to JSON Pointer by replacing `.` with `/` and prepending `/`
     e.g., `general.auto_start` → `/general/auto_start`
   - Uses `value.pointer(&pointer_path)` to extract the value
   - If not found: `eprintln!("Error: unknown key '{key}'")` and return EXIT_ERROR
   - If json: prints the sub-value as JSON
   - If human: prints the value (strip quotes from strings)

5. `run_set` function:
   - Calls `usecases.get_settings().execute().await` to load current settings
   - Converts to `serde_json::Value` via `serde_json::to_value(&settings)`
   - Converts dot key to JSON Pointer path
   - Uses `value.pointer(&pointer_path)` to check key exists; if not: error
   - Parses the new value string to match the existing type:
     - If existing is bool: parse "true"/"false"
     - If existing is number (u64): parse as u64
     - If existing is number (f64): parse as f64
     - If existing is string: use as-is
     - If existing is null: use as string
     - Otherwise: error "cannot set complex value"
   - Uses `value.pointer_mut(&pointer_path)` to replace the value
   - Deserializes back to `Settings` via `serde_json::from_value(value)` (validates schema)
   - Calls `usecases.update_settings().execute(new_settings).await`
   - Prints confirmation: `"{key}" set to {new_value}`

6. Output types:
   - `SettingsListOutput { settings: serde_json::Value }` implementing Serialize + Display (Display uses flatten helper)
   - `SettingsGetOutput { key: String, value: serde_json::Value }` implementing Serialize + Display
   - `SettingsSetOutput { key: String, value: serde_json::Value }` implementing Serialize + Display

All error handling uses explicit `match` with `eprintln!` — no `unwrap()` or `expect()`.
</action>

## must_haves

- `settings list` prints all settings as flat dot-notation key=value pairs (human) or full JSON (--json)
- `settings get <key>` retrieves a single setting by dot-notation path
- `settings set <key> <value>` updates a single setting via read-modify-write cycle using UpdateSettings use case
- All three subcommands follow the devices.rs bootstrap pattern (build_cli_context_with_profile → get_storage_paths → build_non_gui_runtime)
- Error handling uses explicit match with eprintln!, no unwrap()/expect()
- Supports --json and --verbose global flags
