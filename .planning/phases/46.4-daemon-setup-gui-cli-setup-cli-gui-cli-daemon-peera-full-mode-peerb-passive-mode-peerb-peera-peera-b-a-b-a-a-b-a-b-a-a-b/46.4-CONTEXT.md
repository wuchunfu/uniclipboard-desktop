# Phase 46.4: CLI + daemon setup flow for dual-peer validation - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a CLI entrypoint for the existing daemon-backed setup flow so the team can debug and validate the end-to-end setup path without depending on the GUI path being healthy first.

This phase is specifically about:

- exposing the existing `SetupOrchestrator` / daemon pairing flow through CLI commands
- making the CLI usable in a strict two-terminal same-machine workflow:
  - `peerA` in `full` mode
  - `peerB` in `passive` mode
- validating the full setup happy path plus a minimal retry path

This phase is **not** a new pairing architecture, not a new setup state machine, and not a GUI redesign. CLI and GUI must continue to target the same business flow and daemon-owned pairing host.

</domain>

<decisions>
## Implementation Decisions

### Setup command surface

- Use a single `uniclipboard-cli setup ...` command family as the public entrypoint.
- `peerA` should use a long-lived `setup host` style command that creates the encrypted space, enters pairing/setup host mode, and stays attached while waiting for inbound join requests and confirmations.
- `peerB` should use a guided `setup join` style command that drives the join flow from discovery through device selection, request submission, confirmation, and passphrase entry.
- Provide a `setup status` command so operators can inspect the current setup state, `sessionId`, peer context, and next-step hint without reading raw daemon logs.
- Do not split this phase into low-level `pairing` and `space` command families for normal operation. The CLI should reflect the higher-level setup experience.

### Interaction model

- Default UX is interactive-first for human debugging.
- `setup join` should present discovered peers as a numbered list rather than requiring raw `peerId` by default.
- `setup host` should allow the local operator to accept or reject the inbound request directly inside the long-lived command session.
- Passphrase input on the joiner side should be hidden by default.
- Non-interactive overrides may exist for later automation, but they are secondary to the guided human flow in this phase.
- Failure should keep the command resident when practical so the operator can retry from the current session context instead of always restarting from scratch.

### Daemon and peer runtime topology

- The daemon remains an independent long-lived process; CLI is only a client/operator surface over that daemon.
- CLI should probe the local profile-scoped daemon and is allowed to help start it when missing, because the required operator experience is strict two-terminal validation rather than multi-terminal manual orchestration.
- Pairing/setup host availability is explicit and temporary:
  - CLI enters host mode by asking the daemon to open the setup/pairing host surface
  - on completion or cancellation, CLI closes the host/lease
  - CLI must **not** stop the daemon process at the end of a run
- Same-machine debugging should standardize on fixed profile names:
  - `UC_PROFILE=peerA`
  - `UC_PROFILE=peerB`
- Runtime mode should continue to use the existing environment contract:
  - `UC_CLIPBOARD_MODE=full` for `peerA`
  - `UC_CLIPBOARD_MODE=passive` for `peerB`
- The required operator layout is exactly two terminals, one per peer. The phase should not require separate dedicated daemon terminals for the supported validation path.
- Commands should print an identity banner at startup with enough context to disambiguate the two terminals, at minimum:
  - profile
  - mode
  - device name
  - peer ID

### Reset and repeatability

- This phase should include an explicit reset path suitable for repeated same-machine setup debugging.
- Reset is important enough to be part of the phase deliverable, not merely an undocumented manual cleanup trick.
- The reset path should make it feasible to rerun the `peerA` / `peerB` validation loop multiple times on the same machine without hand-editing state.

### Acceptance and validation contract

- Completion requires both:
  - a human-runnable two-terminal end-to-end walkthrough
  - an automated smoke layer that exercises the CLI/daemon surface at a repeatable level
- The required happy path is:
  - `peerA` starts host mode and creates a new encrypted space
  - `peerB` discovers `peerA`
  - `peerB` requests to join `peerA`
  - `peerA` confirms
  - `peerB` enters the encryption passphrase
  - `peerA` verifies the passphrase flow successfully
  - `peerB` completes joining `peerA`
- Successful terminal-state validation must include:
  - both peers show each other in paired-device queries
  - `peerB` shows joined/encryption-ready space status, not merely pairing success
- At least one negative path must be covered in this phase:
  - wrong passphrase followed by retry is the preferred failure case
- Acceptance also requires proving that after reset, the same `peerA` / `peerB` flow can be run again successfully.

### Claude's Discretion

- Exact subcommand names and flag names under `setup`, as long as the command family remains unified and the user-facing semantics above hold.
- Exact output formatting for the identity banner and status view.
- Exact mechanism used by CLI to probe/start the daemon, as long as the supported operator workflow remains two-terminal and deterministic.
- Exact shape of the automated smoke layer, as long as it validates the locked acceptance contract above.

</decisions>

<specifics>
## Specific Ideas

- The user explicitly wants CLI and GUI to remain two different entrypoints into the **same** underlying setup flow, not two separate implementations.
- The primary motivation is speed of development and debugging while the GUI setup path is not yet healthy.
- The preferred same-machine validation story is:
  - Terminal A: `peerA`, `full` mode
  - Terminal B: `peerB`, `passive` mode
- The user explicitly clarified a critical runtime rule:
  - daemon is independently running
  - CLI interacts with daemon to open/close the setup/pairing host
  - successful pairing/setup closes the host/lease, not the daemon process itself

</specifics>

<code_context>

## Existing Code Insights

### Reusable Assets

- `SetupOrchestrator` already exposes the setup actions CLI needs to drive: start host/new space, start join flow, choose peer, confirm trust, submit passphrase, cancel, and inspect state.
- `SetupActionExecutor` already contains the pairing/session and join-space continuation logic; the CLI should drive this orchestrator instead of reimplementing the flow manually.
- `DaemonHttpClient` already resolves profile-aware daemon URL/token and provides a clear pattern for daemon-backed CLI commands.
- Existing CLI commands already define the output convention:
  - human-readable by default
  - pretty JSON behind `--json`
- Daemon HTTP routes already include pairing/session mutation endpoints plus discoverability/participant lease controls, so the phase can likely compose on top of existing daemon transport rather than inventing a second control path.

### Established Patterns

- Long-lived business ownership belongs to daemon/app layers, not to entrypoints.
- Runtime-mode separation is already a project-level non-negotiable, so CLI must stay an adapter over shared app/daemon logic.
- Profile-aware isolation is already documented and implemented; the phase should lean into that instead of inventing ad hoc same-machine test names.
- CLI direct-mode commands and daemon-backed commands already coexist in `uc-cli`; this phase should preserve that style rather than collapsing everything into one transport model.

### Integration Points

- New `setup` commands will land naturally under `src-tauri/crates/uc-cli/src/commands/`.
- CLI needs a coherent bridge between:
  - local daemon lifecycle/probe/start behavior
  - daemon pairing host lease controls
  - setup state inspection and guided operator prompts
- Planner should pay particular attention to how `setup host` and `setup join` remain interactive while still reflecting daemon-truth and setup-orchestrator state changes.
- Validation will need a reliable way to bind peer identity and profile-scoped resources so `peerA` and `peerB` do not drift or collide on one machine.

</code_context>

## Deferred Ideas

- Rich TUI/terminal UI beyond what is needed for guided debugging
- Additional negative-path coverage beyond the minimum wrong-passphrase retry contract
- Broad new CLI product surface outside setup debugging, such as full pairing administration or operator-only inspection tools
- Any GUI fix/workaround folded into this phase instead of keeping CLI as the urgent alternate entrypoint

</deferred>
