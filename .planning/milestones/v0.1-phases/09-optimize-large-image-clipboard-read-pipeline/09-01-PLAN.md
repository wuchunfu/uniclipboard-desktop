<objective>
Optimize the macOS clipboard image capture pipeline to eliminate the 3-second TIFF-to-PNG transcode blocking the watcher thread, deduplicate known TIFF aliases in the raw fallback loop, and reduce peak memory from ~71MB to ~34MB. The background blob worker is updated to convert TIFF to PNG before writing to blob store, ensuring downstream consumers (dashboard, sync) receive PNG.

Purpose: Image clipboard captures currently block the watcher thread for ~3s and allocate ~71MB. This makes clipboard sync feel sluggish for image-heavy workflows.
Output: Optimized read_snapshot in common.rs, TIFF-to-PNG conversion in background_blob_worker.rs
</objective>

       ```rust
       fn should_convert_to_png(mime: Option<&MimeType>) -> bool {
           match mime {
               Some(m) => {
                   let s = m.as_str();
                   s.starts_with("image/") && s != "image/png" && s != "image/webp"
               }
               None => false,
           }
       }

       fn convert_image_to_png(image_bytes: &[u8]) -> Result<Vec<u8>> {
           let image = image::load_from_memory(image_bytes)
               .context("decode image for PNG conversion")?;
           let mut png_bytes = Vec::new();
           image.write_to(
               &mut std::io::Cursor::new(&mut png_bytes),
               image::ImageFormat::Png,
           ).context("encode image as PNG")?;
           Ok(png_bytes)
       }
       ```

    2. In `process_once()`, after retrieving `raw_bytes` from cache/spool (around line 192), check if format conversion is needed:
       - Read the representation's MIME type from the repo (already fetched later for thumbnail — move the repo read earlier or add one)
       - If `should_convert_to_png(mime)`, convert `raw_bytes` to PNG via `convert_image_to_png()`
       - Use the converted bytes for blob_writer.write_if_absent and thumbnail generation
       - Also update the representation's MIME type in the DB to `image/png` after successful conversion via the existing `update_processing_result` path (or a new repo method if needed)
       - Log the conversion at debug level with original size and converted size

    3. Important: The `content_hash` should be computed on the CONVERTED bytes (PNG), not the original TIFF, because the blob store content should be addressable by what's actually stored.

    4. For thumbnail generation, pass the converted PNG bytes (not original TIFF) — this avoids double-decode since the thumbnail generator would otherwise decode TIFF again.

    5. Add `image` crate to uc-infra dependencies if not already present (check Cargo.toml).
