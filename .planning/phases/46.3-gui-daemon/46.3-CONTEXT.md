# Phase 46.3: 修复 GUI 启动 daemon 的生命周期托管与版本不匹配静默替换 - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

修复桌面 GUI 作为 daemon 启动入口时的进程托管与兼容判定，使其能够在本机预期端点上识别不兼容的旧 daemon，按受控策略静默替换，并在失败时进入确定的启动错误态。

本阶段聚焦于：

- GUI 启动时对本地 daemon 的兼容判定
- GUI 对预期本地 daemon 端点的生命周期接管与替换策略
- daemon 不可用/不兼容时的启动错误体验与恢复路径

本阶段不包括：

- 扩大 daemon 远程/跨机控制能力
- 引入新的 daemon 业务能力或新 API 域
- 把 Phase 47 的前端直连 daemon cutover 提前并入
- 为旧 daemon 保留长期兼容或回退开关

</domain>

<decisions>
## Implementation Decisions

### 兼容判定契约

- GUI 与 daemon 视为同一发布单元，当前 GUI 打包版本与运行中 daemon 版本只要不一致，就判定为不兼容。
- 版本号一致也不代表可接受；如果运行中 daemon 缺少 GUI 依赖的关键能力、关键路由或关键契约，仍然判定为不兼容。
- `/health` 返回异常格式、读不到可信版本信息、或健康响应不可信时，不按“暂时故障后继续尝试兼容”处理，而是直接归入不兼容分支。
- Phase 46.3 的升级哲学是“同包一致优先”，不是“尽量兼容旧 daemon”。

### 不兼容 daemon 的替换策略

- 当 GUI 在 UniClipboard 预期的本地 daemon 端点上发现不兼容 daemon 时，允许 GUI 自动终止旧 daemon 并拉起当前版本 daemon。
- 自动替换的边界限定在 UniClipboard 自己的预期本地端点上；本阶段不扩展为通用进程管理器，也不处理其他地址/其他服务。
- 自动替换成功时保持静默，不额外打扰用户；只在替换失败时显式告知。
- 一次自动替换流程只允许有界重试；若终止旧 daemon 或启动新 daemon 仍失败，必须收敛到明确错误态，而不是无限后台重试。

### 启动失败与恢复体验

- 如果 GUI 最终无法获得健康且兼容的 daemon，不应静默进入主界面；应停在专门的启动错误页。
- 错误提示必须是持久、明确、可执行恢复动作的状态，不接受仅用一次性 toast 掩盖问题。
- 恢复路径同时提供“有界自动重试”和“用户手动重试入口”。
- 如果应用在错误页期间检测到 daemon 已恢复健康并兼容，GUI 应自动继续正常启动流程并切回主界面，不要求用户重启应用。

### Claude's Discretion

- 兼容探针的具体组合方式，例如版本比较、健康探针、路由/能力探针的先后顺序
- 自动终止旧 daemon 的具体实现机制与时序控制
- 启动错误页的具体文案、视觉样式与重试节奏
- 追踪与日志字段的细节，以及与 `StartupBarrier` / `DaemonConnectionState` 的集成方式

</decisions>

## Existing Code Insights

### Reusable Assets

- `src-tauri/crates/uc-tauri/src/bootstrap/run.rs`: 已有 `bootstrap_daemon_connection()`、`probe_daemon_health()`、`emit_daemon_connection_info_if_ready()`，是 46.3 最直接的修补入口
- `src-tauri/crates/uc-tauri/src/bootstrap/runtime.rs`: `DaemonConnectionState` 已经提供单一的内存态连接信息缓存，可继续作为 GUI 内部真实连接源
- `src-tauri/crates/uc-tauri/src/commands/startup.rs`: `StartupBarrier` 已能区分 backend/frontend ready，适合作为启动错误页和恢复切换的协调点
- `src-tauri/crates/uc-tauri/src/bootstrap/realtime_runtime.rs`: 现有 realtime/runtime consumers 与 GUI pairing lease keeper 已经体现“daemon ready 之后才真正可工作”的依赖关系

### Established Patterns

- GUI 负责 daemon bootstrap，但 daemon 负责长期业务宿主；本阶段不能把业务宿主重新拉回 Tauri
- 连接信息只保存在受控本地文件与 Rust managed state 中，不能持久化到前端存储
- 46.1/46.2 已经锁定“硬切换、无运行时回退”的姿态，因此 46.3 的失败路径也应是显式错误，而不是偷偷回退旧路
- 当前多个下游组件都依赖 `DaemonConnectionState` 已经准备好，所以启动层必须尽早、确定地给出“ready / not ready / incompatible”结论，不能把不确定性留到业务调用时才暴露

### Integration Points

- `src-tauri/src/main.rs` 中 setup 阶段启动的 daemon bootstrap 异步任务，是兼容判定、替换、失败收敛的主控点
- `src-tauri/crates/uc-daemon/src/api/{types.rs,routes.rs,query.rs}` 共同构成当前可用于版本/能力判断的 daemon 对外契约
- `src-tauri/crates/uc-tauri/src/bootstrap/realtime_runtime.rs` 与 `src-tauri/crates/uc-tauri/src/daemon_client/pairing.rs` 会直接消费 daemon connection state，适合作为验证 46.3 改动是否真正稳定的下游观察点
- `StartupBarrier` 与主窗口显示时机共同决定“进入主界面”还是“停在启动错误页”的用户可见行为

</code_context>
