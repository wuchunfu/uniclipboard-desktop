## User Constraints (from CONTEXT.md)

### Locked Decisions

## Urgency

Immediate architecture switch. Old pairing-specific WebSocket path must be removed rather than preserved behind compatibility layers or rollout flags.

## Requested Scope

- Remove the pairing-specific WebSocket path and unify runtime subscriptions on a singleton `DaemonWsBridge` long-lived connection.
- Do not preserve old event names, old bridge layers, or old call paths. Frontend and backend switch to the new contract in one cutover.
- Do not implement dual-write mirroring or feature flags. Land this as a direct architecture-switch commit sequence.

## Key Changes

- Architecture:
  - Add `RealtimeTopicPort` and `RealtimeEvent` in `uc-core`.
  - Add and enable `DaemonWsBridge` in `uc-tauri` as the only WebSocket connection with shared reconnect/resubscribe logic.
  - Add topic-consumer orchestration in `uc-app` so business modules no longer manage WebSocket lifecycle directly.
- Remove old implementation:
  - Delete `PairingBridge` connection and subscription logic.
  - Delete independent WebSocket connection capability from `setup_pairing_bridge`.
  - Restrict `start_background_tasks` to launching `DaemonWsBridge` and unified consumers only.
- Contract cutover:
  - Define one new frontend event contract, including event names, fields, and ordering.
  - Remove all legacy `p2p-*` compatibility listeners and emitters.
- Lifecycle and reliability:
  - State machine: `Disconnected -> Connecting -> Subscribing -> Ready -> Degraded`
  - Backoff: exponential with jitter, starting at `250ms` and capped at `30s`
  - Reconnect restores active topics automatically
  - Consumers use bounded queues with explicit backpressure drop policy

## Implementation Order

1. Define `RealtimeTopicPort` and the unified realtime event model in core.
2. Implement `DaemonWsBridge` in Tauri/infra and wire startup so only one runtime WebSocket exists.
3. Implement Pairing/Peers/Setup consumers that map daemon topics into the new frontend event contract.
4. Switch the frontend to the new event contract in one pass and remove legacy listeners.
5. Delete old bridges and subscription paths, then close compilation gaps.
6. Run regression tests and acceptance checks.

## Assumptions

- This is a breaking change. Old frontend listeners and event names will stop working immediately.
- The current branch accepts a medium-sized refactor with coordinated frontend updates.
- No rollout window, compatibility switch, or fallback path is required.

## Summary

Phase 46 already moved pairing ownership into the daemon, but the Tauri side still consumes realtime updates through duplicated websocket clients and duplicated event translation code. The current runtime opens one daemon websocket in `src-tauri/crates/uc-tauri/src/bootstrap/pairing_bridge.rs` and a second daemon websocket in `src-tauri/crates/uc-tauri/src/bootstrap/setup_pairing_bridge.rs`, while `start_background_tasks()` in `src-tauri/crates/uc-tauri/src/bootstrap/wiring.rs` still remains the startup owner for long-lived runtime subscriptions. The daemon transport itself is already topic-multiplexed and snapshot-first, so the missing architecture is on the client side, not the server side.

The correct planning frame is: do not “replace PairingBridge with another bridge.” Build one `DaemonWsBridge` adapter in `uc-tauri` that owns the only websocket, maps `DaemonWsEvent` into a typed core-level `RealtimeEvent`, and dispatches those events to bounded consumer queues behind a `RealtimeTopicPort`. Then move pairing/setup/peer translation into `uc-app` consumers so business modules stop knowing anything about websocket auth, reconnect, or JSON event strings.

**Primary recommendation:** use a typed `uc-core` realtime port plus a singleton `uc-tauri` `DaemonWsBridge`, then perform the cutover in five atomic commits: core port, app consumers, bridge adapter, frontend contract switch, deletion cleanup.

## Current Implementation Map

### Current websocket ownership

- `src-tauri/crates/uc-tauri/src/bootstrap/pairing_bridge.rs`
  - Opens an authenticated websocket in `connect_and_subscribe(...)`.
  - Subscribes to `["pairing", "peers", "paired-devices"]`.
  - Re-emits daemon payloads as legacy `p2p-*` Tauri events.
- `src-tauri/crates/uc-tauri/src/bootstrap/setup_pairing_bridge.rs`
  - Opens a second authenticated websocket in `subscribe()`.
  - Subscribes only to `["pairing"]`.
  - Maps daemon messages back into `PairingDomainEvent` for setup flow use.

  - Creates `PairingBridge::new(...)` at startup and passes it into `start_background_tasks(...)`.
- `src-tauri/crates/uc-daemon/src/api/ws.rs`
  - Already supports one authenticated `/ws` endpoint with topic multiplexing, snapshot-first subscription, and broadcast fanout.

- Two Tauri websocket clients consume the same daemon pairing topic.
- Pairing event translation exists both in `PairingBridge` and `DaemonBackedSetupPairingFacade`.
- Peer discovery/connection translation still bypasses any shared core realtime abstraction and stays Tauri-specific.
- Tests in `src-tauri/crates/uc-tauri/tests/pairing_bridge.rs` are mostly string-constant checks, not real reconnect or fanout verification.

- `src-tauri/crates/uc-daemon/src/api/types.rs`: stable daemon transport DTOs for pairing and peers.

### Pattern 1: Port In Core, Transport Mapping In Infra

**What:** Put the reusable subscription contract in `uc-core`, keep daemon websocket parsing in `uc-tauri`, and let `uc-app` consume typed events.

**When to use:** Any business module that needs realtime updates but should not own websocket or Tauri concerns.

**Example:**

```rust
// Source: repository architecture recommendation; transport semantics verified against
// Tokio channel docs and the current daemon topic model.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum RealtimeTopic {
    Pairing,
    Peers,
    PairedDevices,
}

#[derive(Debug, Clone)]
pub enum RealtimeEvent {
    PairingUpdated(PairingSessionChanged),
    PairingVerificationRequired(PairingVerificationRequired),
    PairingFailed(PairingFailure),
    PeerChanged(PeerChanged),
    PeerNameUpdated(PeerNameUpdated),
    PeerConnectionChanged(PeerConnectionChanged),
    PairedDeviceChanged(PairedDeviceChanged),
}

#[async_trait::async_trait]
pub trait RealtimeTopicPort: Send + Sync {
    async fn subscribe(
        &self,
        consumer: &'static str,
        topics: &[RealtimeTopic],
    ) -> anyhow::Result<tokio::sync::mpsc::Receiver<RealtimeEvent>>;
}
```

### Pattern 2: One Bridge, Many Consumers

**What:** `DaemonWsBridge` owns the websocket, maintains the active topic set, reconnects with backoff+jitter, resubscribes on reconnect, and fans out typed events to per-consumer bounded queues.

**When to use:** Runtime startup and all long-lived daemon realtime consumption.

**Example:**

```rust
// Source: current daemon /ws topic multiplexing in src-tauri/crates/uc-daemon/src/api/ws.rs
// plus Tokio bounded channel semantics from docs.rs.
loop {
    let socket = self.connect().await?;
    self.set_state(BridgeState::Subscribing);
    self.subscribe_all_active_topics(&socket).await?;
    self.set_state(BridgeState::Ready);

    while let Some(event) = socket.next_event().await {
        let realtime = self.decode(event)?;
        self.dispatch(realtime).await;
    }

    self.set_state(BridgeState::Degraded);
    tokio::time::sleep(self.next_backoff_with_jitter()).await;
}
```

### Pattern 3: Consumer-Owned Projection, Not Bridge-Owned Business Logic

**What:** The bridge decodes and dispatches. Consumers decide how to map `RealtimeEvent` into setup orchestration, host events, or frontend contract payloads.

**When to use:** Pairing/setup/peer-specific behavior.

**Example:**

```rust
// Source: current split between PairingBridge and DaemonBackedSetupPairingFacade,
// reorganized so both use the same app-layer consumer.
pub async fn run_setup_consumer(
    realtime: Arc<dyn RealtimeTopicPort>,
    setup_pairing_sink: Arc<dyn SetupPairingEventSink>,
) -> anyhow::Result<()> {
    let mut rx = realtime.subscribe("setup", &[RealtimeTopic::Pairing]).await?;
    while let Some(event) = rx.recv().await {
        if let Some(domain_event) = map_pairing_realtime_to_setup_domain(event) {
            setup_pairing_sink.handle(domain_event).await?;
        }
    }
    Ok(())
}
```

### Pattern 4: One Frontend Event Channel With Typed Envelope

**What:** Replace many `p2p-*` Tauri event names with one frontend realtime event channel carrying `{topic, type, ts, payload}` in camelCase.

**When to use:** Phase 46.1 frontend cutover.

**Example:**

```typescript
// Source: inferred contract shape from daemon event envelope + Tauri event docs.
type FrontendRealtimeEvent =
  | {
      topic: 'pairing'
      type: 'pairing.verificationRequired'
      ts: number
      payload: PairingVerificationPayload
    }
  | { topic: 'pairing'; type: 'pairing.updated'; ts: number; payload: PairingSessionPayload }
  | { topic: 'peers'; type: 'peers.changed'; ts: number; payload: PeerChangedPayload }
```

### Anti-Patterns to Avoid

- **Transport DTO leakage into `uc-app`:** do not let `DaemonWsEvent` or `serde_json::Value` become app-layer dependencies.
- **One websocket per consumer:** this recreates the exact ownership problem this phase is supposed to remove.
- **Bridge-owned setup/pairing business branching:** lifecycle and parsing belong in the bridge; business effects belong in consumers.
- **Unbounded queues:** they hide load problems and contradict the explicit backpressure requirement.
- **Mixing new contract rollout with compatibility mirroring:** the context explicitly forbids dual paths and compatibility flags.

## Don't Hand-Roll

| Problem                               | Don't Build                                                        | Use Instead                                           | Why                                                                                          |
| ------------------------------------- | ------------------------------------------------------------------ | ----------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| Reconnect and resubscribe per feature | Separate loops in pairing/setup/peers modules                      | One `DaemonWsBridge` state machine                    | Prevents duplicate sockets, drift in retry policy, and inconsistent ready/degraded semantics |
| App-layer JSON parsing                | `serde_json::Value` switches in `uc-app` or React                  | Typed `RealtimeEvent` mapping in `uc-tauri`           | Keeps boundaries clean and makes compiler-enforced cutover safer                             |
| Implicit fanout buffering             | Shared unbounded channels or direct broadcast receivers everywhere | Bounded per-consumer `mpsc` plus explicit drop policy | Tokio bounded channels provide backpressure; lag must be deliberate, not accidental          |
| Legacy compatibility event mirroring  | Keep `p2p-*` emitters “for now”                                    | One frontend event contract, one pass cutover         | The context requires a breaking switch with no compatibility layer                           |

**Key insight:** the daemon already solved topic multiplexing on the server side. Custom client-side duplication is now the architectural smell, not a missing library feature.

## Common Pitfalls

### Pitfall 1: Duplicate websocket ownership survives under a new name

**What goes wrong:** `DaemonWsBridge` gets added, but `DaemonBackedSetupPairingFacade::subscribe()` still opens its own websocket.

**Why it happens:** setup flow currently treats realtime subscription as part of the facade instead of a shared runtime service.

**How to avoid:** make `SetupPairingFacadePort` consume an app-layer setup consumer fed by `RealtimeTopicPort`; delete websocket code from `setup_pairing_bridge.rs`.

**Warning signs:** more than one `connect_async(request)` remains in `src-tauri/crates/uc-tauri/src/bootstrap/`.

### Pitfall 2: Port definition and adapter implementation land in one commit

**What goes wrong:** the phase becomes impossible to review atomically and violates the repo’s hexagonal commit boundary rule.

**Why it happens:** `RealtimeTopicPort` feels “small,” so it gets added together with `DaemonWsBridge`.

**How to avoid:** split commits strictly as `arch:` core port/model, then `impl:` app consumers, then `impl:` Tauri bridge.

**Warning signs:** one diff touches `uc-core`, `uc-app`, `uc-tauri`, and frontend listeners at once.

### Pitfall 3: Backpressure policy is implicit instead of encoded

**What goes wrong:** slow consumers lag until events are silently lost or memory grows without bound.

**Why it happens:** `broadcast` and `mpsc` are introduced without a documented consumer drop policy.

**How to avoid:** use bounded `mpsc` per consumer, `try_send` on dispatch, and choose policy explicitly:

- Latest-wins for coalescible peer presence/name/connection changes
- Never drop terminal pairing events (`verification_required`, `complete`, `failed`)
- Log and count dropped/coalesced events at the bridge boundary

**Warning signs:** queue size is not fixed, or drop behavior is described only in comments.

### Pitfall 4: CamelCase and `type` field regress during the cutover

**What goes wrong:** frontend listeners receive `session_id` or `event_type` and silently stop matching.

**Why it happens:** Tauri payloads and daemon payloads already have multiple naming conventions in play.

**How to avoid:** define one frontend realtime payload type, keep `#[serde(rename_all = "camelCase")]`, and keep `type` as the event discriminator. Add contract tests that assert camelCase keys are present and snake_case keys are absent.

**Warning signs:** any new realtime payload struct lacks `rename_all`, or frontend types still assume legacy field names.

### Pitfall 5: Deleting legacy `p2p-*` listeners before the new frontend contract is wired

**What goes wrong:** the backend cutover compiles, but discovery/pairing/setup UI stops reacting.

**Why it happens:** existing frontend hooks (`src/api/p2p.ts`, `src/hooks/useDeviceDiscovery.ts`, `src/components/PairingNotificationProvider.tsx`) are listener-centric and need a coordinated switch.

**How to avoid:** do the frontend event contract change and listener migration in one dedicated commit before deleting old Tauri emit paths.

**Warning signs:** `p2p-*` emitters are removed while `listen('p2p-...')` still exists in TypeScript.

### Bounded consumer queue with explicit overflow handling

```rust
// Source: Tokio bounded mpsc semantics
// https://docs.rs/tokio/latest/tokio/sync/mpsc/fn.channel.html
let (tx, rx) = tokio::sync::mpsc::channel::<RealtimeEvent>(64);

match tx.try_send(event) {
    Ok(()) => {}
    Err(tokio::sync::mpsc::error::TrySendError::Full(event)) => {
        if event.is_coalescible() {
            tracing::warn!(consumer = consumer_name, "dropping coalescible realtime event");
        } else {
            // Escalate terminal-event delivery failure explicitly.
            return Err(anyhow::anyhow!("terminal realtime event queue full"));
        }
    }
    Err(tokio::sync::mpsc::error::TrySendError::Closed(_)) => {}
}
```

### Detecting lag on broadcast fanout

```rust
// Source: Tokio broadcast lag behavior
// https://docs.rs/tokio/latest/tokio/sync/broadcast/index.html
match broadcast_rx.recv().await {
    Ok(event) => dispatch(event).await,
    Err(tokio::sync::broadcast::error::RecvError::Lagged(skipped)) => {
        tracing::warn!(skipped, "bridge fanout lagged; consumer needs resync");
    }
    Err(tokio::sync::broadcast::error::RecvError::Closed) => break,
}
```

## Recommended Atomic Commit Split

1. `arch:` add `RealtimeTopicPort`, `RealtimeTopic`, and typed `RealtimeEvent` in `uc-core` with unit tests only.
2. `impl:` add `uc-app` pairing/setup/peer realtime consumers behind the new port using test doubles; no Tauri code yet.
3. `impl:` add `uc-tauri` `DaemonWsBridge` implementing `RealtimeTopicPort`, with one websocket, reconnect/resubscribe logic, bounded dispatch, and bridge-focused tests.
4. `impl:` switch frontend and Tauri event emission to one new realtime contract; update React hooks/components/tests in the same commit.
5. `refactor:` delete `PairingBridge`, delete websocket code from `setup_pairing_bridge.rs`, delete legacy `p2p-*` listeners/emitters, and close compile gaps.

This split matches the repository’s strict boundary rules:

- Do not mix `uc-core` port definition with `uc-tauri` adapter implementation.
- Do not mix frontend contract cutover with structural bridge introduction unless the app is already compiling against the new bridge.

## Open Questions

1. **Exact frontend event name**
   - What we know: the context requires one frontend event contract and current code uses many `p2p-*` names.
   - What's unclear: the exact replacement event name is not locked in `46.1-CONTEXT.md`.
   - Recommendation: choose one explicit name in planning, keep it singular, and migrate all frontend listeners in one commit.

2. **Terminal-event overflow policy**
   - What we know: bounded queues and explicit drop policy are required.
   - What's unclear: whether terminal pairing events should block, fail fast, or bypass normal queueing when a consumer is saturated.
   - Recommendation: encode “terminal pairing events must not be silently dropped” as a plan invariant and test it directly.

3. **Whether topic subscriptions need dynamic unsubscribe in this phase**
   - What we know: current topics are effectively startup-time concerns (`pairing`, `peers`, `paired-devices`).
   - What's unclear: whether the bridge must support runtime add/remove now or whether startup union is enough.
   - Recommendation: plan for registration/refcount in the interface, but keep the first implementation simple if all current consumers are long-lived.
