# Plan 46-03: Tauri Compatibility Bridge For Existing Pairing Contract

## Goal

Keep the current desktop frontend contract working while removing Tauri as the pairing business host. Tauri should forward writes to the daemon, subscribe to daemon pairing/discovery topics, and re-emit the existing event names/payloads to the webview.

## Tasks

</read_first>
<action>
Create `src-tauri/crates/uc-tauri/src/daemon_client/pairing.rs` with one explicit client type:

- `pub struct TauriDaemonPairingClient`

The client must read `DaemonConnectionState`, attach `Authorization: Bearer <token>`, and expose these exact methods:

- `initiate_pairing(peer_id: String)`
- `accept_pairing(session_id: &str)`
- `reject_pairing(session_id: &str)`
- `cancel_pairing(session_id: &str)`
- `verify_pairing(session_id: &str, pin_matches: bool)`
- `set_pairing_participant_ready(client_kind: &str, ready: bool)`

Update `src-tauri/crates/uc-tauri/src/commands/pairing.rs` so the mutation commands keep their current Tauri command names and response payloads, but no longer own `State<'_, Arc<PairingOrchestrator>>` for write operations:

- `initiate_p2p_pairing`
- `accept_p2p_pairing`
- `reject_p2p_pairing`
- `verify_p2p_pairing_pin`

Concrete command behavior requirements:

- `initiate_p2p_pairing` calls daemon `POST /pairing/sessions` and still returns `P2PPairingResponse { sessionId, success, error }`
- `accept_p2p_pairing` calls daemon `POST /pairing/sessions/:session_id/accept`
- `reject_p2p_pairing` calls daemon `POST /pairing/sessions/:session_id/reject`
- `verify_p2p_pairing_pin` calls daemon `POST /pairing/sessions/:session_id/verify` using the existing `pinMatches` boolean instead of directly calling local orchestrator accept/reject

Preserve all `_trace: Option<TraceMetadata>` span instrumentation. Do not change any TypeScript frontend file in this task. The Rust adapter must satisfy the existing `src/api/p2p.ts` contract unchanged.
</action>

<action>
Create `src-tauri/crates/uc-tauri/src/bootstrap/pairing_bridge.rs` and export it from `src-tauri/crates/uc-tauri/src/bootstrap/mod.rs`.

The bridge must connect to `DaemonConnectionInfo.ws_url`, authenticate with the daemon bearer token, send one subscribe request with these exact topics:

- `pairing`
- `peers`
- `paired-devices`

Then translate daemon events back into the current frontend event names and payloads without changing the TypeScript contract:

- daemon `pairing.verification_required` -> emit `p2p-pairing-verification`
- daemon inbound request / waiting-for-user state -> emit `p2p-pairing-verification` with `kind: "request"`
- daemon verifying state -> emit `p2p-pairing-verification` with `kind: "verifying"`
- daemon terminal success -> emit `p2p-pairing-verification` with `kind: "complete"`
- daemon terminal failure -> emit `p2p-pairing-verification` with `kind: "failed"`
- daemon peer/discovery deltas -> emit `p2p-peer-discovery-changed`, `p2p-peer-name-updated`, and `p2p-peer-connection-changed`

Update `src-tauri/crates/uc-tauri/src/bootstrap/wiring.rs` so startup no longer owns the pairing business loops. Remove the startup task ownership of:

- `run_pairing_action_loop(...)`
- `run_pairing_event_loop(...)`

and replace it with bridge startup that:

- registers the desktop shell as a ready participant via daemon API
- subscribes to daemon websocket topics
- re-emits frontend-facing Tauri events

Do not change `src/api/p2p.ts`, `src/hooks/useDeviceDiscovery.ts`, `src/components/PairingNotificationProvider.tsx`, or `src/components/PairingDialog.tsx`. Rust bridge code must honor those existing event names and payload fields.

Add `src-tauri/crates/uc-tauri/tests/pairing_bridge.rs` with tests for:

- `bridge_preserves_existing_pairing_event_contract`
- `bridge_re_emits_peer_discovery_events_without_frontend_changes`
- `bridge_keeps_setup_flow_semantics`
