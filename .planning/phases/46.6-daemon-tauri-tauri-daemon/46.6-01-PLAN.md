  <action>
    1. 新增 `src-tauri/crates/uc-tauri/src/bootstrap/daemon_lifecycle.rs`，定义一个专门的运行期 owner contract，例如：
       - `enum SpawnReason { Absent, Replacement }`
       - `struct OwnedDaemonChild { pid: u32, spawn_reason: SpawnReason, child: std::process::Child }`
       - `struct GuiOwnedDaemonState`，内部通过 `Mutex<Option<OwnedDaemonChild>>` 持有 live child，并通过 `AtomicBool` 或等价 guard 维护 `exit_in_progress`
       - 必须提供这些显式方法：`record_spawned(child, spawn_reason)`、`clear()`、`snapshot_pid()`、`begin_exit_cleanup()`、`finish_exit_cleanup()`
       生产代码禁止 `unwrap/expect`，所有 lock poison 都要日志化并恢复。
    2. 在 `src-tauri/crates/uc-tauri/src/bootstrap/mod.rs` 中导出新的 lifecycle 模块，供 `main.rs` 和 `run.rs` 使用。
    3. 在 `src-tauri/src/main.rs` 中创建一份长生命周期 `GuiOwnedDaemonState`，并通过 `.manage()` 注册；不要把 live child handle 存进 46.3 的 `DaemonBootstrapOwnershipState`。
    4. 修改 `src-tauri/crates/uc-tauri/src/bootstrap/run.rs` 的真实 bootstrap 入口，使其接收共享的 `GuiOwnedDaemonState` 引用，而不是在函数内部仅创建局部 `DaemonBootstrapOwnershipState::default()`。
    5. 保留 46.3 的 `DaemonBootstrapOwnershipState` 作为 bootstrap facts snapshot，但把 live child ownership 改成：
       - `ProbeOutcome::Compatible(_)`：显式 `GuiOwnedDaemonState::clear()`，表示 GUI 并未拥有 daemon
       - `ProbeOutcome::Absent`：`spawn_daemon_process()` 返回 `std::process::Child`，将整个 `Child` 交给 `GuiOwnedDaemonState::record_spawned(child, SpawnReason::Absent)`，并同步更新 snapshot pid
       - `ProbeOutcome::Incompatible` replacement 成功后：将 replacement 产生的新 `Child` 交给 `GuiOwnedDaemonState::record_spawned(child, SpawnReason::Replacement)`，并同步更新 snapshot pid
    6. `src-tauri/src/main.rs` 中的 bootstrap 调用改为传入这份 managed lifecycle owner state，确保 bootstrap 完成后 ownership 不丢失。
    7. 保持现有 `WindowEvent::CloseRequested` 逻辑不变：`main` 窗口仍然 `prevent_close()` + `window.hide()`，不得在这条路径添加任何 daemon terminate/cleanup 调用。
    8. 不要把“连接到已有 compatible daemon”的路径误标为 GUI-owned daemon。
  </action>
