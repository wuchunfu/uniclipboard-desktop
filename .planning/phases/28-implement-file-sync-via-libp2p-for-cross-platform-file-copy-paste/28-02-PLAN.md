# Plan 28-02: File classification fix, settings model, and content type filter update

## Objective

Fix the critical file classification bug where `file://` URIs are misclassified as `Link`, extend the settings model with file sync fields, and make the `File` content type category filterable.

## must_haves

- [ ] `file://` URIs in `text/uri-list` classified as `File` instead of `Link`
- [ ] `http(s)://` URIs in `text/uri-list` still classified as `Link`
- [ ] `is_content_type_allowed()` checks `ct.file` for `File` category
- [ ] `FileSyncSettings` struct with all 6 fields and defaults
- [ ] TypeScript `FileSyncSettings` interface mirrors Rust struct
- [ ] Backward-compatible deserialization (old settings without file_sync)

## Tasks

<task id="28-02-01">
<title>Fix file classification and make File category filterable</title>
<description>
Update `uc-core/src/settings/content_type_filter.rs`:

1. **Fix `classify_snapshot()`**: When `text/uri-list` MIME is encountered, inspect the representation data to determine sub-classification:
   - Parse data as UTF-8 string
   - Check if any non-empty, non-comment line starts with `file://` (case-insensitive)
   - If yes: return `ContentTypeCategory::File`
   - Otherwise (http://, https://, or other schemes): return `ContentTypeCategory::Link`
   - If data cannot be parsed as UTF-8: fall back to `Link`

2. **Update `is_content_type_allowed()`**: Change `ContentTypeCategory::File` from always-true to checking `ct.file`:
   ```rust
   ContentTypeCategory::File => ct.file,
   ```
   Keep RichText, Link, CodeSnippet, Unknown as always-true (unimplemented).

3. **Update tests**:
   - Modify `make_snapshot()` helper to support setting representation data content
   - Add test: `text/uri-list` with `file:///home/user/doc.pdf` data classifies as `File`
   - Add test: `text/uri-list` with `https://example.com` data classifies as `Link`
   - Add test: `text/uri-list` with mixed lines (comment `#`, file://, http://) — first non-comment URI determines
   - Add test: `text/uri-list` with non-UTF-8 data falls back to `Link`
   - Update `unimplemented_types_always_allowed` test: `File` should now check `ct.file`
   - Add test: `File` category disallowed when `ct.file` is false
   - Add test: `File` category allowed when `ct.file` is true
</description>

<task id="28-02-02">
<title>Add FileSyncSettings to settings model and TypeScript interface</title>
<description>
1. **Rust settings model** (`uc-core/src/settings/model.rs`):
   - Add `FileSyncSettings` struct:
     ```rust
     #[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
     pub struct FileSyncSettings {
         pub file_sync_enabled: bool,
         pub small_file_threshold: u64,  // bytes, default 10MB
         pub max_file_size: u64,         // bytes, default 5GB
         pub file_cache_quota_per_device: u64,  // bytes, default 500MB
         pub file_retention_hours: u32,   // default 24
         pub file_auto_cleanup: bool,     // default true
     }
     ```
   - Add `#[serde(default)] pub file_sync: FileSyncSettings` to `Settings` struct

2. **Rust defaults** (`uc-core/src/settings/defaults.rs`):
   - Add `impl Default for FileSyncSettings`:
     - `file_sync_enabled: true`
     - `small_file_threshold: 10 * 1024 * 1024` (10MB)
     - `max_file_size: 5 * 1024 * 1024 * 1024` (5GB)
     - `file_cache_quota_per_device: 500 * 1024 * 1024` (500MB)
     - `file_retention_hours: 24`
     - `file_auto_cleanup: true`
   - Update `Settings::default()` to include `file_sync: FileSyncSettings::default()`

3. **TypeScript types** (`src/types/setting.ts`):
   - Add `FileSyncSettings` interface matching Rust struct
   - Add `file_sync?: FileSyncSettings` to `Settings` interface (optional for backward compat)

4. **Tests**:
   - Test backward-compat: deserialize JSON without `file_sync` field succeeds with defaults
   - Test round-trip: serialize + deserialize `FileSyncSettings` preserves values
   - Test default values match specification
</description>
