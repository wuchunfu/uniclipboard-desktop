**Step 3: Replace the decode call site (~line 972).**
The current code calls `uc_infra::clipboard::ChunkedDecoder::decode_from(sync_reader, &master_key)` inside `spawn_blocking`. Replace with buffer-then-decrypt approach:

```rust
let decryptor = self.transfer_decryptor.clone(); // clone Arc before moving into spawn_blocking
// Inside spawn_blocking:
let decode_result = tokio::task::spawn_blocking(move || {
    use std::io::Read;
    use tokio_util::io::SyncIoBridge;
    let mut sync_reader = SyncIoBridge::new(reader);
    let mut buf = Vec::new();
    sync_reader.read_to_end(&mut buf)
        .map_err(|e| anyhow::anyhow!("stream read failed: {e}"))?;
    // Use tokio runtime handle to call async decrypt from blocking context
    let rt = tokio::runtime::Handle::current();
    rt.block_on(decryptor.decrypt(&buf, &master_key))
        .map_err(|e| anyhow::anyhow!("decrypt failed: {e}"))
}).await;
```

Note: `TransferPayloadDecryptorPort::decrypt` is async. Inside `spawn_blocking`, use `tokio::runtime::Handle::current().block_on()` to call it. Alternatively, if the concrete implementation is CPU-bound (no actual async I/O), consider whether the port trait should have a sync variant. Per user decision, use the existing port as-is with buffer-then-decrypt. If `block_on` inside `spawn_blocking` proves problematic, restructure to: (1) buffer in spawn_blocking, (2) return buf, (3) call decrypt outside spawn_blocking in the async context. This second approach is cleaner:

```rust
let buf = tokio::task::spawn_blocking(move || {
    use std::io::Read;
    use tokio_util::io::SyncIoBridge;
    let mut sync_reader = SyncIoBridge::new(reader);
    let mut buf = Vec::new();
    sync_reader.read_to_end(&mut buf)
        .map_err(|e| anyhow::anyhow!("stream read failed: {e}"))?;
    Ok::<Vec<u8>, anyhow::Error>(buf)
}).await.map_err(|e| anyhow::anyhow!("buffer task panicked: {e}"))??;

let decrypted = self.transfer_decryptor.decrypt(&buf, &master_key).await
    .map_err(|e| anyhow::anyhow!("decrypt failed: {e}"))?;
```

Use whichever compiles cleanly. The second approach (buffer in spawn_blocking, decrypt in async) is preferred since it avoids `block_on` inside `spawn_blocking`.
