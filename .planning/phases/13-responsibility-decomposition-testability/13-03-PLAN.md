<!-- Key insight from research -->
The orchestrator manages concurrent sessions via HashMap<SessionId, PairingSessionContext>.
Protocol handler methods (handle_challenge, handle_keyslot_offer, etc.) need read/write
access to sessions but should be grouped as protocol-level concerns.
Session manager methods (create session, cleanup expired, timeout) are lifecycle concerns.
The orchestrator keeps the sessions map and passes references.

    1. Read the full `orchestrator.rs` (2107 lines) to categorize every method into one of three buckets:
       a. **Protocol handling** — methods that process protocol messages: handle_challenge, handle_keyslot_offer, handle_challenge_response, handle_response, handle_confirm, handle_reject, handle_cancel, handle_busy, handle_transport_error, and the internal crypto/verification helper methods they call
       b. **Session lifecycle** — methods that create, find, cleanup, or timeout sessions: session creation in initiate_pairing/handle_incoming_request, cleanup_expired_sessions, get_session_peer, get_session_role, and internal session lookup/mutation helpers
       c. **Orchestrator coordination** — the public API methods that coordinate between protocol handling and session management, event emission. These stay in orchestrator.

    2. Create `protocol_handler.rs` with `PairingProtocolHandler`:
       - This is a concrete struct (NOT a trait — per research recommendation to avoid over-abstraction)
       - Owns port `Arc` references needed for protocol operations: `encryption_session`, `paired_device_repo`, `network_ports.pairing` (transport), `key_material`, `key_scope`, and any other ports used exclusively by protocol methods
       - Methods receive session context references as parameters (borrowing from orchestrator's session map, not owning)
       - Each protocol handler method returns domain events or state changes, not directly emitting events

    3. Create `session_manager.rs` with `PairingSessionManager`:
       - Concrete struct owning the sessions `HashMap` (or `Arc<Mutex<HashMap>>` if currently behind a lock)
       - Methods: `create_session`, `get_session`, `get_session_mut`, `remove_session`, `cleanup_expired`, `get_peer_info`, `get_role`
       - Owns the `PairingConfig` for timeout settings
       - Returns session references; does NOT own port references

    4. Update `orchestrator.rs`:
       - Add `protocol_handler: PairingProtocolHandler` and `session_manager: PairingSessionManager` fields
       - Update `new()` to construct both sub-components, passing port Arc references to protocol handler and session map to session manager
       - Each public API method becomes a thin coordinator:
         ```
         pub async fn handle_challenge(...) -> Result<()> {
             let session = self.session_manager.get_session_mut(session_id)?;
             let events = self.protocol_handler.process_challenge(session, ...)?;
             self.emit_events(events).await;
         }
         ```
       - Keep event emission (channel send) in the orchestrator — it's the coordination concern
       - Move inline helper methods to the appropriate sub-component

    5. Update `mod.rs` to add `mod protocol_handler;` and `mod session_manager;` (internal, no pub export needed)

    CRITICAL pitfalls:
    - The sessions map (`HashMap<SessionId, PairingSessionContext>`) must be in ONE place. If it's behind `tokio::sync::Mutex`, the session manager owns the mutex. The orchestrator and protocol handler borrow sessions via session manager methods.
    - Do NOT create circular dependencies between protocol_handler and session_manager. Dependency flow: orchestrator → {protocol_handler, session_manager}. Protocol handler does NOT import session manager.
    - User-facing methods (user_accept_pairing, user_reject_pairing) may combine session lookup + protocol action + event emission — these stay as orchestrator coordination methods that call into both sub-components.
    - Keep the `#[cfg(test)] mod tests` block in orchestrator.rs. Inline tests that test the public API should stay. Only add `use crate::testing::*;` if they use shared noops (from Plan 01).
