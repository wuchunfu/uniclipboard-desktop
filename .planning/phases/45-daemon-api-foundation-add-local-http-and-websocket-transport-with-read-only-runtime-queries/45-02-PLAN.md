# Plan 45-02: Daemon HTTP And WebSocket Server

## Goal

Stand up the new loopback daemon API server, serve the read-only endpoints, and implement the Phase 45 WebSocket contract while keeping the legacy Unix socket RPC server alive during the transition.

Create `src-tauri/crates/uc-daemon/src/api/server.rs` and `src-tauri/crates/uc-daemon/src/api/routes.rs`.

`server.rs` must define:

- a shared application state struct holding the query service, auth token, and any broadcast/snapshot helpers needed by the routes and WebSocket handler
- `run_http_server(...)` or equivalent async entrypoint that binds explicitly to `127.0.0.1`
- logic to derive `DaemonConnectionInfo` from the chosen listen address
- a `build_router(...) -> axum::Router` helper so route tests can exercise the router without booting the full daemon process

Use a fixed helper in `src-tauri/crates/uc-daemon/src/socket.rs` for the HTTP listen address, for example:

- `pub fn resolve_daemon_http_addr() -> SocketAddr`

The default bind target must be exactly `127.0.0.1` with an explicit port source. If the port is configurable, read it from one daemon-owned helper instead of hardcoding multiple copies.

`routes.rs` must define the exact Phase 45 routes:

- `GET /health`
- `GET /status`
- `GET /peers`
- `GET /paired-devices`
- `GET /pairing/sessions/:session_id`

Route behavior requirements:

- `/health` may be minimal and machine-readable; if left unauthenticated it must remain liveness-only and expose no runtime detail
- `/status`, `/peers`, `/paired-devices`, and `/pairing/sessions/:session_id` require bearer-token auth
- `/pairing/sessions/:session_id` returns a normal not-found result when no daemon-side summary exists yet
- route handlers call the query service from Plan 45-01 instead of reimplementing data assembly inside handlers

Use these concrete response semantics so implementation does not drift:

- `/health` returns `200` with JSON body `{"status":"ok","version":"<pkg-version>"}`
- `/status` returns `200` with `StatusResponse`
- `/peers` returns `200` with JSON array of `PeerSnapshotDto`
- `/paired-devices` returns `200` with JSON array of `PairedDeviceDto`
- `/pairing/sessions/:session_id` returns `200` with `PairingSessionSummaryDto` when found, `404` with a small machine-readable error body when absent
- invalid or missing bearer token returns `401`

Update `src-tauri/crates/uc-daemon/src/app.rs` so daemon startup launches the HTTP server task alongside the existing RPC accept loop. Keep RPC startup/shutdown behavior intact during this phase. Update shutdown logic so both transport servers are cancelled and drained cleanly.

The daemon lifecycle change must be explicit:

- keep Unix socket stale-socket verification and bind-first behavior for RPC
- start workers after transport binds succeed
- spawn one task for `run_rpc_accept_loop(...)`
- spawn one task for `run_http_server(...)`
- treat unexpected exit of either server task as a warning-worthy daemon failure condition in the main `tokio::select!`
- on shutdown, cancel the shared token first, then drain worker tasks, then await both server tasks with timeout, then stop workers in reverse order

Do not remove JSON-RPC in this plan. The HTTP server is additive.

Add `src-tauri/crates/uc-daemon/tests/http_api.rs` with integration-style tests that build the router or server state directly and assert:

- `/health` is reachable on loopback semantics without auth
- `/status` returns `401` without bearer token
- `/status` returns `200` with a valid bearer token
- `/paired-devices` returns `200` and serializes an array body
- `/pairing/sessions/:session_id` returns `404` when the session is absent

<action>
Create `src-tauri/crates/uc-daemon/src/api/ws.rs` and wire it through `api/mod.rs`.

The WebSocket module must implement:

- an authenticated upgrade path under the daemon API server
- a subscribe message that accepts a list of topics, for example `status`, `peers`, `paired-devices`, and `pairing`
- snapshot-first semantics: immediately after successful subscribe, emit one snapshot event per subscribed topic
- incremental event streaming after the snapshot

Use the exact event envelope fields from the context:

- `topic`
- `type`
- `sessionId`
- `ts`
- `payload`

Use one concrete subscribe request shape:

```json
{ "action": "subscribe", "topics": ["peers", "paired-devices"] }
```

Use these exact snapshot event types for the first emission:

- `status.snapshot`
- `peers.snapshot`
- `paired-devices.snapshot`
- `pairing.snapshot`

For the Phase 45 incremental side, define the concrete event-type strings now even if some topics only emit snapshots in this phase:

- `status.updated`
- `peers.changed`
- `paired-devices.changed`
- `pairing.updated`

The WebSocket layer must not expose raw internal state-machine transitions that are not part of the domain-facing read model. For pairing, use the existing app-layer domain event shapes only as a future shape reference. In Phase 45, pairing is explicitly snapshot-only because the daemon is not yet the pairing host. The subscribe protocol and envelope must already be stable, but incremental pairing events are deferred to Phase 46.

Implement the WebSocket task boundary so one connection does not block daemon progress:

- receive loop parses subscribe messages from the client
- send loop writes snapshot and incremental events to the socket
- any background fanout to a specific client uses channel-based handoff, not blocking work inside the main server accept path

For pairing topic behavior in this phase:

- `pairing.snapshot` is supported
- `pairing.updated` is reserved for Phase 46 and later, not a Phase 45 must-have
- `KeyslotReceived` is not serialized directly because it contains high-sensitivity material

Do not serialize `KeySlotFile`, `challenge`, or raw failure internals into WebSocket payloads. Phase 45 pairing payloads must remain metadata-only.

Add integration tests:

- `src-tauri/crates/uc-daemon/tests/http_api.rs`
- `src-tauri/crates/uc-daemon/tests/websocket_api.rs`

The WebSocket tests must cover:

- upgrade rejected without valid bearer token
- subscribe request with `["peers"]` yields a first event whose `type` is `peers.snapshot`
- subscribe request with multiple topics yields one snapshot event per topic
- serialized event contains `sessionId` key and does not contain `session_id`
- pairing snapshot payload does not contain `keyslotFile` or raw challenge bytes
