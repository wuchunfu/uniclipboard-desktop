# Plan 45-01: Daemon API Contract And Auth Foundation

## Goal

Create the transport contract that all later Phase 45 work builds on: API DTOs, token persistence/auth helpers, daemon connection info, and a read-only query service that can answer the planned HTTP routes without depending on Tauri command code.

## Tasks

<action>
Update `src-tauri/crates/uc-daemon/Cargo.toml` to add the HTTP transport dependencies required by the phase. Use:

- `axum` for route definitions and WebSocket upgrade support
- `hyper`/`tokio` ecosystem types only as required by `axum`
- `rand` or `uuid`-backed secure random bytes for bearer-token generation
- `tempfile` in dev-dependencies if auth token persistence tests need isolated files

Create `src-tauri/crates/uc-daemon/src/api/mod.rs` and re-export three new submodules:

- `auth`
- `types`
- `query`

Update `src-tauri/crates/uc-daemon/src/lib.rs` to expose `pub mod api;`.

Create `src-tauri/crates/uc-daemon/src/api/auth.rs` with the following concrete responsibilities:

- `pub struct DaemonConnectionInfo { pub base_url: String, pub ws_url: String, pub token: String }`
- `pub struct DaemonAuthToken(String)` or equivalent newtype used internally
- `pub fn resolve_daemon_token_path(...) -> PathBuf` or equivalent helper with an explicit daemon-local filename such as `uniclipboard-daemon.token`
- `pub fn load_or_create_auth_token(token_path: &Path) -> anyhow::Result<DaemonAuthToken>` that reads an existing token file or generates and persists a new random token
- `pub fn build_connection_info(host: &str, port: u16, token: &DaemonAuthToken) -> DaemonConnectionInfo`
- `pub fn parse_bearer_token(header_value: &str) -> Option<&str>` or equivalent header parser used by Phase 45 route auth

Implement token persistence in a dedicated daemon-local file, not in the socket path helper and not in frontend storage. Prefer colocating path resolution with other daemon connection helpers by updating `src-tauri/crates/uc-daemon/src/socket.rs` to own both:

- Unix socket path resolution
- loopback HTTP listen defaults or token-path companion helpers

Use the exact URL shape below when constructing `DaemonConnectionInfo`:

- `base_url = format!("http://{host}:{port}")`
- `ws_url = format!("ws://{host}:{port}/ws")`

The token path logic must be explicit in code comments or function names. Do not use URL query strings for auth. Do not log the raw token value. On Unix, the implementation must create the file with restricted permissions or repair too-wide permissions when discovered.

Add `src-tauri/crates/uc-daemon/tests/api_auth.rs` with tests for:

- load-creates-token-when-missing
- load-reuses-existing-token
- bearer-header-parser accepts `Bearer abc` and rejects malformed values
- connection-info builder returns `http://127.0.0.1:<port>` and `ws://127.0.0.1:<port>/ws`
- token path helper ends with `uniclipboard-daemon.token`
  </action>

<action>
Create `src-tauri/crates/uc-daemon/src/api/types.rs` with transport DTOs for:

- `HealthResponse`
- `StatusResponse` for HTTP use, reusing worker-health semantics but not JSON-RPC framing
- `PeerSnapshotDto`
- `PairedDeviceDto`
- `PairingSessionSummaryDto`
- `DaemonWsSubscribeRequest`
- `DaemonWsEvent`

The WebSocket event envelope must include the exact fields locked in context:

- `topic`
- `type`
- `session_id` on the Rust side, serialized as `sessionId`
- `ts`
- `payload`

All event payload structs intended for frontend consumption must derive `serde::Serialize` and include `#[serde(rename_all = "camelCase")]`.

Use these concrete DTO fields so later plans do not need to rediscover them:

- `HealthResponse { status: String, version: String }`
- `StatusResponse { version: String, uptime_seconds: u64, workers: Vec<WorkerStatusDto>, connected_peers: u32 }`
- `WorkerStatusDto { name: String, health: String }`
- `PeerSnapshotDto { peer_id: String, device_name: Option<String>, addresses: Vec<String>, is_paired: bool, connected: bool, pairing_state: String }`
- `PairedDeviceDto { peer_id: String, device_name: String, pairing_state: String, last_seen_at_ms: Option<i64>, connected: bool }`
- `PairingSessionSummaryDto { session_id: String, peer_id: Option<String>, device_name: Option<String>, state: String, updated_at_ms: i64 }`
- `DaemonWsSubscribeRequest { action: String, topics: Vec<String> }`
- `DaemonWsEvent { topic: String, event_type: String, session_id: Option<String>, ts: i64, payload: serde_json::Value }`

The Rust field name for event type may be `event_type`, but it must serialize as JSON key `type`. The Rust field name for `session_id` must serialize as `sessionId`.

Create `src-tauri/crates/uc-daemon/src/api/query.rs` with a `DaemonQueryService` or equivalent that:

- owns or borrows the daemon runtime dependencies needed to answer read-only requests
- uses `CoreUseCases::get_p2p_peers_snapshot()` for `/peers`
- uses `CoreUseCases::list_paired_devices()` or filtered peer snapshots for `/paired-devices`
- derives `status` from daemon runtime state plus version information
- exposes a pairing-session summary lookup that returns `None` when the daemon does not yet know the session instead of proxying into Tauri-owned state

Give the service an explicit constructor and method surface:

- `pub fn new(runtime: Arc<CoreRuntime>, state: Arc<RwLock<RuntimeState>>) -> Self`
- `pub async fn health(&self) -> HealthResponse`
- `pub async fn status(&self) -> anyhow::Result<StatusResponse>`
- `pub async fn peers(&self) -> anyhow::Result<Vec<PeerSnapshotDto>>`
- `pub async fn paired_devices(&self) -> anyhow::Result<Vec<PairedDeviceDto>>`
- `pub async fn pairing_session(&self, session_id: &str) -> anyhow::Result<Option<PairingSessionSummaryDto>>`

Update `src-tauri/crates/uc-daemon/src/state.rs` as needed so the daemon can serve worker health and connection summaries to the new query layer without coupling to RPC-specific types. If this requires separating internal state from `rpc::types::WorkerStatus`, do that now so the new API layer does not depend on JSON-RPC DTOs.

Sensitive fields must be omitted from all DTOs. The allowed default fields are `peerId`, `deviceName`, `addresses`, `isPaired`, `connected`, and `pairingState`. Do not serialize `sharedSecret`, raw keys, decrypted clipboard content, or `KeySlotFile`.

Add `src-tauri/crates/uc-daemon/tests/api_query.rs` with tests for:

- empty status response
- peers query maps `is_connected` to `connected`
- paired-devices query does not expose `identity_fingerprint`
- pairing-session query returns `None` when no daemon-side record exists
- WebSocket DTO serialization yields `sessionId` and `type` keys, not `session_id` and `event_type`

<action>
Refactor `src-tauri/crates/uc-daemon/src/state.rs` so the daemon runtime snapshot is transport-neutral. The target state for this task is:

- `RuntimeState` stores daemon-owned worker snapshot data, not JSON-RPC DTOs from `rpc/types.rs`
- JSON-RPC handler code maps the internal worker snapshot to `rpc::types::WorkerStatus`
- the new API query service maps the same internal worker snapshot to HTTP DTOs

If needed, introduce an internal snapshot type in `state.rs`, for example:

- `DaemonWorkerSnapshot { name: String, health: WorkerHealth }`

Then update:

- `src-tauri/crates/uc-daemon/src/app.rs` initialization of runtime state
- `src-tauri/crates/uc-daemon/src/rpc/handler.rs` status-response mapping
- `src-tauri/crates/uc-daemon/src/api/query.rs` status-response mapping

The point of this task is to stop Phase 45 from building the new HTTP contract on top of legacy JSON-RPC types.
</action>
