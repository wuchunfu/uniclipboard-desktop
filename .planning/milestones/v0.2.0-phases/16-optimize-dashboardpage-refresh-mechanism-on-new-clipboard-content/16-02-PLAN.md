Purpose: Replace the DashboardPage's full-reload-on-every-event pattern with incremental updates for local captures, reducing unnecessary DB queries and improving responsiveness.
Output: `useClipboardEvents` hook managing all event lifecycle, simplified DashboardPage as render layer.

    1. **clipboardItems.ts**: Add `getClipboardEntry(entryId: string)` function. It calls `invokeWithTrace<ClipboardEntriesResponse>('get_clipboard_entry', { entryId })`. If response is `not_ready` or entries is empty, return null.     2. **useClipboardEvents.ts**: Create hook at `src/hooks/useClipboardEvents.ts`. The hook encapsulates ALL the event/state management currently scattered in DashboardPage:

    ```typescript
    interface UseClipboardEventsReturn {
      hasMore: boolean
      handleLoadMore: () => void
      encryptionReady: boolean
    }

    export function useClipboardEvents(currentFilter: Filter): UseClipboardEventsReturn
    ```

    Internal implementation:
    - **Refs**: `encryptionReadyRef`, `pendingInitialLoadRef`, `loadInFlightRef`, `offsetRef`, `hasMoreRef`, `currentFilterRef`, `throttleTimeoutRef`, `lastReloadTimestampRef` (all from DashboardPage, NO global module state)
    - **State**: `hasMore` (synced with hasMoreRef)
    - **THROTTLE_WINDOW_MS = 300** (reduced from 500 per user decision)
    - **PAGE_SIZE = 20** (same as current)

    - **loadData function**: Same logic as current DashboardPage.loadData. Dispatches `fetchClipboardItems` with filter/limit/offset. Updates offsetRef and hasMore. On reset, resets offset to 0.

    - **Filter change effect**: When `currentFilter` changes, update `currentFilterRef`. If encryption not ready, set `pendingInitialLoadRef = true`. Otherwise call `loadData({ specificFilter: currentFilter, reset: true })`.

    - **Clipboard event listener effect**: Listen to `clipboard://event`. On `NewContent`:
      - Gate on `encryptionReadyRef.current === true`
      - If `event.payload.origin === 'local'`:
        - Call `getClipboardEntry(event.payload.entry_id)`
        - If successful, dispatch `prependItem(item)` and increment `offsetRef.current += 1`
        - If failed, fall back to `loadData({ reset: true })` (silent fallback per user decision)
      - If `event.payload.origin === 'remote'` (or no origin for backward compat):
        - Apply throttle logic (300ms window, trailing). On throttle fire, call `loadData({ reset: true })`
      - On `Deleted`:
        - Dispatch `removeItem(event.payload.entry_id)`
        - Decrement `offsetRef.current -= 1` (but floor at 0)

    - **Encryption listener effect**: Listen to `encryption://event`. On `SessionReady`, set `encryptionReadyRef.current = true`, dispatch `setNotReady(false)`. Also check initial status via `getEncryptionSessionStatus()` on mount.

    - **Cleanup**: All listeners properly unlistened on unmount. Clear throttle timeouts. NO global module-level state.

    - **handleLoadMore**: Same as current -- gates on encryption ready, calls `loadData()` (non-reset, appending).

    3. **useClipboardEvents.test.ts**: Create test file. Since this hook uses Tauri's `listen()` and `invoke()`, mock `@tauri-apps/api/event` and `@/api/clipboardItems`. Test:
    - P16-05: Local origin event triggers getClipboardEntry call and dispatches prependItem
    - P16-06: Remote origin event triggers throttled full reload (fetchClipboardItems dispatch)
    - Deleted event dispatches removeItem
    - Encryption not ready: events are ignored
    - Use `renderHook` from `@testing-library/react` with a Redux provider wrapper
