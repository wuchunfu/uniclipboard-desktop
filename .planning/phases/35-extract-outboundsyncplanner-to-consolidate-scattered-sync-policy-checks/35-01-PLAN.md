<design_decisions>

## APFS Resolution and File Size Contract

File path extraction, APFS resolution, AND file size retrieval all happen in the RUNTIME
(uc-tauri) BEFORE calling the planner. The runtime builds `Vec<FileCandidate>` where each
`FileCandidate` contains the resolved path and its file size (from `std::fs::metadata()`).

The planner receives these pre-computed candidates and performs PURE LOGIC filtering:
comparing `candidate.size` against `max_file_size` from settings. The planner makes NO
filesystem calls whatsoever -- it is a pure function of its inputs.

The planner's signature is:

```rust
pub async fn plan(
    &self,
    snapshot: SystemClipboardSnapshot,
    origin: ClipboardChangeOrigin,
    file_candidates: Vec<FileCandidate>,  // Pre-computed by runtime: resolved path + size
    extracted_paths_count: usize,         // Number of paths BEFORE metadata filtering (for all_files_excluded detection)
) -> OutboundSyncPlan
```

## extracted_paths_count Parameter

The runtime extracts file paths from the snapshot, then calls `std::fs::metadata()` on each.
Files where metadata() fails are excluded at the runtime level (with tracing::warn).
The planner needs to know the ORIGINAL count of extracted paths to detect the case where
ALL files were excluded (by metadata failure and/or size filtering). Without this count,
the planner cannot distinguish "no files existed in the snapshot" from "all files were
excluded by metadata failures" -- and would incorrectly allow clipboard sync in the latter case.

The all_files_excluded guard logic is:
`extracted_paths_count > 0 && plan.files.is_empty()` after size filtering -> clipboard: None

This preserves the existing behavior in runtime.rs lines 1273-1280.

## LocalRestore Semantics

Current behavior (verified from sync_outbound.rs line 177): only RemotePush is skipped.
LocalRestore DOES trigger clipboard sync. Since this phase is "no user-visible behavior change",
the planner preserves this: LocalRestore -> clipboard: Some, files: [].

## classify_snapshot() Scope

classify_snapshot() is NOT called by the planner in this phase. Content type classification
is a per-peer filtering concern handled by apply_sync_policy() inside the use cases.
The "classify once" optimization is a future concern. This avoids expanding scope.
</design_decisions>
